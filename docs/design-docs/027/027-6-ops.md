## Deployment

### Staging

```bash
# No migration needed
pnpm run deploy:staging:data-service
```

### Production

```bash
# No migration needed
pnpm run deploy:production:data-service
```

**No user-application deployment needed** -- this is a backend-only fix.

**Deployment is atomic**: Both `SessionAgent` and `PipelineOrchestratorAgent` are Durable Objects within the same Worker (`data-service`). Deploying the Worker updates both simultaneously. There is no window where the caller sends the old shape and the callee expects the new shape.

---

## Rollback Plan

If issues are discovered after deployment:

1. **Redeploy previous version**: `pnpm run deploy:staging:data-service` (or production) with the previous commit. The old code resumes hardcoding confidence to 0 and skipping the confidence gate.
2. **No database rollback needed**: No schema changes were made. New `pipeline_outcomes` rows with real confidence values are harmless to the old code, which never reads them for gating purposes.
3. **No data corruption risk**: The fix only changes what values are written to an existing column and adds a filtering check. Rolling back means reverting to the pre-fix behavior (no gate, zero confidence), which is the current known state.

## Post-Deployment Validation

After deploying to staging, run the following manual test sequence:

```
1. Open session settings, set orchestration mode to "pipeline"
2. Set minConfidenceThreshold to 0.9 (high, to make filtering observable)
3. Add a symbol to watchlist
4. Trigger analysis cycle
5. Observe discussion feed:
   - If LLM confidence < 0.9: expect "Confidence X.XX below threshold 0.90" message
   - If LLM confidence >= 0.9: expect proposal generated
6. If a proposal was generated and executed:
   - Wait for outcome resolution
   - Query pipeline_outcomes table via wrangler DO storage:
     SELECT confidence FROM pipeline_outcomes ORDER BY created_at DESC LIMIT 1
   - Confirm value is non-zero and matches the proposal's confidence
7. Switch to debate mode, trigger analysis
   - Confirm same threshold behavior
8. Reset minConfidenceThreshold to 0.7 (default)
```
