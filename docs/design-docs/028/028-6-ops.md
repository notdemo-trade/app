To verify the fix works:

2. Wait for an analysis cycle to produce a trade proposal
3. Before approving, temporarily invalidate the broker credentials (e.g., revoke the API key in Alpaca dashboard)
4. Approve the proposal in the UI
5. **Before fix**: Proposal status stays `'approved'` in the UI and database
6. **After fix**: Proposal status transitions to `'failed'` with an error message like "Execution failed: 403 Forbidden"

8. Click "Retry Execution" in the UI
9. Proposal transitions to `'executed'`

## Deployment

### No Database Migration

This fix does not require any PostgreSQL migration. The change is entirely within the DO SQLite layer (which stores `status` as untyped `TEXT`) and TypeScript types.

### Deploy Sequence

```bash
# 1. Build data-ops (updated type)
cd packages/data-ops && pnpm run build

# 2. Deploy data-service (updated session agent)
pnpm run deploy:staging:data-service

# 3. Deploy user-application (updated UI)
pnpm run deploy:staging:user-application
```

**Order matters**: Deploy `data-service` before `user-application`. If the UI is deployed first, it will render the `'failed'` badge for proposals that the old backend still marks as `'approved'`. This is a minor visual inconsistency during the deployment window but causes no functional issues.

**Backward compatibility**: The old UI (pre-deploy) does not have `'failed'` in its `STATUS_CONFIG` record. If the new backend returns a proposal with `status: 'failed'`, the old UI will render it with `undefined` status config. This results in a missing badge (no crash, just no status indicator). The `STATUS_CONFIG` record lookup `STATUS_CONFIG[proposal.status]` returns `undefined` for unknown keys, and the existing `statusConfig.variant` access would throw. To prevent this, deploy the UI shortly after the backend.

## Rollback Plan

### Backend Rollback

Redeploy the previous version of `data-service`. The previous code writes `status: 'approved'` on failure (the original bug). Any proposals that were already set to `'failed'` by the new code will:
- Not be displayed with the correct badge (old UI has no `'failed'` entry)
- Not be swept by `expireProposals()` (old code only sweeps `'pending'`)
- Be cleaned up by `resetData()` if manually triggered

This is acceptable as a temporary state during rollback.

### Frontend Rollback

Redeploy the previous version of `user-application`. Proposals with `'failed'` status will not render correctly (missing badge). The list page may show proposals without a status indicator.

### Full Rollback

Deploy both old versions. The only persistent artifact is any `'failed'` status rows in DO SQLite. These can be manually cleaned up via:

```ts
// In a temporary admin callable or via resetData()
this.sql`UPDATE trade_proposals SET status = 'expired'
    WHERE status = 'failed'`;
```

## Open Questions

1. **Should `'approved'` be included in the expiration sweep?** Including `'approved'` in `expireProposals()` provides a safety net against any future zombie scenarios but introduces a theoretical race condition with in-flight executions. The race is benign (execution overwrites expiration) but adds complexity to reasoning about the state machine. **Recommendation**: Include it -- the safety benefit outweighs the theoretical concern.

2. **Should the retry button be available on the proposal list page or only on the detail page?** The current design puts the retry action on the detail page to keep the list page clean. However, if users frequently encounter failures, a list-level retry button would reduce clicks. **Recommendation**: Start with detail-page only; add list-page retry if user feedback warrants it.

3. **Should the error message from the failed execution be persisted in the database?** Currently, the error message is only returned in the RPC response. If the user navigates away, the error message is lost. Adding an `error_message` column to `trade_proposals` would allow displaying the failure reason in the proposal detail view even after page refresh. **Recommendation**: Defer to a follow-up. The fix addresses the core bug (zombie status). Error message persistence is an enhancement.
