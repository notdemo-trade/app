```ts
const STATUS_CONFIG: Record<
    TradeProposal['status'],
    {
        key: string;
        variant: 'default' | 'secondary' | 'success' | 'destructive' | 'warning' | 'outline';
    }
> = {
    pending: { key: 'proposal.status.pending', variant: 'warning' },
    approved: { key: 'proposal.status.approved', variant: 'success' },
    rejected: { key: 'proposal.status.rejected', variant: 'destructive' },
    expired: { key: 'proposal.status.expired', variant: 'secondary' },
    executed: { key: 'proposal.status.executed', variant: 'success' },
    failed: { key: 'proposal.status.failed', variant: 'destructive' },
};
```

The `'failed'` status uses the `'destructive'` variant (red) to clearly signal that something went wrong and user attention is needed. This is the same variant used for `'rejected'`, which is appropriate because both are negative outcomes. The distinction is in the label text ("Failed" vs. "Rejected").

**TypeScript enforcement**: Because `STATUS_CONFIG` is typed as `Record<TradeProposal['status'], ...>`, adding `'failed'` to the type union without adding it to the record would cause a compile error. This compile-time check ensures the UI always covers all possible statuses.

## Proposal List Item: Retry Indicator

### Modified Component: `ProposalListItem` in `proposals.index.tsx`

Add a visual indicator for failed proposals in the list view. Since the list item links to a detail page, the retry action itself belongs on the detail page (see below). The list item shows the failure state:

```tsx
function ProposalListItem({ proposal }: { proposal: TradeProposal }) {
    const t = useTranslations();
    const isBuy = proposal.action === 'buy';
    const statusConfig = STATUS_CONFIG[proposal.status];
    const hasWarnings = proposal.warnings && proposal.warnings.length > 0;
    const isFailed = proposal.status === 'failed';

    return (
        <Link
            to="/session/proposals/$proposalId"
            params={{ proposalId: proposal.id }}
            className="block"
        >
            <Card className="transition-colors hover:bg-accent/50">
                <CardContent className="flex items-center gap-4 py-3">
                    {/* ... existing action icon, symbol, confidence ... */}

                    {hasWarnings && (
                        <Badge variant="warning" className="text-xs">
                            {t('common.warning')}
                        </Badge>
                    )}

                    <span className="flex-1 truncate text-sm text-muted-foreground">
                        {proposal.rationale}
                    </span>

                    <Badge variant={statusConfig.variant} className="text-xs">
                        {t(statusConfig.key)}
                    </Badge>

                    {isFailed && (
                        <Badge variant="outline" className="text-xs">
                            {t('proposal.retryAvailable')}
                        </Badge>
                    )}

                    {/* ... existing timestamp ... */}
                </CardContent>
            </Card>
        </Link>
    );
}
```

The "Retry Available" badge is a subtle secondary indicator that communicates the proposal can be acted on, directing the user to click through to the detail page.

## Visual Design

### Status Badge Colors

| Status | Variant | Appearance | Semantics |
|--------|---------|------------|-----------|
| `pending` | `warning` | Yellow/amber | Awaiting user decision |
| `approved` | `success` | Green | User approved (transient) |
| `rejected` | `destructive` | Red | User declined |
| `expired` | `secondary` | Gray | Timed out |
| `executed` | `success` | Green | Order placed |
| `failed` | `destructive` | Red | Execution error, retryable |

The `'failed'` badge is red like `'rejected'`, but the label text ("Failed") distinguishes it. Users learn to associate red badges with proposals that had negative outcomes.

### Retry Button

- Uses the default (primary) button variant -- not destructive, because retrying is a constructive action
- Disabled state while the retry RPC is in flight
- Loading text changes to "Retrying..." during the request

### Error Message Display

When a proposal is `'failed'`, the error message from the broker is available in the response `message` field. The UI can display this in an alert or tooltip to help the user understand why the execution failed and whether a retry is likely to succeed.
