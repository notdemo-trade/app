# Phase 25: Fix Broken Outcome Distribution — Part 1: Spec

## Overview

The outcome distribution system -- responsible for feeding resolved trade results back into the debate and pipeline orchestrators so they can update persona scores, pipeline scores, and confidence calibration -- is completely non-functional. Every outcome that resolves passes an invalid session ID to the orchestrator agents, causing all scoring queries to match zero rows and silently return.

This bug was introduced when `resolveOrchestratorSessionId()` was implemented as a stub that constructs a `userId:symbol` string instead of returning the actual debate/pipeline session UUID. The method has a comment acknowledging this: `// For now, use the convention: {userId}:{symbol}`.

### Why This Matters

- **Persona scoring is dead**: `persona_scores` (win rate, Sharpe ratio, calibration) are never populated. The `getPersonaScores()` callable always returns empty arrays.
- **Pipeline scoring is dead**: `pipeline_scores` and `pipeline_outcomes` are never populated.
- **Confidence dampening never calibrates**: In `debate-orchestrator-agent.ts:224-227`, `applyConfidenceDampening()` is called with `score = null` for every persona (no score data exists), so the default multiplier 1.0 is always used. The system cannot learn from its mistakes.
- **Pattern detection is inactive**: `persona_patterns` (best/worst symbol tracking, action bias patterns) are never updated because `updatePatterns()` is never reached -- it runs after `recomputeScores()` which is only called from `recordPersonaOutcome()`.
- **Silent failure**: The `catch` block in `distributeOutcome()` (line 1092-1094) swallows all errors with a comment "Non-critical: outcome distribution failure shouldn't break tracking." This means the bug produces zero error logs or user-visible indicators.

## Context & Background

### How Orchestrator Session IDs Work

Each time the `SessionAgent` triggers an analysis cycle for a symbol, it calls either `debate.runDebate()` or `pipeline.runPipeline()` on the corresponding orchestrator agent. Inside these methods:

**Debate orchestrator** (`debate-orchestrator-agent.ts:166-167`):
```ts
const sessionId = crypto.randomUUID();
// e.g., "a3f2b1c4-5678-9abc-def0-123456789abc"
```

This UUID is used as the primary key for the `debate_sessions` table and the `session_id` column in `persona_analyses`. All persona analyses for that debate are tagged with this UUID.

**Pipeline orchestrator** (`pipeline-orchestrator-agent.ts:120-121`):
```ts
const sessionId = crypto.randomUUID();
// e.g., "b4c3d2e1-6789-0abc-def1-234567890bcd"
```

This UUID is used as the primary key for `pipeline_sessions` and `pipeline_steps`.

### The Return Path

Both orchestrators return the session object in their results:

- `RunDebateResult.session.id` -- the UUID from `debate_sessions`
- `RunPipelineResult.session.id` -- the UUID from `pipeline_sessions`

The `SessionAgent` currently receives these results in `runDebateAnalysis()` (line 501-514) and `runPipelineAnalysis()` (line 538-550), but **discards the session ID**. It extracts only the consensus/proposal and ignores `result.session.id`.

### The Broken Link

Later, when a trade outcome resolves, `createOutcomeTracking()` needs to store the orchestrator session ID so that `distributeOutcome()` can pass it back to the orchestrator. Instead of reading a stored UUID, it calls `resolveOrchestratorSessionId()`:

```ts
// session-agent.ts:958-963
private resolveOrchestratorSessionId(proposal: TradeProposal, _mode: string): string {
    const userId = this.name;
    return `${userId}:${proposal.symbol}`;
}
```

This produces values like `"usr_abc123:AAPL"` -- which bear no resemblance to the UUID `"a3f2b1c4-5678-..."` stored in the orchestrator's tables.

### Downstream Failure Chain

```
resolveOrchestratorSessionId() returns "usr_abc123:AAPL"
  |
  +-> stored in proposal_outcomes.orchestrator_session_id
  |
  +-> distributeOutcome() reads it and passes to:
       |
       +-> debate.recordPersonaOutcome(proposalId, "usr_abc123:AAPL", outcome)
       |     |
       |     +-> SELECT ... FROM persona_analyses WHERE session_id = "usr_abc123:AAPL"
       |     +-> 0 rows -> no persona_outcomes inserted -> no recomputeScores() called
       |
       +-> pipeline.recordStepOutcome(proposalId, "usr_abc123:AAPL", outcome)
             |
             +-> SELECT ... FROM pipeline_sessions WHERE id = "usr_abc123:AAPL"
             +-> 0 rows -> early return -> no pipeline_outcomes inserted
```

---

## Root Cause

The `resolveOrchestratorSessionId()` method was implemented as a placeholder stub during the initial outcome tracking implementation and was never connected to actual session data. The method's `_mode` parameter (prefixed with underscore to suppress unused-variable warnings) further confirms it was intended to be completed later.

The actual session UUID is created inside the orchestrator agents (`crypto.randomUUID()`) and returned in the `RunDebateResult.session.id` / `RunPipelineResult.session.id`, but the `SessionAgent` never persists this value. The data is available in the analysis flow but lost before the outcome tracking flow needs it.

## Impact Assessment

### Severity: High

Every resolved trade outcome fails to update scoring data. The system has been accumulating proposals and outcomes but never closing the feedback loop.

### Affected Components

| Component | Status | Effect |
|-----------|--------|--------|
| `persona_outcomes` table (DO SQLite) | Empty | No per-persona outcome records |
| `persona_scores` table (DO SQLite) | Empty | No aggregated scores (win rate, Sharpe, calibration) |
| `persona_patterns` table (DO SQLite) | Empty | No pattern detection (best/worst symbols, action bias) |
| `pipeline_outcomes` table (DO SQLite) | Empty | No per-pipeline outcome records |
| `pipeline_scores` table (DO SQLite) | Empty | No aggregated pipeline scores |
| Confidence dampening | Always 1.0 | Every persona analysis uses raw confidence, no historical adjustment |
| Persona comparison in consensus | All nulls | Moderator sees no win rates or performance data for any persona |
| `getPersonaScores()` callable | Empty array | Any UI/API consuming persona performance data gets nothing |
| `getPipelineScores()` callable | Empty array | Any UI/API consuming pipeline performance data gets nothing |

### Data Recovery

Existing `proposal_outcomes` rows have incorrect `orchestrator_session_id` values. However, the `thread_id` on each proposal links to a `discussion_threads` row, which records the `symbol` and `orchestration_mode`. The orchestrator agent for that `userId:symbol` stores all historical debate/pipeline sessions with timestamps. A backfill script could potentially correlate `proposal_outcomes.created_at` with the closest `debate_sessions.started_at` or `pipeline_sessions.started_at` to retroactively fix session IDs. This is a potential future enhancement, not part of this fix.

## Goals

1. **Store the orchestrator session UUID alongside the trade proposal** at the time of analysis, before the proposal enters the pending/approval flow
2. **Read the stored UUID during outcome tracking** instead of calling `resolveOrchestratorSessionId()`
3. **Remove `resolveOrchestratorSessionId()`** entirely -- no more stub method
4. **Unblock the full scoring pipeline**: `persona_outcomes` -> `persona_scores` -> confidence dampening, and `pipeline_outcomes` -> `pipeline_scores`
5. **Minimal change surface**: Modify only the data flow in `SessionAgent` and the `trade_proposals` table schema. No changes to orchestrator agents, API endpoints, or UI components.

## Non-Goals

- **Backfill historical data**: Existing `proposal_outcomes` with wrong session IDs are not retroactively fixed. A separate script could be written later.
- **Fix the DO name mismatch in `distributeOutcome()`**: The `distributeOutcome()` method uses `getAgentByName(this.env.DebateOrchestratorAgent, userId)` (just `userId`), while `runDebateAnalysis()` uses `getAgentByName(this.env.DebateOrchestratorAgent, userId + ':' + symbol)`. Since Durable Objects are addressed by name, these reach different DO instances. However, the `persona_analyses` data lives in the `userId:symbol` DO, not the `userId` DO. This is a separate issue to be tracked independently -- fixing the session ID alone does not resolve it. **This must be addressed as a companion fix.**
- **Modify orchestrator agents**: `recordPersonaOutcome()` and `recordStepOutcome()` are correct once they receive valid session IDs. No changes needed.
- **Add observability/alerting**: The silent `catch` in `distributeOutcome()` should eventually log warnings, but that is a separate improvement.

## Alternatives Considered

### Alternative A: Look up session ID by timestamp correlation at distribution time

Instead of storing the session ID on the proposal, query the orchestrator's `debate_sessions` or `pipeline_sessions` table at outcome resolution time to find the session that was active around the proposal's creation timestamp.

**Rejected**: Brittle -- relies on timestamp proximity, could match the wrong session if multiple analyses run in quick succession for the same symbol. Also requires a new RPC call to the orchestrator at resolution time.

### Alternative B: Store session ID in `proposal_outcomes` only (not `trade_proposals`)

Wait until `createOutcomeTracking()` runs and query the orchestrator at that time for the active session.

**Rejected**: By the time a proposal is approved and executed, the orchestrator may have run new sessions for the same symbol. The session that produced the proposal is no longer "active." The session ID must be captured at analysis time.

### Alternative C: Use the discussion thread ID as a proxy

The `discussion_threads.id` is unique per analysis cycle and could serve as a lookup key.

**Rejected**: The thread ID is a `SessionAgent` concept (DO SQLite), not stored in the orchestrator's tables. Adding a thread_id column to `debate_sessions` / `pipeline_sessions` would require orchestrator changes, which violates the minimal-change-surface goal.

## Open Questions

1. **DO name mismatch in `distributeOutcome()`**: As noted in Non-Goals, `distributeOutcome()` creates the debate DO with name `userId` but analysis uses `userId:symbol`. This means `recordPersonaOutcome()` runs on a different DO instance that has no `persona_analyses` data. **This must be fixed alongside the session ID issue for the scoring pipeline to actually work.** The recommended fix is to store the `symbol` from the proposal and use `userId:symbol` as the DO name in `distributeOutcome()`. This is scoped as part of the implementation.

2. **Pipeline proposal storage path**: The pipeline path in `runPipelineAnalysis()` uses `this.storeProposal(proposal)` directly (line 554) rather than going through `this.createProposal()`. This means the pipeline path needs a separate integration point for storing the session ID. The implementation must handle both code paths.
