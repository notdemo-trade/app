# Phase 25: Fix Broken Outcome Distribution — Part 2: Data

## DO SQLite Schema Change

### Modified Table: `trade_proposals`

Add a nullable `orchestrator_session_id` column to store the UUID returned by the debate or pipeline orchestrator after analysis.

**Current DDL** (`session-agent.ts:1289-1312`):

```sql
CREATE TABLE IF NOT EXISTS trade_proposals (
    id                TEXT PRIMARY KEY,
    thread_id         TEXT NOT NULL,
    symbol            TEXT NOT NULL,
    action            TEXT NOT NULL,
    confidence        REAL NOT NULL,
    rationale         TEXT NOT NULL,
    entry_price       REAL,
    target_price      REAL,
    stop_loss         REAL,
    qty               REAL,
    notional          REAL,
    position_size_pct REAL NOT NULL,
    risks             TEXT NOT NULL DEFAULT '[]',
    warnings          TEXT NOT NULL DEFAULT '[]',
    expires_at        INTEGER NOT NULL,
    status            TEXT NOT NULL DEFAULT 'pending',
    created_at        INTEGER NOT NULL,
    decided_at        INTEGER,
    order_id          TEXT,
    filled_qty        REAL,
    filled_avg_price  REAL,
    outcome_status    TEXT NOT NULL DEFAULT 'none'
)
```

**New column**:

```sql
orchestrator_session_id TEXT
```

Nullable because:
- Existing proposals created before this fix have no session ID stored
- `createOutcomeTracking()` must handle the null case gracefully (fall back to the current broken behavior, or skip distribution -- either way no worse than today)

## Migration

### DO SQLite Migration in `migrateTradeProposals()`

The existing `migrateTradeProposals()` method (`session-agent.ts:1374-1393`) uses the `PRAGMA table_info` pattern to conditionally add columns to existing tables. This is the established pattern for DO SQLite migrations in this codebase.

**Current migration method:**

```ts
private migrateTradeProposals(): void {
    const columns = this.sql<{ name: string }>`PRAGMA table_info(trade_proposals)`;
    const columnNames = new Set(columns.map((c) => c.name));

    if (!columnNames.has('order_id')) {
        this.sql`ALTER TABLE trade_proposals ADD COLUMN order_id TEXT`;
    }
    if (!columnNames.has('filled_qty')) {
        this.sql`ALTER TABLE trade_proposals ADD COLUMN filled_qty REAL`;
    }
    if (!columnNames.has('filled_avg_price')) {
        this.sql`ALTER TABLE trade_proposals ADD COLUMN filled_avg_price REAL`;
    }
    if (!columnNames.has('outcome_status')) {
        this.sql`ALTER TABLE trade_proposals ADD COLUMN outcome_status TEXT NOT NULL DEFAULT 'none'`;
    }
    if (!columnNames.has('warnings')) {
        this.sql`ALTER TABLE trade_proposals ADD COLUMN warnings TEXT NOT NULL DEFAULT '[]'`;
    }
}
```

**Add to the end of this method:**

```ts
if (!columnNames.has('orchestrator_session_id')) {
    this.sql`ALTER TABLE trade_proposals ADD COLUMN orchestrator_session_id TEXT`;
}
```

### Why not update the `CREATE TABLE` DDL?

The `CREATE TABLE IF NOT EXISTS` statement only runs when the table does not exist (new DO instances). Existing DO instances already have the table without the new column. The migration method handles the delta for existing instances. Both paths must be updated:

1. **`CREATE TABLE IF NOT EXISTS` DDL**: Add `orchestrator_session_id TEXT` to the column list for new DO instances
2. **`migrateTradeProposals()`**: Add the conditional `ALTER TABLE` for existing DO instances

## Data Compatibility

### Forward Compatibility

- New proposals created after the fix will have `orchestrator_session_id` populated with a valid UUID
- `createOutcomeTracking()` will read this field and store it correctly in `proposal_outcomes`

### Backward Compatibility

- Existing proposals created before the fix will have `orchestrator_session_id = NULL`
- `createOutcomeTracking()` must handle this: if `proposal.orchestratorSessionId` is null, fall back to the current `resolveOrchestratorSessionId()` behavior (which produces an invalid ID, but is no worse than today)
- Alternatively, skip outcome distribution entirely for null-session proposals -- this is cleaner and avoids pointless queries that return zero rows

### Recommended Approach: Skip distribution for null sessions

```ts
if (!proposal.orchestratorSessionId) {
    // Pre-fix proposal -- no valid session ID available, skip distribution
    return;
}
```

This is safe because proposals without a session ID were never going to produce valid scoring data anyway.
