# Phase 22: Technical Analysis Configuration Profiles — Part 4: API & Server Functions

## Overview

REST API endpoints on data-service (Hono) for managing TA config, plus TanStack Start server functions on user-application for settings page data access.

## REST API Endpoints (data-service)

### Route Registration


### Handler File

```ts
// apps/data-service/src/hono/handlers/ta-config.ts

import { Hono } from 'hono';
import { zValidator } from '@hono/zod-validator';
import {
  TechnicalAnalysisConfigSchema,
  TechnicalAnalysisConfigUpdateSchema,
  PresetNameSchema,
} from '@repo/data-ops/ta-config';
import { getTaConfigService, updateTaConfigService, applyPresetService, resetTaConfigService } from '../services/ta-config';
import { authMiddleware } from '../middleware/auth';
import { resultToResponse } from '../utils/result-to-response';
import { z } from 'zod';

const taConfigRoutes = new Hono<{ Bindings: Env }>();

// All endpoints require authentication
taConfigRoutes.use('*', authMiddleware());

/**
 * GET /api/technical-config
 * Returns user's config or defaults if no custom config exists.
 */
taConfigRoutes.get('/', async (c) => {
  const userId = c.get('userId');
  const result = await getTaConfigService(userId);
  return resultToResponse(c, result);
});

/**
 * PUT /api/technical-config
 * Full or partial update of user's config. Validates all fields.
 * Creates row if missing, updates if exists.
 */
taConfigRoutes.put(
  '/',
  zValidator('json', TechnicalAnalysisConfigSchema),
  async (c) => {
    const userId = c.get('userId');
    const config = c.req.valid('json');
    const result = await updateTaConfigService(userId, config);
    return resultToResponse(c, result);
  },
);

/**
 * POST /api/technical-config/preset/:name
 * Apply a preset profile. Overwrites all fields with preset values.
 */
taConfigRoutes.post(
  '/preset/:name',
  zValidator('param', z.object({ name: PresetNameSchema })),
  async (c) => {
    const userId = c.get('userId');
    const { name } = c.req.valid('param');
    const result = await applyPresetService(userId, name);
    return resultToResponse(c, result);
  },
);

/**
 * POST /api/technical-config/reset
 * Delete user's config row, reverting to defaults.
 */
taConfigRoutes.post('/reset', async (c) => {
  const userId = c.get('userId');
  const result = await resetTaConfigService(userId);
  return resultToResponse(c, result);
});

export { taConfigRoutes };
```

## Service Layer (data-service)

```ts
// apps/data-service/src/hono/services/ta-config.ts

import {
  getTaConfig,
  upsertTaConfig,
  deleteTaConfig,
  PRESET_CONFIGS,
  DEFAULT_TA_CONFIG,
  type TechnicalAnalysisConfig,
  type PresetName,
} from '@repo/data-ops/ta-config';
import { ok, err, type Result } from '../types/result';
import { AppError } from '../types/result';

export async function getTaConfigService(
  userId: string,
): Promise<Result<TechnicalAnalysisConfig>> {
  try {
    const config = await getTaConfig(userId);
    return ok(config);
  } catch (error) {
    return err(new AppError('Failed to fetch technical analysis config', 500, 'INTERNAL_ERROR'));
  }
}

export async function updateTaConfigService(
  userId: string,
  config: TechnicalAnalysisConfig,
): Promise<Result<TechnicalAnalysisConfig>> {
  try {
    const updated = await upsertTaConfig(userId, config);
    return ok(updated);
  } catch (error) {
    return err(new AppError('Failed to update technical analysis config', 500, 'INTERNAL_ERROR'));
  }
}

export async function applyPresetService(
  userId: string,
  presetName: PresetName,
): Promise<Result<TechnicalAnalysisConfig>> {
  const preset = PRESET_CONFIGS[presetName];
  if (!preset) {
    return err(new AppError(`Unknown preset: ${presetName}`, 400, 'INVALID_PRESET'));
  }

  try {
    const updated = await upsertTaConfig(userId, preset);
    return ok(updated);
  } catch (error) {
    return err(new AppError('Failed to apply preset', 500, 'INTERNAL_ERROR'));
  }
}

export async function resetTaConfigService(
  userId: string,
): Promise<Result<TechnicalAnalysisConfig>> {
  try {
    await deleteTaConfig(userId);
    return ok({ ...DEFAULT_TA_CONFIG });
  } catch (error) {
    return err(new AppError('Failed to reset config', 500, 'INTERNAL_ERROR'));
  }
}
```

## Server Functions (user-application)

For the settings page, use server functions with direct DB access (Pattern 1: Server Fn -> data-ops -> DB).

```ts
// apps/user-application/src/core/functions/ta-config/direct.ts

import { createServerFn } from '@tanstack/react-start';
import { z } from 'zod';
import {
  getTaConfig,
  upsertTaConfig,
  deleteTaConfig,
  PRESET_CONFIGS,
  DEFAULT_TA_CONFIG,
  TechnicalAnalysisConfigSchema,
  PresetNameSchema,
  type TechnicalAnalysisConfig,
} from '@repo/data-ops/ta-config';
import { protectedFunctionMiddleware } from '../../middleware/auth';

export const getUserTaConfig = createServerFn({ method: 'GET' })
  .middleware([protectedFunctionMiddleware])
  .handler(async ({ context }) => {
    return getTaConfig(context.session.user.id);
  });

export const updateUserTaConfig = createServerFn({ method: 'POST' })
  .middleware([protectedFunctionMiddleware])
  .validator(TechnicalAnalysisConfigSchema)
  .handler(async ({ context, data }) => {
    return upsertTaConfig(context.session.user.id, data);
  });

export const applyTaPreset = createServerFn({ method: 'POST' })
  .middleware([protectedFunctionMiddleware])
  .validator(z.object({ presetName: PresetNameSchema }))
  .handler(async ({ context, data }) => {
    const preset = PRESET_CONFIGS[data.presetName];
    if (!preset) throw new Error(`Unknown preset: ${data.presetName}`);
    return upsertTaConfig(context.session.user.id, preset);
  });

export const resetTaConfig = createServerFn({ method: 'POST' })
  .middleware([protectedFunctionMiddleware])
  .handler(async ({ context }) => {
    await deleteTaConfig(context.session.user.id);
    return { ...DEFAULT_TA_CONFIG };
  });
```

## Frontend Data Access Pattern

The settings page uses `useQuery` + `useMutation` with the server functions:

```ts
// In the settings component
const configQuery = useQuery({
  queryKey: taConfigKeys.detail(),
  queryFn: () => getUserTaConfig(),
});

const updateMutation = useMutation({
  mutationFn: (config: TechnicalAnalysisConfig) => updateUserTaConfig({ data: config }),
  onSuccess: () => queryClient.invalidateQueries({ queryKey: taConfigKeys.all }),
});

const presetMutation = useMutation({
  mutationFn: (presetName: PresetName) => applyTaPreset({ data: { presetName } }),
  onSuccess: () => queryClient.invalidateQueries({ queryKey: taConfigKeys.all }),
});

const resetMutation = useMutation({
  mutationFn: () => resetTaConfig(),
  onSuccess: () => queryClient.invalidateQueries({ queryKey: taConfigKeys.all }),
});
```

## Agent-Side Config Access

When `TechnicalAnalysisAgent` calls `getTaConfig(userId)`, it uses data-ops directly (since the agent runs on `data-service` Worker which has `initDatabase()` called in its entry point). No HTTP call needed -- direct Postgres query.

The same applies to the `AlpacaMarketDataAgent` receiving `cacheFreshnessSec` as a parameter from `TechnicalAnalysisAgent` -- no separate config fetch needed at the data agent level.

### Flow Summary

```
Settings UI (user-application)
  → Server function → data-ops → Postgres (read/write config)

Analysis cycle (data-service agents)
  → TechnicalAnalysisAgent.analyze()
    → getTaConfig(userId) → Postgres (read config)
    → AlpacaMarketDataAgent.fetchBars({ cacheFreshnessSec: config.cacheFreshnessSec })
    → computeTechnicals(symbol, bars, config)
    → detectSignals(indicators, config)

Frontend indicator display (user-application)
  → Server function → getTaConfig(userId) → config for display thresholds
  → Or: config included in analysis result returned to frontend
```
