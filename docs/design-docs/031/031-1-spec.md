# Phase 31: Fix Short Position Handling in Exit Logic -- Part 1: Spec

## Overview

Three related bugs exist in the session agent's exit logic and the debate orchestrator's session builder. All three stem from assumptions that only long positions exist or from missing data propagation. The fixes are localized to two files and require no schema changes, no new endpoints, and no frontend modifications.

### Why This Matters

- **Financial correctness**: Duplicate exit proposals for short positions can result in multiple order submissions for the same position, potentially causing order rejections or over-trading.
- **Outcome tracking integrity**: Misclassified exit reasons corrupt the performance tracking data that feeds persona scoring (Phase 20 M7 memory system). A short position closed at its stop-loss being recorded as `manual_close` means the stop-loss hit rate metric is artificially low and the manual-close rate is artificially high.
- **Debate context loss**: Downstream consumers of `DebateSession` (such as the discussion thread broadcast and session history) lose the ticker symbol, making it impossible to correlate debate results with specific assets.

---

## Context & Background

### Position Sides in the Trading System

The system supports both long and short positions through Alpaca's broker API:

| Position Side | Entry Action | Exit Action | Stop-Loss Triggers When | Target Triggers When |
|---------------|-------------|-------------|------------------------|---------------------|
| Long | `buy` | `sell` | Price drops to/below stop | Price rises to/above target |
| Short | `sell` | `buy` | Price rises to/above stop | Price drops to/below target |

The `ProposalOutcome.action` field records the original entry action: `'buy'` for a long position, `'sell'` for a short position. The exit action is always the inverse.

### Correctly Implemented: `checkExitConditions()`

The `checkExitConditions()` method (`session-agent.ts` lines 1104-1120) already handles both sides correctly:

```ts
private checkExitConditions(
    currentPrice: number,
    proposal: TradeProposal,
    side: 'long' | 'short',
): 'stop_loss' | 'target_hit' | null {
    if (side === 'long') {
        if (proposal.stopLoss !== null && currentPrice <= proposal.stopLoss) return 'stop_loss';
        if (proposal.targetPrice !== null && currentPrice >= proposal.targetPrice)
            return 'target_hit';
    } else {
        // Short position: reversed logic
        if (proposal.stopLoss !== null && currentPrice >= proposal.stopLoss) return 'stop_loss';
        if (proposal.targetPrice !== null && currentPrice <= proposal.targetPrice)
            return 'target_hit';
    }
    return null;
}
```

This method is called from the tracking loop when deciding whether to create an exit proposal. The bugs are in the _downstream_ functions that execute after this check passes.

## Goals

1. **Fix exit proposal dedup** to use the position-side-aware exit action instead of hardcoded `'sell'`.
2. **Fix `determineExitReason()`** to apply the correct price comparison direction based on the original entry action.
3. **Fix `getDebateSession()`** to propagate the symbol from `RunDebateParams` to the returned `DebateSession`.
4. **Zero regressions** on long position behavior -- all existing long-side logic must produce identical results.

## Non-Goals

- **Changing the `checkExitConditions()` method**: Already correct; no changes needed.
- **Adding new exit reason types**: The existing `ExitReason` union (`'stop_loss' | 'target_hit' | 'manual_close' | 'time_exit'`) is sufficient.
- **Modifying the `TradeProposal` schema**: No new fields needed.
- **Backfilling misclassified outcomes**: Existing `proposal_outcomes` rows with incorrect `exit_reason` values are not corrected retroactively. The persona scoring system will self-correct over time as new, correctly classified outcomes accumulate.
- **Adding tests**: This phase documents the bug fixes only. Tests should be written as part of the implementation per the project's "reproduce the bug with a test first" rule.

## Bug Analysis

### Bug 1: Exit Proposal Dedup Only Checks `action = 'sell'`

**Location**: `session-agent.ts` lines 1128-1132

**Current code**:

```ts
// Dedup guard: skip if pending sell proposal already exists for this symbol
const existingPending = this.sql<CountRow>`
    SELECT COUNT(*) as cnt FROM trade_proposals
    WHERE symbol = ${outcome.symbol} AND action = 'sell' AND status = 'pending'`;
if ((existingPending[0]?.cnt ?? 0) > 0) return;
```

**Problem**: The dedup guard checks for pending proposals with `action = 'sell'`, but short position exits use `action = 'buy'`. The correct exit action is computed 3 lines later:

```ts
const exitAction = position.side === 'long' ? 'sell' : 'buy';
```

**Consequence**: For short positions, the dedup query finds zero pending `sell` proposals (because the correct exit would be `buy`), so the guard never triggers. Each tracking cycle (every ~5 minutes) that detects the exit condition creates a new pending exit proposal.

**Why long positions are unaffected**: Long position exits use `action = 'sell'`, which matches the hardcoded `'sell'` in the query.

### Bug 2: `determineExitReason()` Uses Long-Side Logic for All Positions

**Location**: `session-agent.ts` lines 1202-1229

**Current code**:

```ts
private determineExitReason(
    exitOrder: OrderLogEntry | undefined,
    outcome: ProposalOutcome,
): ExitReason {
    if (!exitOrder) return 'manual_close';

    const proposal = this.sql<ProposalRow>`
        SELECT * FROM trade_proposals WHERE id = ${outcome.proposalId}`;
    const row = proposal[0];
    if (!row) return 'manual_close';

    const p = rowToProposal(row);
    if (
        p.stopLoss !== null &&
        exitOrder.filledAvgPrice !== null &&
        exitOrder.filledAvgPrice <= p.stopLoss    // <-- long-side logic
    ) {
        return 'stop_loss';
    }
    if (
        p.targetPrice !== null &&
        exitOrder.filledAvgPrice !== null &&
        exitOrder.filledAvgPrice >= p.targetPrice  // <-- long-side logic
    ) {
        return 'target_hit';
    }
    return 'manual_close';
}
```

**Problem**: For a short position:

- **Stop-loss**: The stop price is _above_ the entry price (e.g., entry at $100, stop at $105). The exit order fills at $105. The check `$105 <= $105` returns `true` -- this _accidentally_ works for the exact stop price but fails for any fill above the stop (e.g., slippage to $106: `$106 <= $105` is `false`).
- **Target-hit**: The target price is _below_ the entry price (e.g., entry at $100, target at $90). The exit order fills at $90. The check `$90 >= $90` returns `true` -- again accidentally works at the exact price but fails for better fills (e.g., $89: `$89 >= $90` is `false`).

The long-side comparisons are inverted for shorts: stop-loss should check `>=` and target should check `<=`.

**Why `checkExitConditions()` is correct but this function is wrong**: `checkExitConditions()` receives a `side` parameter and branches on it. `determineExitReason()` has no `side` parameter and applies long-side logic unconditionally.

### Bug 3: `getDebateSession()` Returns Empty Symbol

**Location**: `debate-orchestrator-agent.ts` lines 453-469

**Current code**:

```ts
private getDebateSession(
    sessionId: string,
    analyses: PersonaAnalysis[],
    debateRounds: DebateRound[],
    consensus: ConsensusResult,
): DebateSession {
    return {
        id: sessionId,
        symbol: '',           // <-- BUG: hardcoded empty string
        status: 'completed',
        initialAnalyses: analyses,
        debateRounds,
        consensus,
        startedAt: Date.now(),
        completedAt: Date.now(),
    };
}
```

**Caller** (line 270):

```ts
const session = this.getDebateSession(sessionId, analyses, debateRounds, consensus);
```

**Problem**: `params.symbol` is available in `runDebate()` but is not passed to `getDebateSession()`. The `DebateSession` interface requires `symbol: string`, and the empty string satisfies the type checker but provides no useful information.

**Consequence**: Any consumer of `RunDebateResult.session` (e.g., the discussion thread builder in `SessionAgent`, or debug logging) sees an empty symbol.
