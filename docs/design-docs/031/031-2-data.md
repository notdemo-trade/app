# Phase 31: Fix Short Position Handling in Exit Logic -- Part 2: Data

## Schema Changes

**None.** This phase modifies only runtime logic. No PostgreSQL migrations, no DO SQLite DDL changes, no new Zod schemas, and no new types are introduced.

---

## Relevant Data Structures

The following existing types and tables are referenced by the bug fixes. They are documented here for context; none are modified.

### `ProposalOutcome` (TypeScript interface)

**File**: `packages/data-ops/src/agents/memory/types.ts`

```ts
export interface ProposalOutcome {
    id: string;
    proposalId: string;
    threadId: string;
    orchestrationMode: 'debate' | 'pipeline';
    orchestratorSessionId: string;
    symbol: string;
    action: 'buy' | 'sell';       // original entry action: buy = long, sell = short
    entryPrice: number;
    entryQty: number;
    status: 'tracking' | 'resolved';
    exitPrice: number | null;
    exitReason: ExitReason | null;
    realizedPnl: number | null;
    realizedPnlPct: number | null;
    holdingDurationMs: number | null;
    resolvedAt: number | null;
    createdAt: number;
}
```

**Key field for fix**: `action` tells us the position side. If `action === 'buy'`, the position is long. If `action === 'sell'`, the position is short. This field is already available in `determineExitReason()` via the `outcome` parameter.

### `ExitReason` (Type alias)

**File**: `packages/data-ops/src/agents/memory/types.ts`

```ts
export type ExitReason = 'stop_loss' | 'target_hit' | 'manual_close' | 'time_exit';
```

No new values are added. The fix ensures `'stop_loss'` and `'target_hit'` are correctly assigned for short positions.

## DO SQLite Tables (No Changes)

### `trade_proposals` (session-agent DO)

The dedup query in Bug 1 reads from this table. The table schema is unchanged; only the query's `WHERE` clause is modified.

```sql
CREATE TABLE IF NOT EXISTS trade_proposals (
    id              TEXT PRIMARY KEY,
    thread_id       TEXT NOT NULL,
    symbol          TEXT NOT NULL,
    action          TEXT NOT NULL,        -- 'buy' or 'sell'
    confidence      REAL NOT NULL,
    rationale       TEXT NOT NULL,
    entry_price     REAL,
    target_price    REAL,
    stop_loss       REAL,
    qty             REAL,
    notional        REAL,
    position_size_pct REAL NOT NULL,
    risks           TEXT NOT NULL,
    warnings        TEXT NOT NULL DEFAULT '[]',
    expires_at      INTEGER NOT NULL,
    status          TEXT NOT NULL DEFAULT 'pending',
    created_at      INTEGER NOT NULL,
    decided_at      INTEGER,
    order_id        TEXT,
    filled_qty      REAL,
    filled_avg_price REAL,
    outcome_status  TEXT NOT NULL DEFAULT 'none'

```

### `proposal_outcomes` (session-agent DO)

The `determineExitReason()` result is written to the `exit_reason` column of this table. The table schema is unchanged; only the value written is corrected for short positions.
