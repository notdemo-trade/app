# Phase 31: Fix Short Position Handling in Exit Logic -- Part 5: UI

## No UI Changes

This phase does not modify any frontend components, routes, server functions, or i18n keys.

---

## Frontend Impact Assessment

### Proposal List (Session Page)

The session page displays pending and historical proposals via the `getProposals()` callable. The fix changes _how many_ exit proposals are created for short positions (one instead of many), not how they are displayed.

**Observable difference**: For short positions that hit stop-loss or target, users will see a single exit proposal instead of potentially many duplicate proposals. This is the correct and expected behavior -- it matches what users already see for long positions.

### Outcome History

The outcomes view displays `exit_reason` values. With the fix, short position outcomes will show accurate `stop_loss` and `target_hit` labels instead of incorrectly showing `manual_close`.

**Observable difference**: The exit reason label changes from `manual_close` to the correct value. This requires no UI code changes -- the component already renders all `ExitReason` values.

### Debate Session Display

The discussion thread UI receives debate session data via WebSocket broadcast. The `symbol` field is used in thread headers and metadata display.

**Observable difference**: Debate threads will show the actual ticker symbol instead of an empty string. This requires no UI code changes -- the component already renders the `symbol` field.
