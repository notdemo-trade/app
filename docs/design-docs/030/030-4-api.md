# Phase 030: Schedule Lifecycle Management -- Part 4: API

## No API Changes

This is an internal bug fix. No REST endpoints, request/response schemas, or Hono handlers are added, modified, or removed.

---

## Affected RPC Callables

The fix modifies the behavior of existing `@callable()` methods on `SessionAgent`. These are invoked via the Durable Object RPC protocol from the frontend's WebSocket connection, not via REST API.

### `start()` Callable

| Aspect | Before | After |
|--------|--------|-------|
| Signature | `start(): Promise<SessionState>` | `start(): Promise<SessionState>` (unchanged) |
| Return value | `SessionState` with `enabled: true` | Same (unchanged) |
| Side effect | Creates 1 analysis + 1 outcome tracking schedule (additive) | Cancels all existing schedules, then creates 1 analysis + 1 outcome tracking schedule |
| Idempotent? | No (each call adds a schedule) | Yes (each call produces the same 2-schedule state) |

### `stop()` Callable

| Aspect | Before | After |
|--------|--------|-------|
| Signature | `stop(): Promise<SessionState>` | `stop(): Promise<SessionState>` (unchanged) |
| Return value | `SessionState` with `enabled: false` | Same (unchanged) |
| Side effect | Cancels only `runScheduledCycle` schedules | Cancels ALL schedules |
| Idempotent? | Partially (analysis schedules removed, outcome leaks) | Yes (always results in 0 schedules) |

### `getStatus()` Callable

Not modified. Returns `SessionState` which does not include schedule information. Schedule count is not exposed in the session state -- this is internal DO state.

### `updateConfig()` Callable

Not modified. Still calls `rescheduleAnalysisCycle()` when the interval changes. Does not interact with outcome tracking schedules.

---

## No Frontend Server Function Changes

The server functions in `apps/user-application/src/core/functions/session/` that invoke `start()` and `stop()` via the WebSocket connection are not modified. The fix is transparent to the frontend -- the same RPC methods are called with the same signatures and return the same response shape.

---

## WebSocket Protocol

The session WebSocket connection (managed by `session-connection.ts` in the frontend) invokes `start()` and `stop()` as RPC calls. The message format is unchanged:

```ts
// Frontend call (unchanged)
await agent.start();   // RPC -> SessionAgent.start()
await agent.stop();    // RPC -> SessionAgent.stop()
```

The frontend does not need to know about schedule internals. It calls `start()`/`stop()` and receives `SessionState` back.

---

## Verification via API

While there are no API changes, the fix can be verified through indirect observation:

### 1. Schedule Count (DO Internal)

After the fix, calling `getSchedules()` inside the DO should return:
- 2 schedules after `start()`
- 0 schedules after `stop()`

This is not directly observable via API but can be verified via:
- Adding a temporary `@callable()` debug method (dev only)
- Checking Cloudflare dashboard DO metrics
- Observing broker API call patterns in logs

### 2. Broker API Call Pattern

After `stop()`, there should be zero broker API calls from the session. Before the fix, `runOutcomeTrackingCycle()` continues calling `broker.getClock()` every 5 minutes. This can be verified by:
- Monitoring Alpaca API logs for calls from the session user
- Checking `lastError` in session state (if broker calls fail, errors accumulate even when stopped)
- Checking Cloudflare Worker analytics for outbound fetch requests after stopping

### 3. Session State

The `enabled` field in `SessionState` should accurately reflect whether schedules are running:
- `enabled: true` means exactly 2 schedules active
- `enabled: false` means exactly 0 schedules active

Before the fix, `enabled: false` did not guarantee 0 schedules.
