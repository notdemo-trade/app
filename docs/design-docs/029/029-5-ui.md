# Phase 29: Risk Management Config Enforcement — Part 5: UI

## No UI Changes

This phase does not add or modify any frontend components, routes, or styles. All enforcement logic is server-side within the `SessionAgent` Durable Object.

The existing UI surfaces are sufficient to communicate enforcement outcomes:

---

## Existing UI Coverage

### Trade Approval Rejection

When a user approves a proposal and enforcement blocks execution, `approveProposal()` returns `{ status: 'error', message: '...' }`. The session UI already renders error responses from proposal actions:

**Current behavior**: If `broker.placeOrder()` throws, the error message is returned and displayed.

**After this phase**: Enforcement rejections return the same response shape. The error message is human-readable (e.g., "Maximum 3 positions reached. Close a position first.") and will display in the existing error handling path.

No frontend changes are required for this to work.

### Config Settings Page

The trading config settings page (`/settings/trading`) already displays all risk management fields. Users can view and edit `maxNotionalPerTrade`, `maxPositions`, `maxPositionValue`, `maxDailyLossPct`, `cooldownMinutesAfterLoss`, `tradingHoursOnly`, `allowShortSelling`, `tickerBlacklist`, and `tickerAllowlist`.

After this phase, these fields go from "display-only decorations" to "actively enforced settings." No visual changes are needed -- the settings page already communicates what each field does. The difference is that the system now respects the configured values.

### Session Status Display

The session dashboard shows `SessionState` including `lastCycleAt` and `cycleCount`. When pre-analysis guards skip a cycle, the state still updates `lastCycleAt` (the cycle ran, it just had no work to do). The `threadIds` array in the cycle result is empty, so no new threads appear. This is visually identical to a cycle where the LLM recommended "hold" for all symbols.

## Future Consideration: Enforcement Log

A future phase could add visibility into enforcement decisions. This is NOT part of Phase 29 but is documented here for reference.

### Option A: Enforcement Events in SessionState

Add a `lastEnforcementEvent` field to `SessionState`:

```ts
interface EnforcementEvent {
  type: 'guard_skip' | 'trade_blocked';
  guard: string;      // 'market_hours' | 'daily_loss' | 'cooldown' | 'max_positions' | ...
  message: string;
  timestamp: number;
}

interface SessionState {
  // ... existing fields ...
  lastEnforcementEvent: EnforcementEvent | null;
}
```

The frontend could display this as a subtle banner or log entry in the session panel.

### Option B: Enforcement History Table in DO SQLite

Add an `enforcement_log` table:

```sql
CREATE TABLE enforcement_log (
  id TEXT PRIMARY KEY,
  guard TEXT NOT NULL,
  action TEXT NOT NULL,    -- 'skip_analysis' | 'block_trade'
  message TEXT NOT NULL,
  proposal_id TEXT,        -- only for trade blocks
  created_at INTEGER NOT NULL
);
```

The session UI could display recent enforcement events in a dedicated section.

### Option C: Inline Thread Messages

When a guard blocks analysis for a symbol or blocks a trade, add a `DiscussionMessage` with `sender: { type: 'system' }` and `phase: 'execution'` to the relevant thread. This would make enforcement visible in the existing thread timeline.

**Recommendation for future phase**: Option C provides the best UX with minimal new infrastructure. Enforcement events appear naturally in the thread context where the user expects to see execution status.

## i18n Considerations

The enforcement error messages returned by `handleTradeDecision()` are currently in English only. They are plain strings, not i18n keys. This is consistent with other error messages in `SessionAgent` (e.g., "Proposal not found", "Trade rejected by user").

If the frontend needs to localize enforcement messages in a future phase, the messages could be replaced with structured error codes:

```ts
// Future: structured error for i18n
return {
  status: 'error',
  code: 'MAX_POSITIONS_REACHED',
  params: { maxPositions: effectiveConfig.maxPositions },
  message: `Maximum ${effectiveConfig.maxPositions} positions reached.`, // fallback
};
```

This is NOT part of Phase 29 and would require changes to both the return type and the frontend error rendering.
