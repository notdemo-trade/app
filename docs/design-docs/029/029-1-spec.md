# Phase 29: Risk Management Config Enforcement — Part 1: Spec

## Overview

The `EffectiveConfig` type in `packages/data-ops/src/agents/session/types.ts` declares nine risk-management fields. The `resolveEffectiveConfig()` function in `resolve-config.ts` carefully merges values from the PostgreSQL `user_trading_config` table, the DO SQLite `session_config`, the active strategy template, and hardcoded defaults. Users can configure these values through the trading config settings UI.

None of these resolved values are enforced anywhere in the execution pipeline.

The `triggerAnalysis()` method iterates the watchlist without checking market hours, ticker filters, daily loss limits, or cooldown periods. The `handleTradeDecision()` method places orders without checking notional caps, position counts, position value limits, or short-selling permissions. The `createProposal()` method adds a warning for short-selling situations but does not block proposal creation.

This means a user who sets `maxPositions: 3` and expects the system to stop at 3 positions will instead accumulate unlimited positions. A user who disables short selling will still receive sell proposals for symbols they do not hold, and if they approve those proposals, the orders will be submitted to the broker.

### Why This Matters

- **False sense of safety**: Users configure risk limits through the UI, see them reflected in the config, and reasonably assume they are enforced. They are not.
- **Financial risk**: Without notional caps, a position-sizing calculation error or an aggressive LLM recommendation could create an outsized trade. Without a daily loss circuit breaker, a series of bad trades on a volatile day can compound losses.
- **Wasted resources**: Analysis cycles running outside market hours consume LLM API credits and generate proposals that expire before the market opens. Analyzing blacklisted tickers wastes the same resources.
- **Incorrect behavior**: The `allowShortSelling: false` setting should prevent sell orders for unheld symbols, but the current implementation only adds a non-blocking warning to the proposal's `warnings` array.

## Context & Background

### Pipeline Architecture

The analysis-to-execution pipeline flows through three methods in `SessionAgent`:

```
runScheduledCycle()
  -> triggerAnalysis()
       -> for each symbol in watchlist:
            runAnalysisForSymbol(symbol, config, portfolioCtx)
              -> runDebateAnalysis() or runPipelineAnalysis()
                -> createProposal()  [stores pending proposal in DO SQLite]

User approves proposal via WebSocket RPC:
  approveProposal(proposalId)
    -> handleTradeDecision(proposalId, approved=true)
       -> broker.placeOrder()
```

### Where Enforcement Belongs

Risk checks fall into two categories based on where they should be enforced:

**Pre-analysis guards** (before spending LLM credits):
- `tradingHoursOnly` -- skip analysis when market is closed
- `tickerBlacklist` / `tickerAllowlist` -- filter watchlist before iterating
- `maxDailyLossPct` -- circuit breaker, stop all analysis for the day
- `cooldownMinutesAfterLoss` -- pause analysis after a losing trade resolves

**Pre-execution guards** (before placing a broker order):
- `maxNotionalPerTrade` -- cap the dollar amount of a single order
- `maxPositions` -- reject buy orders when position count is at limit
- `maxPositionValue` -- cap the dollar amount going into a single position
- `allowShortSelling` -- block sell orders for unheld symbols

## Goals

1. **Enforce all nine risk fields** at the correct pipeline stage so that configured limits are respected.
2. **Block, not warn, on policy violations**: `allowShortSelling: false` must prevent order execution, not just add a warning to a proposal that a user can still approve.
3. **Fail with clear messages**: Every enforcement rejection must produce a message explaining what limit was hit and what the configured value is.
4. **Minimize broker API calls**: Pre-analysis guards should avoid unnecessary `getClock()` or `getPositions()` calls when possible by checking local state first.
5. **No schema changes**: All config fields already exist in `EffectiveConfig`, `TradingConfig`, and the PostgreSQL schema. No migrations required.
6. **No API changes**: The enforcement is internal to `SessionAgent`. The existing `approveProposal()` / `rejectProposal()` callable interface is unchanged; it simply returns more specific error messages when enforcement blocks execution.
7. **Preserve existing behavior for hold/reject**: Only buy and sell actions are subject to enforcement. Hold recommendations and user rejections are unchanged.

## Non-Goals

- **New config fields**: No new risk parameters are introduced. This phase enforces what already exists.
- **Real-time P&L tracking**: The daily loss check uses the Alpaca portfolio history endpoint, not a real-time P&L calculation within the agent.
- **Partial order adjustment**: When `maxNotionalPerTrade` is exceeded, the order is rejected, not silently reduced. (Capping silently would be surprising. The user should adjust the config or the proposal.)
- **UI changes**: No new settings pages or enforcement visualizations. A future phase could add an enforcement log to the session UI.
- **Proposal-level enforcement**: Proposals are still created for any LLM recommendation. Enforcement happens at execution time (except for pre-analysis guards which prevent the analysis entirely). This keeps the debate/pipeline output visible to the user even when the system would block execution.
- **Strategy-level overrides**: Enforcement uses `EffectiveConfig` values directly. There is no per-strategy enforcement bypass.
- **Cooldown based on proposal rejection**: Cooldown only triggers after a realized loss (resolved outcome with negative P&L), not after a user rejects a proposal.

## Enforcement Points Summary

### Pre-Analysis Guards (in `triggerAnalysis()`)

| # | Check | When to Skip | Fail Behavior |
|---|-------|-------------|---------------|
| G1 | Market hours | `tradingHoursOnly && !clock.isOpen` | Return `{ threadIds: [] }`, log skip reason |
| G2 | Ticker blacklist | `tickerBlacklist.includes(symbol)` | Skip symbol, continue to next |
| G3 | Ticker allowlist | `tickerAllowlist && !tickerAllowlist.includes(symbol)` | Skip symbol, continue to next |
| G4 | Daily loss circuit breaker | `todayLossPct < -maxDailyLossPct` | Return `{ threadIds: [] }`, log skip reason |
| G5 | Cooldown after loss | Last resolved loss within `cooldownMinutesAfterLoss` | Return `{ threadIds: [] }`, log skip reason |

### Pre-Execution Guards (in `handleTradeDecision()`)

| # | Check | Condition | Fail Behavior |
|---|-------|-----------|---------------|
| E1 | Max notional per trade | `notional > maxNotionalPerTrade` | Return error: `Notional $X exceeds max $Y per trade` |
| E2 | Max position value | `notional > maxPositionValue` | Return error: `Position value $X exceeds max $Y` |
| E3 | Max positions | `action === 'buy' && positions.length >= maxPositions` | Return error: `Max N positions reached` |
| E4 | Short selling block | `action === 'sell' && !allowShortSelling && !hasPosition` | Return error: `Short selling disabled, no position held` |

### Proposal-Level Enhancement (in `createProposal()`)

| # | Check | Current | After |
|---|-------|---------|-------|
| P1 | Short selling warning | Warning added, proposal created | **Block proposal creation** when `!allowShortSelling && !hasPosition` -- do not store the proposal |

## Decision: Block at Proposal vs. Execution

For the short-selling case, there are two options:

**Option A**: Block at proposal creation (`createProposal()`) so the user never sees a sell proposal for an unheld symbol when short selling is disabled.

**Option B**: Allow proposal creation (current behavior with warning) but block at execution in `handleTradeDecision()`.

**Decision: Both.** Block at proposal creation (Option A) to prevent misleading proposals from appearing in the UI. Also block at execution (Option B) as a defense-in-depth measure -- if a sell proposal for an unheld symbol somehow reaches `handleTradeDecision()` (e.g., position was sold between proposal creation and approval), it is caught there too.

For all other enforcement checks (notional caps, position limits), blocking at execution is sufficient because the user needs visibility into what the LLM recommends even if the system would block execution. The proposal serves as information; the execution guard prevents harm.

---

## Alternatives Considered

### Alternative A: Enforce at Proposal Creation for All Checks

Run all enforcement (notional caps, position limits, etc.) at `createProposal()` time so proposals are pre-filtered.

**Rejected**: Position counts, account balances, and market conditions can change between proposal creation and user approval (proposals have a configurable timeout, default 15 minutes). A proposal that was valid at creation time may violate limits at execution time, and vice versa. Enforcement at execution time uses the freshest data.

### Alternative B: Cap Notional Silently Instead of Rejecting

When `maxNotionalPerTrade` is exceeded, automatically reduce the notional to the configured max and proceed.

**Rejected**: Silent adjustment changes the user's intended position size without explicit consent. The user configured the LLM to recommend a certain size; if the system silently reduces it, the risk/reward profile differs from what was analyzed. Rejection with a clear message is more transparent. A future enhancement could offer "adjust and proceed" as an option.

### Alternative C: Per-Symbol Enforcement in `runAnalysisForSymbol()`

Instead of filtering the symbol list in `triggerAnalysis()`, check each symbol at the start of `runAnalysisForSymbol()`.

**Rejected**: Functionally equivalent but wastes a thread creation + insertion for filtered symbols. Filtering the list before iteration is cleaner and avoids empty discussion threads in the database.

## Open Questions

1. **Portfolio history granularity**: Alpaca's `getPortfolioHistory()` is currently called with `period: '1M', timeframe: '1D'`. For the daily loss check, we need today's loss percentage. The last element of `profitLossPct` represents the most recent day. If the market is mid-day, this value is the intra-day loss so far. Is this sufficient, or should we compute P&L from account equity changes? **Recommendation**: Use the portfolio history endpoint as-is. It is already available via `AlpacaBrokerAgent.getPortfolioHistory()` and provides the needed data without additional broker API calls.

2. **Cooldown scope**: Should the cooldown after a losing trade apply globally (skip all analysis) or per-symbol (only skip the symbol that lost)? **Recommendation**: Global cooldown. A loss on one symbol may indicate broader market conditions that affect all positions. Per-symbol cooldown would be a more granular feature for a future phase.

3. **Notional calculation timing**: In `handleTradeDecision()`, when `qty` and `notional` are both null, notional is computed from `positionSizePct * account.cash`. Should the enforcement checks run before or after this computation? **Recommendation**: After. The enforcement checks the final dollar amount that would be sent to the broker, regardless of how it was derived.
