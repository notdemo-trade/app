# Phase 20: SRP Multi-Agent Architecture Refactor — Part 1: Spec

## Overview

Refactor the existing 3-agent system into a 4-category SRP-based multi-agent architecture with pluggable orchestration modes, real-time discussion visibility, SDK-native human-in-the-loop trade approval, and Workers AI as a new zero-cost LLM provider.

### Current Architecture Problems

1. **TechnicalAnalysisAgent** mixes data fetching (Alpaca API) with indicator computation — violates SRP
2. **OrchestratorAgent** is monolithic — no broker abstraction, no discussion model, no orchestration modes
3. **No broker agent** — trading operations scattered, no pluggable broker interface
4. **No discussion visibility** — users can't see agent reasoning or debate
5. **No approval flow** — recommendations exist but no HITL execution path
6. **No free-tier LLM option** — all providers require API keys and incur cost

### Refactor Outcome

- 4 clean agent categories: Data, Analysis, Broker, Session/Orchestrator
- 2 orchestration modes: Multi-Persona Debate + Sequential Pipeline
- Real-time discussion feed visible to users
- SDK-native `needsApproval` for trade execution
- Workers AI (`env.AI`) as zero-cost LLM provider via `@ai-sdk/cloudflare`
- User-selectable LLM provider in UI (including Workers AI)
- Iterative migration — zero breakage at each phase

## Context

### Agents SDK Patterns Used

| Pattern | Usage |
|---------|-------|
| `@callable()` | ONLY for client→agent WebSocket methods (SessionAgent) |
| Direct DO RPC | Agent-to-agent calls via `getAgentByName()` — no `@callable()` needed |
| `AIChatAgent` | Base class for SessionAgent — free message persistence + `useAgentChat` |
| `needsApproval` | Built-in HITL on `executeTrade` tool |
| `this.setState()` | Real-time state sync to connected WebSocket clients |
| `this.sql` | Per-instance SQLite for relational data, logs, history |
| `this.schedule()` | One-shot alarms (proposal expiration) |
| `this.scheduleEvery()` | Recurring tasks (data polling, analysis cycles) |
| `maxPersistedMessages` | Mandatory limit on AIChatAgent message history |
| TC39 decorators | Not `experimentalDecorators` |

### Key Constraint

All values in `setState()` must be JSON-serializable. Use `number` (timestamps) not `Date`, no `Map` or `Function`.

## Agent Taxonomy (4 Categories)

### Category 1: Data Agents

Fetch + normalize + cache raw data. Zero analysis logic.

| Agent | Instance Key | Source | Phase |
|-------|-------------|--------|------|
| `AlpacaMarketDataAgent` | `{userId}:{symbol}` | Alpaca bars/quotes/snapshots | M2 |
| `SentimentDataAgent` | `{userId}` | StockTwits/Twitter | Future |
| `NewsDataAgent` | `{userId}` | NewsAPI/Benzinga | Future |
| `SecFilingsDataAgent` | `{userId}:{ticker}` | SEC EDGAR | Future |
| `FredDataAgent` | `{userId}:{seriesId}` | FRED API | Future |

**AlpacaMarketDataAgent** extracts the data-fetching logic currently in `TechnicalAnalysisAgent`. Reuses `AlpacaMarketDataProvider` from `packages/data-ops/src/providers/alpaca/market-data.ts`.

RPC methods (no `@callable()` — called by other agents):
- `fetchBars(symbol, timeframe, params)` — fetch + cache bars
- `getLatestBars(symbol, timeframe, limit)` — return cached bars
- `fetchSnapshot(symbol)` — current quote/trade

### Category 2: Analysis Agents

Receive data as input, produce analysis/signals. Zero data fetching.

| Agent | Instance Key | Role | Phase |
|-------|-------------|------|------|
| `TechnicalAnalysisAgent` | `{userId}:{symbol}` | Compute indicators from provided bars, detect signals | M3 (refactor) |
| `SentimentAnalysisAgent` | `{userId}` | Analyze raw sentiment data | Future |
| `FundamentalAnalysisAgent` | `{userId}:{symbol}` | Analyze fundamentals/filings | Future |

**Refactored TechnicalAnalysisAgent**: `analyze()` receives `Bar[]` as parameter instead of fetching internally. Backward-compat shim during M3: if no bars provided, fetch from `AlpacaMarketDataAgent`.

### Category 3: Broker Agents

Full broker interface. One agent per broker provider.

| Agent | Instance Key | Broker | Phase |
|-------|-------------|--------|------|
| `AlpacaBrokerAgent` | `{userId}` | Alpaca Trading API | M2 |

Wraps existing `AlpacaTradingProvider` from `packages/data-ops/src/providers/alpaca/trading.ts`.

**Pluggable interface** (defined in data-ops):

```ts
interface BrokerAgent {
  getAccount(): Promise<BrokerAccount>
  getPositions(): Promise<BrokerPosition[]>
  placeOrder(order: OrderRequest): Promise<OrderResult>
  cancelOrder(orderId: string): Promise<void>
  getPortfolioHistory(params?: HistoryParams): Promise<PortfolioHistory>
  getClock(): Promise<MarketClock>
}
```

Future brokers (IBKR, Schwab) implement the same interface. SessionAgent resolves broker dynamically:

```ts
const brokerBinding = this.env[userConfig.brokerType] // "AlpacaBrokerAgent" | "IBKRBrokerAgent"
const broker = await getAgentByName(brokerBinding, userId)
```

RPC methods (no `@callable()` — called by SessionAgent/orchestrators):
- `getAccount()`, `getPositions()`, `placeOrder()`, `cancelOrder()`, `getPortfolioHistory()`, `getClock()`

### Category 4: Session & Orchestrator Agents

| Agent | Instance Key | Base Class | Role | Phase |
|-------|-------------|-----------|------|------|
| `SessionAgent` | `{userId}` | `AIChatAgent` | UI entry point, WebSocket, message persistence, HITL | M2/M6 |
| `DebateOrchestratorAgent` | `{userId}` | `Agent` | Multi-persona debate orchestration | M5 |
| `PipelineOrchestratorAgent` | `{userId}` | `Agent` | Sequential pipeline orchestration | M5 |

**SessionAgent** extends `AIChatAgent`:
- `maxPersistedMessages = 500`
- Automatic message persistence in SQLite via `this.messages`
- `useAgentChat` hook for client-side chat UI
- `needsApproval` on `executeTrade` tool — SDK-native HITL
- Streaming LLM responses via `onChatMessage()` + `streamText()`
- Delegates to `DebateOrchestratorAgent` or `PipelineOrchestratorAgent` based on user preference
- Broadcasts `DiscussionThread` updates via `setState()`

## Orchestration Modes

### Mode A: Multi-Persona Debate

Three configurable personas analyze the same data independently, then debate in rounds.

**Default Personas**:

| Persona | Role | Default Bias |
|---------|------|-------------|
| Bull Analyst | Finds reasons to buy | Optimistic, momentum-focused |
| Bear Analyst | Finds reasons to sell | Skeptical, risk-focused |
| Risk Manager | Assesses risk, position sizing | Neutral, portfolio-focused |

**Protocol**:
1. **Independent Analysis** — 3 personas analyze same data in parallel via `Promise.all()`
2. **Debate Rounds** (1-5, configurable, default 2) — each persona sees others' analyses and responds
3. **Consensus Synthesis** — neutral moderator LLM produces `ConsensusResult` with confidence score

All LLM calls go through existing `LLMAnalysisAgent` with new RPC methods:
- `analyzeAsPersona(persona, data, strategy)` — analysis from persona's perspective
- `runDebateRound(analyses, roundNumber)` — persona responses to others
- `synthesizeConsensus(allAnalyses, debateRounds)` — final synthesis

```
DebateOrchestratorAgent.runDebate(data, strategy)
  ├── Promise.all([
  │     llm.analyzeAsPersona("bull_analyst", data, strategy),
  │     llm.analyzeAsPersona("bear_analyst", data, strategy),
  │     llm.analyzeAsPersona("risk_manager", data, strategy),
  │   ])
  ├── for round in 1..numRounds:
  │     llm.runDebateRound(analyses, round)
  └── llm.synthesizeConsensus(allAnalyses, allRounds)
      → ConsensusResult { action, confidence, rationale, dissent }
```

### Mode B: Sequential Pipeline

Linear chain of steps, each adding to a shared `PipelineContext`.

| Step | Agent | Input | Output |
|------|-------|-------|--------|
| 1. Fetch Data | AlpacaMarketDataAgent | symbol, timeframe | Raw bars |
| 2. Technical Analysis | TechnicalAnalysisAgent | bars | Indicators + signals |
| 3. LLM Analysis | LLMAnalysisAgent | signals + strategy | Recommendation |
| 4. Risk Validation | LLMAnalysisAgent | recommendation + portfolio | Risk-adjusted recommendation |
| 5. Generate Proposal | (orchestrator) | validated recommendation | TradeProposal |

Each step adds its output to `PipelineContext`. If any step fails, pipeline halts with error.

```
PipelineOrchestratorAgent.runPipeline(symbol, strategy)
  ├── step 1: marketData.fetchBars(symbol, timeframe)
  ├── step 2: ta.analyze(bars)
  ├── step 3: llm.analyze({ signals, strategy })
  ├── step 4: llm.validateRisk({ recommendation, positions })
  └── step 5: generateProposal(validatedRecommendation)
      → TradeProposal
```

## Discussion Model

All agent interactions are captured as `DiscussionMessage`s, grouped into `DiscussionThread`s per symbol analysis cycle. Stored in SessionAgent SQLite, broadcast via WebSocket.

```ts
interface DiscussionThread {
  id: string
  orchestrationMode: 'debate' | 'pipeline'
  symbol: string
  status: 'in_progress' | 'completed' | 'failed'
  startedAt: number       // timestamp ms
  completedAt: number | null
  messages: DiscussionMessage[]
  proposal: TradeProposal | null
}

interface DiscussionMessage {
  id: string
  threadId: string
  timestamp: number       // timestamp ms
  sender: MessageSender
  phase: DiscussionPhase
  content: string
  metadata: Record<string, unknown>
}

type MessageSender =
  | { type: 'system' }
  | { type: 'data_agent'; name: string }
  | { type: 'analysis_agent'; name: string }
  | { type: 'persona'; persona: PersonaId }
  | { type: 'moderator' }
  | { type: 'broker'; name: string }
  | { type: 'user' }

type PersonaId = 'bull_analyst' | 'bear_analyst' | 'risk_manager' | string

type DiscussionPhase =
  | 'data_collection'
  | 'analysis'
  | 'debate_round'
  | 'consensus'
  | 'proposal'
  | 'human_decision'
  | 'execution'
  | 'completed'
```

## Human-in-the-Loop

Uses **built-in `needsApproval` pattern** from Agents SDK instead of custom approval flow.

### SessionAgent Tool Definition

```ts
tools: {
  executeTrade: tool({
    description: "Execute a trade based on agent recommendation",
    inputSchema: tradeProposalSchema,
    needsApproval: async ({ estimatedNotional }) => estimatedNotional > 0,
    execute: async (params) => {
      const broker = await getAgentByName(this.env.AlpacaBrokerAgent, userId)
      return broker.placeOrder(params)
    },
  }),
}
```

### Client-Side Rendering

```tsx
// Filter tool parts awaiting approval
messages
  .flatMap(m => m.parts)
  .filter(p => p.type === "tool" && p.state === "approval-required")
  .map(p => (
    <TradeProposalCard
      proposal={p.args}
      onApprove={() => addToolApprovalResponse({ id: p.toolCallId, approved: true })}
      onReject={() => addToolApprovalResponse({ id: p.toolCallId, approved: false })}
    />
  ))
```

### Flow

```
Orchestrator produces recommendation with confidence >= threshold
  → SessionAgent.onChatMessage() invokes `executeTrade` tool
  → SDK detects needsApproval=true → pauses execution
  → Client sees tool part with state="approval-required"
  → UI renders TradeProposalCard with [Approve] / [Reject]
  → User decides → addToolApprovalResponse()
  → SDK resumes execute() or cancels
  → If approved: AlpacaBrokerAgent.placeOrder() → execution result
  → Proposal auto-expires after timeout (default 15min) via this.schedule()
```

## UX Philosophy

**Minimal input, maximum autonomy.** User configures once, agents operate autonomously, user only approves/rejects proposals.

### Onboarding (one-time)

1. Choose broker (Alpaca MVP)
2. Enter broker credentials (existing Phase 2 flow)
3. Choose orchestration mode (Debate / Pipeline)
4. Set watchlist symbols (or "auto" from portfolio)
5. Choose LLM provider — Workers AI (Free, no key needed) or existing providers (OpenAI, Anthropic, Google, xAI, DeepSeek — requires API key)

### Post-Onboarding — Zero-Touch

- Agents autonomously: fetch data → analyze → debate/pipeline → generate proposals
- User sees real-time **discussion feed** (like chat)
- Trade proposals appear as approval cards
- User clicks **Approve** or **Reject** — only required input

### Advanced Options (collapsed by default)

| Option | Default | Description |
|--------|---------|-------------|
| LLM Provider | Workers AI (Free) | OpenAI, Anthropic, Google, xAI, DeepSeek, or Workers AI |
| LLM Model | Per-provider default | Model to use for analysis and debate |
| Persona system prompts | Built-in bull/bear/risk | Customize persona behavior |
| Orchestrator system prompt | Built-in | Customize consensus synthesis |
| Debate rounds | 2 | 1-5 rounds of argument exchange |
| Analysis interval | 120s | How often agents collect + analyze |
| Min confidence threshold | 0.7 | Below this, proposal not shown to user |
| Position sizing | 5% cash | Max % of portfolio per position |
| Auto-approve | OFF | Future: auto-approve below $ threshold |
| Strategy template | Moderate | Conservative / Moderate / Aggressive |

## Workers AI — New LLM Provider

### Motivation

Workers AI runs on Cloudflare's edge network via `env.AI` binding. It requires **no API key** and has a **free tier** (10,000 neurons/day). This gives users an immediate, zero-config LLM option — especially valuable during onboarding before they've added third-party API keys.

### Integration Approach

Workers AI is added as a 7th provider (`'workers-ai'`) alongside the existing 5 (openai, anthropic, google, xai, deepseek). The Vercel AI SDK stays as the abstraction layer via `@ai-sdk/cloudflare`.

### Provider Configuration

| Aspect | Workers AI | Other Providers |
|--------|-----------|----------------|
| API key required | No (uses `env.AI` binding) | Yes |
| Credential storage | None needed | Encrypted in Postgres |
| Pricing | Free tier (0 cost) | Per-token pricing |
| Model format | `@cf/meta/llama-3.3-70b-instruct-fp8-fast` | Provider-specific IDs |
| Binding | `env.AI` (Cloudflare AI binding) | HTTP API via SDK |

### Default Models

| Model ID | Use Case |
|----------|----------|
| `@cf/meta/llama-3.3-70b-instruct-fp8-fast` | General analysis, debate, consensus |
| `@cf/qwen/qwen2.5-coder-32b-instruct` | Structured JSON output, risk validation |

### Type Changes

### Wrangler Binding

```jsonc

```

### User Experience

- During onboarding (Step 5: "Choose LLM provider"), Workers AI appears as "Workers AI (Free)" with no credential entry required
- Users can switch between providers at any time in session settings
- Workers AI is the automatic fallback if no other provider credentials exist
- Provider selection stored in SessionAgent config (`session_config.llm_provider`)

### Limitations

- Workers AI models are smaller/less capable than frontier models (GPT-4o, Claude Sonnet)
- No streaming support in some models
- Rate limits tied to Cloudflare account neuron budget
- JSON mode support varies by model

## Data Flow — Debate Mode (Complete)

```
UI (WebSocket) <─── useAgentChat ───> SessionAgent (AIChatAgent)
                                         │
                                         ├── AlpacaMarketDataAgent.fetchBars(symbol)
                                         │     → raw bars (cached in DO SQLite)
                                         │
                                         ├── TechnicalAnalysisAgent.analyze(bars)
                                         │     → indicators + signals
                                         │
                                         ├── DebateOrchestratorAgent.runDebate({ signals, strategy })
                                         │     ├── LLM.analyzeAsPersona("bull", data)  ─┐
                                         │     ├── LLM.analyzeAsPersona("bear", data)  ─┤ Promise.all
                                         │     ├── LLM.analyzeAsPersona("risk", data)  ─┘
                                         │     ├── LLM.runDebateRound(analyses, round=1)
                                         │     ├── LLM.runDebateRound(analyses, round=2)
                                         │     └── LLM.synthesizeConsensus(all)
                                         │           → ConsensusResult + TradeProposal
                                         │
                                         ├── [Discussion messages broadcast via setState()]
                                         │
                                         └── If confidence >= threshold:
                                               executeTrade tool → needsApproval → UI card
                                               User approves → AlpacaBrokerAgent.placeOrder()
```

## Data Flow — Pipeline Mode (Complete)

```
UI (WebSocket) <─── useAgentChat ───> SessionAgent (AIChatAgent)
                                         │
                                         └── PipelineOrchestratorAgent.runPipeline(symbol, strategy)
                                               ├── Step 1: AlpacaMarketDataAgent.fetchBars()
                                               ├── Step 2: TechnicalAnalysisAgent.analyze(bars)
                                               ├── Step 3: LLMAnalysisAgent.analyze(signals)
                                               ├── Step 4: LLMAnalysisAgent.validateRisk(rec, positions)
                                               └── Step 5: Generate TradeProposal
                                                     → If confidence >= threshold:
                                                       executeTrade tool → needsApproval → UI card
```

```ts
export interface DiscussionMessage {
  id: string
  threadId: string
  timestamp: number
  sender: MessageSender
  phase: DiscussionPhase
  content: string
  metadata: Record<string, unknown>
}

export type MessageSender =
  | { type: 'system' }
  | { type: 'data_agent'; name: string }
  | { type: 'analysis_agent'; name: string }
  | { type: 'persona'; persona: PersonaId }
  | { type: 'moderator' }
  | { type: 'broker'; name: string }
  | { type: 'user' }

export type DiscussionPhase =
  | 'data_collection'
  | 'analysis'
  | 'debate_round'
  | 'consensus'
  | 'proposal'
  | 'human_decision'
  | 'execution'
  | 'completed'

export interface TradeProposal {
  id: string
  threadId: string
  symbol: string
  action: 'buy' | 'sell'
  confidence: number
  rationale: string
  entryPrice: number | null
  targetPrice: number | null
  stopLoss: number | null
  qty: number | null
  notional: number | null
  positionSizePct: number
  risks: string[]
  expiresAt: number
  status: 'pending' | 'approved' | 'rejected' | 'expired' | 'executed'
  createdAt: number
  decidedAt: number | null
}
```
