# Phase 32: Proposal Dedup Guards & Minor Data Fixes -- Part 5: UI

## No UI Changes

All three fixes are backend-only. No new routes, components, or UI modifications are required.

---

## Observable Behavior Changes in Existing UI

### Session Dashboard

The session dashboard shows analysis cycle results. After this fix:

- **Fewer threads per cycle**: If a watchlist has 5 symbols and 2 already have pending proposals, only 3 threads are created per cycle (instead of 5). The thread count displayed in the cycle summary will be lower.
- **No duplicate proposals in proposal list**: The proposals panel will no longer show multiple pending proposals for the same symbol. Each symbol will have at most one active proposal at a time.

### Proposal Review Panel

- **No change to the approval/rejection flow**: Users still see pending proposals and can approve or reject them. Once a proposal is resolved (any terminal status), the next analysis cycle will generate a new proposal for that symbol if the analysis recommends one.

### Thread History

- **Skip messages not displayed**: The dedup skip returns `threadId: ''`, which means no discussion thread is created. The skip is invisible in the thread history. Users will simply notice that a symbol does not have a new thread until its existing proposal is resolved.

### Outcome Dashboard

- **More accurate P&L over time**: The exit order fix will produce more correct P&L calculations for future outcomes. Historical outcomes are not retroactively corrected, so users may see a discrepancy between old and new outcome accuracy once the fix is deployed.

---

## Potential Future UI Enhancements (Out of Scope)

1. **Skip notification**: A subtle indicator in the session dashboard showing "2 symbols skipped (active proposals exist)" could help users understand why analysis did not run for all symbols. This is a UX improvement, not a bug fix.
2. **Expired proposal sweep button**: A manual "clear expired proposals" action in the proposal panel would allow users to unblock symbols with stale expired proposals without having to explicitly reject each one. Deferred.
3. **Outcome recalculation trigger**: A button to recompute historical outcomes using the corrected exit order logic. Requires a backend endpoint and is out of scope.
