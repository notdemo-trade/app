# Phase 32: Proposal Dedup Guards & Minor Data Fixes -- Part 1: Spec

## Overview

Three correctness bugs in the analysis and outcome-tracking pipelines can produce duplicate proposals, misattributed exit orders, and misleading log entries. All three are data-integrity issues that manifest silently during normal automated operation -- no error is thrown, but the resulting data is wrong.

This phase groups the fixes because they share the same change surface (`session-agent.ts` and `pipeline-orchestrator-agent.ts`), have no external dependencies, require no schema migration, and can be verified in a single test session.

### Why This Matters

- **Duplicate proposals** lead to over-concentration risk. If a user approves two pending BUY proposals for the same symbol, they submit two buy orders. No downstream guard prevents this because the approval flow treats each proposal independently.
- **Wrong exit order matching** corrupts outcome tracking. The `ProposalOutcome` record stores P&L computed from the exit price. Picking the wrong exit order means the P&L is wrong, which feeds into strategy scoring and the portfolio performance dashboard.
- **Empty `threadId`** is not a runtime failure, but it undermines debugging. Any log emitted during pipeline execution before the patch references `threadId: ''`, making it impossible to correlate pipeline activity with the originating discussion thread.

## Context & Background

### Problem 1: No Duplicate Proposal Guard

**Location**: `session-agent.ts:408-457` (`runAnalysisForSymbol`)

The `triggerAnalysis()` method (line 234) iterates all watchlist symbols and calls `runAnalysisForSymbol()` for each. Both `debate` and `pipeline` orchestration modes can produce a `TradeProposal` with `status: 'pending'`. There is no check whether a pending or approved proposal already exists for the symbol being analyzed.

The portfolio context object passed to the LLM includes pending proposals as context, but this is a soft signal -- the LLM may still recommend the same trade. The dedup guard must be a hard check in application code, not a prompt instruction.

**Scenario**:

```
Cycle 1: triggerAnalysis()
  -> runAnalysisForSymbol('AAPL') -> debate -> BUY AAPL proposal (status: pending)

User does not act.

Cycle 2: triggerAnalysis()  (120 seconds later)
  -> runAnalysisForSymbol('AAPL') -> debate -> BUY AAPL proposal (status: pending)

User approves both -> two buy orders -> double intended position size
```

**Why the LLM context is insufficient**: The portfolio context includes `pendingProposals` as part of the system prompt. However:
1. The LLM may not interpret "pending proposal exists" as "do not recommend the same trade"
2. Different analysis cycles may use different market data, leading to a legitimately different rationale for the same action
3. Prompt compliance is non-deterministic -- a hard guard is required for correctness

### Problem 2: `findExitOrder` Picks Wrong Order

**Location**: `session-agent.ts:1189-1200`

```ts
private findExitOrder(
    orders: OrderLogEntry[],
    outcome: ProposalOutcome,
): OrderLogEntry | undefined {
    const exitSide = outcome.action === 'buy' ? 'sell' : 'buy';
    return orders
        .filter(
            (o) => o.side === exitSide && o.status === 'filled' && o.createdAt > outcome.createdAt,
        )
        .sort((a, b) => b.createdAt - a.createdAt)[0];
}
```

The sort is `b.createdAt - a.createdAt` (descending), and `[0]` takes the most recent. If there are multiple filled exit-side orders after the entry (partial liquidations, or exits from multiple positions in the same symbol), the most recent one is likely not the one that closed *this* specific position. The earliest exit after entry is the correct match for a FIFO position model.

**Scenario**:

```
t=0:  BUY AAPL 10 shares (entry, tracked as outcome)
t=1:  SELL AAPL 5 shares (partial exit -- this should match)
t=2:  SELL AAPL 5 shares (second partial exit)

Current code picks t=2 exit (most recent).
Correct behavior: pick t=1 exit (earliest after entry).
```

In the partial liquidation case, the first exit chronologically after entry is the most likely match. This is a heuristic improvement -- perfect matching would require `client_order_id` correlation, which is a larger change deferred to a future phase.

### Problem 3: Pipeline Proposal `threadId` Empty at Construction

**Location**: `pipeline-orchestrator-agent.ts:468-497` (`buildProposal`)

```ts
private buildProposal(ctx: PipelineContext, params: RunPipelineParams): TradeProposal {
    return {
        id: crypto.randomUUID(),
        threadId: '',  // <-- empty string
        symbol: params.symbol,
        // ...
    };
}
```

Then in `session-agent.ts:552-553`:

```ts
if (result.proposal) {
    const proposal = { ...result.proposal, threadId };
    this.storeProposal(proposal);
}
```

The `threadId` is known at the call site in `runPipelineAnalysis()` and could be passed to `runPipeline()` as a parameter. The current spread-patch pattern works but is fragile:
- The `onMessage` callback within the pipeline logs messages with no thread correlation
- If `buildProposal` is ever called in a context where the result is used before being patched (e.g., intermediate validation, conditional logging), the empty `threadId` silently corrupts the data
- The pattern violates the principle that objects should be fully constructed at creation time

## Goals

1. **Hard dedup guard**: Skip analysis for a symbol when a `pending` or `approved` proposal already exists. Return a summary message indicating the skip.
2. **Correct exit order matching**: Change `findExitOrder()` to return the earliest filled exit-side order after entry, not the most recent.
3. **Construct-time `threadId`**: Pass `threadId` to `RunPipelineParams` so `buildProposal()` receives it at construction time. Remove the spread-patch in `runPipelineAnalysis()`.

## Non-Goals

- **`client_order_id` correlation for exit matching**: A precise match using broker-assigned order IDs would require changes to the order placement flow and outcome tracking schema. Deferred to a future phase.
- **Per-action dedup** (e.g., allow a SELL proposal while a BUY proposal is pending for the same symbol): The current guard blocks any active proposal for the symbol regardless of action. This is conservative and correct -- allowing conflicting actions for the same symbol would create confusing UX.
- **UI changes**: The skip message appears in the discussion thread as a system message. No new UI components are needed.
- **Notification when analysis is skipped**: The skip is logged in the thread summary. No toast, alert, or push notification is generated.

## Constraints

| Constraint | Rationale |
|------------|-----------|
| DO SQLite only (no PostgreSQL migration) | All three fixes modify in-memory logic and DO SQLite queries. No Neon schema changes. |
| No new agent types | Fixes are contained within existing `SessionAgent` and `PipelineOrchestratorAgent`. |
| Backward compatible | Existing proposals, outcomes, and threads are unaffected. The dedup guard only applies to future analysis cycles. |
| No config flag | The dedup guard is unconditional. There is no reason to allow duplicate proposals. |
