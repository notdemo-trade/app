# Phase 26: Fix Position Size Units Mismatch — Part 1: Spec

## Overview

The `positionSizePct` field on `TradeProposal` is produced by two orchestration modes (debate and pipeline) and consumed by a single execution path (`handleTradeDecision()`). The execution path divides the value by 100, meaning it expects **whole-number percentages** (5 = 5%). However, the debate-mode fallback injects **fractions** (0.05 = 5%) from `config.positionSizePctOfCash`, resulting in:

```
$100,000 cash * (0.05 / 100) = $50 instead of $5,000
```

This is a 100x position sizing error that has gone undetected because the consensus LLM usually provides a `positionSizePct` value (so the fallback rarely triggers), and pipeline mode uses its own hardcoded fallback of `5`.

### Why This Matters

- Position sizing is the single most impactful parameter in trade execution -- a 100x error means either microscopic positions (losing money to commissions) or, if the sign were reversed, catastrophically oversized positions.
- The pipeline mode silently ignores the user's configured position size, which undermines the entire trading config system built in Phases 2, 21, and 23.
- Strategy template `positionSizeBias` values (0.03, 0.05, 0.1) would produce the same 100x-too-small error if they reached execution via the debate fallback path.

---

## Context & Background

### Two Conventions, One Field

The codebase evolved with two independent conventions for expressing percentages:

**Fraction convention (0.0-1.0):** Used by configuration and strategy systems that were built first.
- `DEFAULT_SESSION_CONFIG.positionSizePctOfCash = 0.05`
- `StrategyTemplate.positionSizeBias` values: 0.03, 0.05, 0.1
- `resolve-config.ts` -- resolves from trading config, session config, or strategy, all in fractions
- PostgreSQL `user_trading_config.position_size_pct_of_cash` with default `0.10`
- DO SQLite `session_config.position_size_pct` with default `0.05`

**Whole-number convention (1-100):** Used by LLM prompts and the execution layer that were built later.
- LLM analysis prompt: `position_size_pct: 1-10`
- LLM consensus prompt: `positionSizePct: number or null (1-10)`
- `parseRecommendation()`: clamps to `Math.max(1, Math.min(10, ...))`
- Pipeline fallback: `5`
- Exit proposal: `positionSizePct: 100`
- `handleTradeDecision()`: `account.cash * (positionSizePct / 100)`

### The Conflict Points

**Point 1 -- Debate fallback (line 835):**

```ts
positionSizePct: consensus.positionSizePct ?? config.positionSizePctOfCash,
// When consensus is null: 0.05 (fraction) enters a whole-number field
```

**Point 2 -- Pipeline bypass (line 471):**

```ts
const positionSizePct = ctx.riskValidation?.adjustedPositionSize ?? rec.position_size_pct ?? 5;
// config.positionSizePctOfCash is never in the fallback chain
```

**Point 3 -- Strategy context display (line 580):**

```ts
`Position size bias: ${strategy.positionSizeBias * 100}%`
// Correctly converts 0.05 to "5%" for the LLM prompt text
// But this is just a display string -- it doesn't affect the returned number
```

### Risk Validation Ambiguity

The `RiskValidation.adjustedPositionSize` field comes from an LLM response (via `parseRiskValidation()`). The LLM prompt asks for the value but does not specify the convention. The LLM sees the incoming `recommendation.position_size_pct` (which is 1-10 from the analysis step) and likely returns a value in the same range. However, this is not guaranteed -- an LLM could return `0.05` meaning "5%". This ambiguity should be resolved with a normalization guard.

---

## Goals

1. **Fix the 100x error**: Debate-mode proposals must compute correct notional values when consensus does not provide a position size.
2. **Pass user config to pipeline**: Pipeline mode must use `config.positionSizePctOfCash` as a fallback, not a hardcoded `5`.
3. **Normalize at boundaries**: Add guards at the points where values enter `TradeProposal.positionSizePct` to catch mixed-convention values.
4. **Minimal blast radius**: Change as few files as possible. Do not refactor the entire codebase to use one convention -- just fix the boundary crossings.

## Non-Goals

- **Refactor all config to whole-number percentages**: This would touch defaults, strategies, trading config schema, resolve-config, session config SQL, and many design docs. Too large for a bug fix.
- **Refactor all LLM prompts to fractions**: This would require re-prompting and re-testing LLM behavior. The LLM naturally thinks in whole numbers ("allocate 5% of cash") and the current prompts work well.
- **Add unit tests for position sizing**: Valuable but separate scope. The universal rules say "when I report a bug, start by writing a test" -- but this design doc captures the fix design; tests are covered in Part 6 verification.
- **Change the `TradeProposal.positionSizePct` type**: Adding a branded type or unit annotation would be a larger refactor.

---

## Solution Design

### Convention Decision: `TradeProposal.positionSizePct` uses whole-number percentages

The execution math (`/ 100`) is correct and should not change. The exit proposal (`positionSizePct: 100`) is correct. The pipeline fallback (`5`) is correct. The LLM-returned values are correct. The only incorrect values are the config/strategy fractions that leak into the field. Therefore:

**Normalize fractions to whole numbers at the point where config values enter `TradeProposal`**, not at execution time.

### Change 1: Fix debate fallback (session-agent.ts, line 835)

Convert the config fraction to a whole-number percentage:

```ts
// Before:
positionSizePct: consensus.positionSizePct ?? config.positionSizePctOfCash,

// After:
positionSizePct: consensus.positionSizePct ?? config.positionSizePctOfCash * 100,
```

This converts `0.05` to `5`, matching the execution math.

### Change 2: Add normalization guard on consensus value

The LLM consensus `positionSizePct` is nullable. When present, it should be 1-10 per the prompt. But an LLM could return `0.05`. Add a guard:

```ts
function normalizePositionSizePct(value: number): number {
    // Values <= 1 are likely fractions (0.05 = 5%), convert to whole-number percentage
    // Exception: exactly 1 is ambiguous but treated as 1% (valid whole number)
    if (value > 0 && value < 1) {
        return value * 100;
    }
    return value;
}
```

Applied to the debate proposal creation:

```ts
const rawPct = consensus.positionSizePct ?? config.positionSizePctOfCash * 100;
positionSizePct: normalizePositionSizePct(rawPct),
```

### Change 3: Pass `positionSizePctOfCash` to pipeline (session-agent.ts + pipeline-orchestrator-agent.ts)

Add the field to `RunPipelineParams`:

```ts
export interface RunPipelineParams {
    // ... existing fields ...
    positionSizePctOfCash?: number;  // fraction (0.0-1.0) from effective config
}
```

Pass it from `runPipelineAnalysis()`:

```ts
const result = await pipeline.runPipeline({
    // ... existing params ...
    positionSizePctOfCash: config.positionSizePctOfCash,
});
```

Use it in `buildProposal()`, converting fraction to whole number:

```ts
const positionSizePct = normalizePositionSizePct(
    ctx.riskValidation?.adjustedPositionSize
        ?? rec.position_size_pct
        ?? (params.positionSizePctOfCash !== undefined ? params.positionSizePctOfCash * 100 : 5)
);
```

### Change 4: Add normalization to risk validation output

The `adjustedPositionSize` from `parseRiskValidation()` comes from an LLM and could be in either convention. Apply the same guard:

```ts
adjustedPositionSize:
    typeof parsed.adjustedPositionSize === 'number'
        ? normalizePositionSizePct(parsed.adjustedPositionSize)
        : null,
```

### Normalization Heuristic Justification

The `normalizePositionSizePct` function uses `value < 1` as the threshold for detecting fractions. This is safe because:

- No valid whole-number percentage position size is less than 1% (the LLM prompt says 1-10)
- The `parseRecommendation()` function already clamps LLM values to `Math.max(1, ...)`
- Config fractions are always < 1 (max is `0.10` for aggressive strategy, `1.0` for the Zod schema limit which represents 100%)
- The edge case of exactly `1.0` (100% in fraction, or 1% in whole number) is ambiguous, but `1.0` as a config fraction would mean "100% of cash" which is intentionally extreme -- and `config.positionSizePctOfCash * 100` would produce `100` before reaching the normalizer, so this path never triggers for config values

## Alternatives Considered

### Alternative A: Normalize everything to fractions, remove `/100` from execution

Change `handleTradeDecision()` to multiply directly (`cash * positionSizePct`), change pipeline fallback from `5` to `0.05`, and change LLM prompts to request fractions.

**Rejected**: Higher blast radius (LLM prompts, parsing, risk validation, exit proposals, all existing stored proposals in DO SQLite). LLMs naturally express percentages as whole numbers. The execution math and LLM layer are self-consistent; only the config-to-proposal boundary is wrong.

### Alternative B: Add a `positionSizePctUnit` discriminator field

Add a field to `TradeProposal` indicating whether `positionSizePct` is a fraction or whole number.

**Rejected**: Overengineered for a bug fix. Adds complexity to every consumer. The normalization guard is simpler and sufficient.

### Alternative C: Fix only the debate fallback, leave pipeline as-is

Just multiply `config.positionSizePctOfCash * 100` at line 835 and call it done.

**Rejected**: Pipeline mode still ignores user config, which is a separate but related defect. Since we are touching the same code area, fixing both is low marginal effort.

## Open Questions

1. **Existing stored proposals with wrong values**: DO SQLite `trade_proposals` may contain rows where `position_size_pct` is `0.05` (from the debate fallback). These are historical and cannot be retroactively corrected. The `positionSizePct` on stored proposals is only used for display purposes after execution (the actual executed notional is computed at execution time and recorded separately). No migration is needed.

2. **LLM consensus returning values outside 1-10**: The consensus prompt says `positionSizePct: number or null (1-10)`, but the `parseConsensusResult()` function does not clamp the value (unlike `parseRecommendation()` which clamps 1-10). Should we add clamping? Deferred -- the normalization guard handles the < 1 case, and values > 10 (e.g., 15%) are valid user-requested position sizes.

3. **`buildStrategyContext()` sends "Position size bias: 5%" to the LLM**: This display string is correct and informational. However, when the LLM sees "5%" in the strategy context and returns `position_size_pct`, it may be influenced by this value. This is working as intended -- the strategy bias should influence the LLM recommendation.
