# Phase 26: Fix Position Size Units Mismatch — Part 4: API

## No API Changes Required

This bug fix is entirely within the agent layer (Durable Objects). No Hono REST endpoints are affected.

### Why no API changes

The position size mismatch occurs inside the `SessionAgent` and `PipelineOrchestratorAgent` Durable Objects during analysis and trade execution. The API endpoints that interact with position sizing are:

**Trading config endpoints** (`/api/trading-config`):
- Accept and return `positionSizePctOfCash` as a fraction (0.01-1.0)
- This convention is correct and does not need to change
- The Zod schema `z.number().min(0.01).max(1.0)` remains valid

**Session config (via WebSocket RPC)**:
- `updateConfig()` callable on `SessionAgent` accepts `positionSizePctOfCash` as a fraction
- Stored in DO SQLite as a fraction
- This convention is correct and does not need to change

**Trade proposal display (via WebSocket state broadcast)**:
- `TradeProposal.positionSizePct` is broadcast to the UI for display
- After the fix, this value will be a consistent whole-number percentage
- The UI already displays it as-is (e.g., "5%") -- no change needed

### No new endpoints

No new API endpoints are introduced by this fix.

### No validation changes

No Zod schemas are modified. The trading config validation (`positionSizePctOfCash: z.number().min(0.01).max(1.0)`) remains correct because it validates the config input, not the computed proposal value.
