## Alternatives Considered

### Token storage: encrypted vs hashed
- **Encrypted**: recoverable, more complex key management
- **Hashed (chosen)**: one-way, simpler, GitHub pattern

### Token format: UUID vs random bytes
- **UUID**: recognizable, shorter
- **Random bytes (chosen)**: more entropy, harder to guess

### Auth library: roll our own vs Better Auth
- **Better Auth (chosen)**: already in template, handles sessions, supports email/password

---


## Security Considerations

- Tokens hashed with SHA-256, never stored plaintext
- Token shown only once on creation
- Revocation is soft-delete (keeps audit trail)
- 1 month expiration by default (auto-expire)
- Two token types for separation of concerns:
  - `access`: general API access (trading, signals, etc.)
  - `kill_switch`: emergency halt only (store separately, more secure)
- Only one active token per type per user (regenerate = revoke old)
- lastUsedAt tracking for security audit
- Token type checked in middleware (kill_switch routes reject access tokens)

---


## Decisions

1. **Token expiration**: 1 month from generation (default)
2. **Token types**: One `access` token + one `kill_switch` token per user (2 max)
3. **Google OAuth**: Keep
4. **Email verification**: Skip for Phase 1
