# Phase 23: Extended User Settings — Part 1: Spec

## Overview

Six runtime parameters are currently hardcoded across agent and UI files. This phase promotes them to user-configurable columns on the existing `user_trading_config` Postgres table, wires them through the agent layer via the existing config propagation path, and adds an "Advanced" section to the trading settings page.

The scope is deliberately narrow: add columns, update reads, add UI controls. No new tables, no new API endpoints (the existing `GET/PATCH /trading-config` already handles arbitrary column sets), and no architectural changes.

### Why Now

- Users have requested control over proposal expiration timing.
- The single hardcoded temperature (0.3) does not suit all trading styles -- conservative traders want lower temperature (more deterministic), aggressive traders want higher (more creative).
- The fixed [30, 90, 180] day performance windows are arbitrary; some users want shorter windows (7, 14 days) for faster feedback.
- Confidence display thresholds need to align with each user's `minConfidenceThreshold` setting.

## Context & Background

### Existing Config Architecture

```
user_trading_config (Postgres)
  -- user-facing settings page writes here
  -- 17 existing columns (position limits, risk, models, feature flags)
  -- read by: server functions, session agent (via resolveEffectiveConfig)

session_config (DO SQLite per SessionAgent)
  -- session-specific settings (orchestration mode, watchlist, etc.)
  -- includes proposalTimeoutSec (already exists, default 900)
```

The `user_trading_config` table is the user's canonical preferences store. The session agent's `loadEffectiveConfig()` already merges trading config (PG) with session config (DO SQLite). This phase adds 6 new columns to the PG table and updates the read sites.

## Goals

1. Add 6 new columns to `user_trading_config` with sensible defaults that match current hardcoded values.
2. Update all hardcoded read sites to read from the user's config.
3. Add an "Advanced" settings section to the existing trading settings page with proper grouping, validation, and tooltips.
4. Maintain backward compatibility -- existing users with no explicit values get the current defaults via column defaults.
5. Validate all new fields both client-side (form validators) and server-side (Zod schema).

## Non-Goals

- Creating a new config table (reuse `user_trading_config`).
- Adding per-task-type temperature overrides (single global preference with per-task scaling).
- Changing the `session_config` DO SQLite schema (the 6 new fields live in PG only).
- Modifying the config resolution hierarchy from Phase 21 (these are PG-only fields, no conflict).
- Adding presets/profiles for advanced settings (future phase).

## Settings Specification

### 1. proposalTimeoutSec

| Attribute | Value |
|-----------|-------|
| Column | `proposal_timeout_sec` |
| Type | `integer` |
| Default | `900` |
| Range | 60 - 3600 |
| Unit | seconds |
| UI Group | Proposals |
| Tooltip | "How long a trade proposal stays active before auto-expiring. Shorter timeouts force faster decisions; longer timeouts give more time to review." |

**Current behavior**: SessionAgent reads `config.proposalTimeoutSec` from DO SQLite `session_config` (which already has this column with default 900). PipelineOrchestratorAgent ignores config and hardcodes `900_000` ms.

**New behavior**: Both agents read from `user_trading_config.proposal_timeout_sec`. The session agent's existing `proposalTimeoutSec` in DO SQLite becomes a fallback if PG config is missing. The pipeline orchestrator is updated to accept the value as a parameter instead of hardcoding it.

### 2. llmTemperature

| Attribute | Value |
|-----------|-------|
| Column | `llm_temperature` |
| Type | `real` |
| Default | `0.3` |
| Range | 0.0 - 1.0 |
| Step | 0.05 |
| UI Group | AI Model |
| Tooltip | "Controls LLM creativity. Lower values (0.1-0.3) produce more deterministic, conservative recommendations. Higher values (0.5-0.8) produce more varied, creative analysis." |

**Scaling strategy**: The user sets a single global temperature preference. Each LLM call site applies a task-specific scaling factor to produce the effective temperature:

| Task | Scale Factor | Example (user=0.3) | Example (user=0.6) |
|------|-------------|---------------------|---------------------|
| Trade recommendation | 1.0x | 0.3 | 0.6 |
| Research report | 1.5x | 0.45 | 0.9 |
| Event classification | 1.0x | 0.3 | 0.6 |
| Report generation | 1.5x | 0.45 | 0.9 |
| Persona analysis | 1.2x | 0.36 | 0.72 |
| Consensus synthesis | 1.0x | 0.3 | 0.6 |
| Risk validation | 0.7x | 0.21 | 0.42 |
| Debate response | 1.5x | 0.45 | 0.9 |

The effective temperature is clamped to [0.0, 1.0] after scaling. This preserves the relative task-type variation while giving users a single knob.

### 3. llmMaxTokens

| Attribute | Value |
|-----------|-------|
| Column | `llm_max_tokens` |
| Type | `integer` |
| Default | `1000` |
| Range | 200 - 4000 |
| Step | 100 |
| UI Group | AI Model |
| Tooltip | "Maximum response length from the AI model. Higher values allow more detailed analysis but cost more tokens. Lower values produce concise recommendations." |

**Scaling strategy**: Similar to temperature, the user sets a global preference and each task type applies a multiplier:

| Task | Scale Factor | Example (user=1000) | Min | Max |
|------|-------------|---------------------|-----|-----|
| Trade recommendation | 0.8x | 800 | 200 | 4000 |
| Research report | 2.0x | 2000 | 500 | 4000 |
| Event classification | 0.5x | 500 | 200 | 2000 |
| Report generation | 2.0x | 2000 | 500 | 4000 |
| Persona analysis | 0.8x | 800 | 200 | 4000 |
| Consensus synthesis | 1.0x | 1000 | 200 | 4000 |
| Risk validation | 0.6x | 600 | 200 | 4000 |
| Debate response | 0.6x | 600 | 200 | 4000 |

The effective value is clamped to each task's [min, max] range after scaling.

### 4. scoreWindows

| Attribute | Value |
|-----------|-------|
| Column | `score_windows` |
| Type | `jsonb` |
| Default | `[30, 90, 180]` |
| Constraints | Array of 1-5 integers, each 7-365, sorted ascending, unique |
| UI Group | Display |
| Tooltip | "Performance analysis time windows in days. Controls which time periods appear in the performance dashboard." |

**Impact**: Changes the `ScoreWindow` type from a fixed union (`30 | 90 | 180`) to `number`. The `VALID_WINDOWS` set in `performance-handlers.ts` reads from the user's config instead of being hardcoded. The `WindowSelector` component renders dynamic buttons based on the user's configured windows.

### 5. confidenceDisplayHigh

| Attribute | Value |
|-----------|-------|
| Column | `confidence_display_high` |
| Type | `real` |
| Default | `0.7` |
| Range | 0.5 - 1.0 |
| Step | 0.05 |
| UI Group | Display |
| Tooltip | "Confidence scores at or above this threshold are displayed as 'high' (green). Should be higher than the medium threshold." |

### 6. confidenceDisplayMed

| Attribute | Value |
|-----------|-------|
| Column | `confidence_display_med` |
| Type | `real` |
| Default | `0.4` |
| Range | 0.1 - 0.7 |
| Step | 0.05 |
| UI Group | Display |
| Tooltip | "Confidence scores at or above this threshold (but below the high threshold) are displayed as 'medium' (yellow). Scores below this are displayed as 'low' (red)." |

**Cross-field validation**: `confidenceDisplayHigh` must be greater than `confidenceDisplayMed`. Enforced in both the Zod schema (`.refine()`) and the form validators.

## Type Definitions

### Extended TradingConfig

```ts
// packages/data-ops/src/trading-config/schema.ts (additions)

export const TradingConfigSchema = z.object({
  // ... existing 17 fields ...

  // New Phase 23 fields
  proposalTimeoutSec: z.number().int().min(60).max(3600).default(900),
  llmTemperature: z.number().min(0).max(1).default(0.3),
  llmMaxTokens: z.number().int().min(200).max(4000).default(1000),
  scoreWindows: z.array(z.number().int().min(7).max(365)).min(1).max(5).default([30, 90, 180]),
  confidenceDisplayHigh: z.number().min(0.5).max(1.0).default(0.7),
  confidenceDisplayMed: z.number().min(0.1).max(0.7).default(0.4),
}).refine(
  (data) => data.confidenceDisplayHigh > data.confidenceDisplayMed,
  {
    message: 'High confidence threshold must be greater than medium threshold',
    path: ['confidenceDisplayHigh'],
  },
);
```

### LLM Task Scaling Constants

```ts
// packages/data-ops/src/trading-config/llm-scaling.ts (new file)

export interface LLMTaskScale {
  temperatureScale: number;
  maxTokensScale: number;
  maxTokensMin: number;
  maxTokensMax: number;
}

export const LLM_TASK_SCALES: Record<string, LLMTaskScale> = {
  trade_recommendation: { temperatureScale: 1.0, maxTokensScale: 0.8, maxTokensMin: 200, maxTokensMax: 4000 },
  research_report:      { temperatureScale: 1.5, maxTokensScale: 2.0, maxTokensMin: 500, maxTokensMax: 4000 },
  event_classification: { temperatureScale: 1.0, maxTokensScale: 0.5, maxTokensMin: 200, maxTokensMax: 2000 },
  report_generation:    { temperatureScale: 1.5, maxTokensScale: 2.0, maxTokensMin: 500, maxTokensMax: 4000 },
  persona_analysis:     { temperatureScale: 1.2, maxTokensScale: 0.8, maxTokensMin: 200, maxTokensMax: 4000 },
  consensus_synthesis:  { temperatureScale: 1.0, maxTokensScale: 1.0, maxTokensMin: 200, maxTokensMax: 4000 },
  risk_validation:      { temperatureScale: 0.7, maxTokensScale: 0.6, maxTokensMin: 200, maxTokensMax: 4000 },
  debate_response:      { temperatureScale: 1.5, maxTokensScale: 0.6, maxTokensMin: 200, maxTokensMax: 4000 },
} as const;

export function resolveTaskLLMParams(
  userTemperature: number,
  userMaxTokens: number,
  taskType: keyof typeof LLM_TASK_SCALES,
): { temperature: number; maxTokens: number } {
  const scale = LLM_TASK_SCALES[taskType];
  return {
    temperature: Math.min(1.0, Math.max(0.0, userTemperature * scale.temperatureScale)),
    maxTokens: Math.min(
      scale.maxTokensMax,
      Math.max(scale.maxTokensMin, Math.round(userMaxTokens * scale.maxTokensScale)),
    ),
  };
}
```

## Data Flow

```
Settings UI (trading.tsx)
  |
  | PATCH /trading-config  { proposalTimeoutSec: 600, llmTemperature: 0.5, ... }
  v
user_trading_config (Postgres)
  |
  | getTradingConfig(userId) -- existing query
  v
SessionAgent.loadEffectiveConfig()
  |
  | passes config to orchestrators and LLM agent
  v
Pipeline/Debate Orchestrator        LLMAnalysisAgent
  | reads proposalTimeoutSec          | reads llmTemperature, llmMaxTokens
  | reads scoreWindows                | applies task-specific scaling
  v                                   v
TradeProposal.expiresAt             llm.complete({ temperature, max_tokens })

Frontend Components
  | reads confidenceDisplayHigh/Med from config query
  | reads scoreWindows from config query
  v
trade-proposal-card.tsx             window-selector.tsx
  | dynamic badge colors              | dynamic window buttons
```

## Alternatives Considered

### Per-task-type temperature columns

Instead of a single `llmTemperature` with scaling, have 8 separate columns (`temperatureRecommendation`, `temperatureResearch`, etc.).

**Rejected**: Too many columns for a niche use case. The scaling approach gives users one intuitive knob while preserving task-appropriate variation. Per-task overrides could be a future "Expert Mode" feature.

### Store scoreWindows in session_config (DO SQLite)

Since scoreWindows affects agent behavior (score recalculation), it could live in session_config.

**Rejected**: Score windows are a user-level display preference, not a session-specific setting. They should persist across sessions and be editable in the main settings page. The agents can read from PG config directly during their score recalculation cycle (which already does PG queries).

### Separate "display preferences" table

Create a `user_display_preferences` table for confidenceDisplayHigh, confidenceDisplayMed, and scoreWindows.

**Rejected**: Unnecessary table proliferation. The `user_trading_config` table already stores user preferences. Adding 6 columns is cleaner than a new table with its own CRUD endpoints.

## Open Questions

1. **Score window migration**: When a user changes their score windows (e.g., from [30, 90, 180] to [14, 30, 60]), should existing score data be recalculated for the new windows, or only computed going forward? Recommendation: recompute on next scoring cycle.

2. **Temperature range UX**: Should the slider show descriptive labels ("Conservative" at 0.1, "Balanced" at 0.3, "Creative" at 0.7) in addition to the numeric value? Recommendation: yes, via helper text below the slider.
