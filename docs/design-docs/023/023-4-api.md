# Phase 23: Extended User Settings — Part 4: API

## Overview

No new API endpoints. The existing `GET/PATCH /trading-config` endpoints already handle arbitrary fields from `user_trading_config`. The Zod schema expansion (Part 2) automatically validates the new fields. This part covers the one behavioral change needed and the server function adjustments.

---

## Existing Endpoints (Unchanged Routes)

### GET /trading-config

Handler: `apps/data-service/src/hono/handlers/trading-config-handlers.ts`

Returns all columns from `user_trading_config` for the authenticated user. The new columns are automatically included in the response because `getTradingConfig()` selects all columns and the return type expands with the table definition.

**No code changes needed.**

### PATCH /trading-config

Handler: `apps/data-service/src/hono/handlers/trading-config-handlers.ts`

Accepts a partial update validated against `UpdateTradingConfigRequestSchema` (which is `TradingConfigSchema.partial()`). Clients can send any subset of the new fields.

**Cross-field validation concern**: The `.refine()` on `TradingConfigSchema` that enforces `confidenceDisplayHigh > confidenceDisplayMed` runs on the base schema but NOT on the `.partial()` variant (Zod refinements do not carry over to `.partial()` by default). This needs a service-layer check.

## Service Layer Change

File: `apps/data-service/src/hono/services/trading-config-service.ts`

### Cross-Field Validation on Partial Update

When a PATCH request includes only `confidenceDisplayHigh` or only `confidenceDisplayMed`, the service must merge with the existing values to validate the cross-field constraint.

```ts
export async function updateUserTradingConfig(
  userId: string,
  updates: UpdateTradingConfigRequest,
): Promise<Result<TradingConfig>> {
  // If either confidence threshold is being updated, validate cross-field
  if (updates.confidenceDisplayHigh !== undefined || updates.confidenceDisplayMed !== undefined) {
    const existing = await getTradingConfig(userId);
    const mergedHigh = updates.confidenceDisplayHigh ?? existing?.confidenceDisplayHigh ?? 0.7;
    const mergedMed = updates.confidenceDisplayMed ?? existing?.confidenceDisplayMed ?? 0.4;

    if (mergedHigh <= mergedMed) {
      return err(new AppError(
        'High confidence threshold must be greater than medium threshold',
        400,
        'VALIDATION_ERROR',
      ));
    }
  }

  // If scoreWindows is being updated, ensure sorted + unique
  if (updates.scoreWindows !== undefined) {
    const sorted = [...updates.scoreWindows].sort((a, b) => a - b);
    if (!updates.scoreWindows.every((v, i) => v === sorted[i])) {
      return err(new AppError('Score windows must be sorted ascending', 400, 'VALIDATION_ERROR'));
    }
    if (new Set(updates.scoreWindows).size !== updates.scoreWindows.length) {
      return err(new AppError('Score windows must be unique', 400, 'VALIDATION_ERROR'));
    }
  }

  return upsertTradingConfig(userId, updates);
}
```

**Note**: The Zod schema-level `.refine()` catches this in full-object validation (e.g., when the settings form sends all fields). The service-layer check is a safety net for partial updates via the API.

## Server Functions (Frontend)

File: `apps/user-application/src/core/functions/trading-config/direct.ts`

### No Changes Needed

The existing server functions work as-is:

```ts
// getUserTradingConfig -- returns full TradingConfig (new fields included)
export const getUserTradingConfig = createServerFn({ method: 'GET' })
  .middleware([protectedFunctionMiddleware])
  .handler(async ({ context }): Promise<TradingConfig> => {
    const config = await getTradingConfig(context.userId);
    return TradingConfigSchema.parse(config ?? {});
  });

// updateUserTradingConfig -- accepts partial updates (new fields included)
export const updateUserTradingConfig = createServerFn({ method: 'POST' })
  .middleware([protectedFunctionMiddleware])
  .inputValidator(UpdateTradingConfigRequestSchema)
  .handler(async ({ data, context }) => {
    return upsertTradingConfig(context.userId, data);
  });
```

The `TradingConfigSchema.parse(config ?? {})` call in `getUserTradingConfig` applies defaults for any missing field, so even if the PG row has a null (impossible with NOT NULL defaults, but defensive), the Zod defaults fill in.

### Cross-Field Validation in Server Function

The `updateUserTradingConfig` server function uses `UpdateTradingConfigRequestSchema` (partial) for input validation. For consistency with the API service, add the same merge-and-check logic:

```ts
export const updateUserTradingConfig = createServerFn({ method: 'POST' })
  .middleware([protectedFunctionMiddleware])
  .inputValidator(UpdateTradingConfigRequestSchema)
  .handler(async ({ data, context }) => {
    // Cross-field validation for confidence thresholds
    if (data.confidenceDisplayHigh !== undefined || data.confidenceDisplayMed !== undefined) {
      const existing = await getTradingConfig(context.userId);
      const mergedHigh = data.confidenceDisplayHigh ?? existing?.confidenceDisplayHigh ?? 0.7;
      const mergedMed = data.confidenceDisplayMed ?? existing?.confidenceDisplayMed ?? 0.4;
      if (mergedHigh <= mergedMed) {
        throw new Error('High confidence threshold must be greater than medium threshold');
      }
    }
    return upsertTradingConfig(context.userId, data);
  });
```

However, in practice the settings form always sends both confidence fields together (they are in the same form section). The cross-field check is primarily a safety net for direct API callers.

## Performance Handler Changes

File: `apps/data-service/src/hono/handlers/performance-handlers.ts`

This is covered in Part 3 (Logic) but summarized here for API completeness.

### Current

```ts
const VALID_WINDOWS = new Set([30, 90, 180]);

function parseWindowDays(raw: string | undefined): ScoreWindow {
  const n = Number(raw);
  return VALID_WINDOWS.has(n) ? (n as ScoreWindow) : 30;
}
```

### New

```ts
import { getTradingConfig } from '@repo/data-ops/trading-config';

async function parseWindowDays(raw: string | undefined, userId: string): Promise<number> {
  const config = await getTradingConfig(userId);
  const userWindows: number[] = (config?.scoreWindows as number[]) ?? [30, 90, 180];
  const validWindows = new Set(userWindows);
  const n = Number(raw);
  return validWindows.has(n) ? n : userWindows[0];
}
```

The function becomes async because it reads the user's config. All callers in the handler already use `async` handlers so this is transparent.
