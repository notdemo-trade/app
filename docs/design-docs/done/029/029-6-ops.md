## Implementation Order

### Step 1: Add Helper Methods

Add the four private helper methods to `SessionAgent`:

**File**: `apps/data-service/src/agents/session-agent.ts`

| Method | Type | Purpose |
|--------|------|---------|
| `isMarketOpen()` | `async` | Calls `broker.getClock()`, returns boolean |

| `isCooldownActive(cooldownMinutes)` | `sync` | Queries DO SQLite `proposal_outcomes` for recent losses |
| `filterWatchlist(symbols, blacklist, allowlist)` | `sync` | Filters symbol array against blacklist/allowlist |

**Verification**: `pnpm run lint` passes. Methods are private so no external interface changes.

**Deployment is safe** because:

Redeploy the previous version of `data-service`. The enforcement checks are removed, returning to the pre-Phase 29 behavior (no enforcement). No database rollback needed.

## Risk Assessment

### Low Risk

- **Ticker filtering**: Pure in-memory array operations. Cannot fail.
- **Cooldown check**: DO SQLite query on an existing table. Same pattern as other reads in `SessionAgent`.

### Medium Risk

- **Market hours gate**: Depends on `broker.getClock()`. If the broker agent is unhealthy, the fail-open default allows analysis to proceed. This is the same broker call used by `runOutcomeTrackingCycle()`.
- **Daily loss check**: Depends on `broker.getPortfolioHistory()`. The fail-open default prevents false halts. The Alpaca endpoint returns daily aggregates; mid-day accuracy depends on the broker's update frequency.

### Enforcement Edge Cases

| Scenario | Behavior | Acceptable? |
|----------|----------|-------------|
| User has 0 positions, `maxPositions: 0` | All buy proposals rejected | Yes -- user configured zero positions |
| `maxNotionalPerTrade: 100` but proposal has no notional (qty-based) | Notional check skipped (qty orders go through uncapped) | Acceptable for Phase 29; qty-based cap is a future enhancement |
| `cooldownMinutesAfterLoss: 0` | Cooldown disabled entirely | Yes -- user opted out |
| `tickerBlacklist` and `tickerAllowlist` both empty/null | No filtering applied | Yes -- no restrictions configured |
| Proposal notional is exactly equal to `maxNotionalPerTrade` | Allowed (`>` check, not `>=`) | Yes -- the limit is exclusive. `$5000` is allowed when max is `$5000`. |
| Broker API returns stale position count | Possible: user sells a position, next buy still sees old count | Low risk -- positions update within seconds on Alpaca. The timing window is negligible. |
