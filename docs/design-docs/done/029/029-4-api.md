# Phase 29: Risk Management Config Enforcement — Part 4: API

## No API Changes

This phase does not add, modify, or remove any Hono REST API endpoints. All enforcement logic is internal to the `SessionAgent` Durable Object.

The existing endpoints that interact with the enforcement surface are:

| Endpoint / RPC | How It Is Affected |
|---------------|-------------------|
| `SessionAgent.triggerAnalysis()` (callable) | Now checks market hours, daily loss, cooldown, and ticker filters before running analysis |
| `SessionAgent.approveProposal(id)` (callable) | Delegates to `handleTradeDecision()` which now checks notional caps, position limits, and short-selling before placing orders |
| `SessionAgent.rejectProposal(id)` (callable) | Unchanged -- rejection is not subject to enforcement |
| `PUT /api/trading-config` | Unchanged -- users can still update risk config fields; enforcement reads them at execution time |

---

## Response Shape Changes

### `approveProposal()` / `handleTradeDecision()` Return Type

The return type is unchanged: `{ status: string; message: string }`.

However, the `status` and `message` values now include enforcement-specific rejections:

**Before (only possible values):**

| `status` | `message` | When |
|----------|-----------|------|
| `'executed'` | `Trade executed: buy AAPL` | Order placed successfully |
| `'rejected'` | `Trade rejected by user` | User rejected |
| `'expired'` | `Proposal has expired` | Past expiration time |
| `'error'` | `Proposal not found` | Invalid proposal ID |
| `'error'` | `Proposal already executed` | Already processed |
| `'error'` | `Execution failed: ...` | Broker error |

**After (new enforcement error messages):**

| `status` | `message` | Guard |
|----------|-----------|-------|
| `'error'` | `Trade notional $7500.00 exceeds max $5000 per trade.` | E1: maxNotionalPerTrade |
| `'error'` | `Position value $6000.00 exceeds max $5000 per position.` | E2: maxPositionValue |
| `'error'` | `Maximum 3 positions reached. Close a position first.` | E3: maxPositions |
| `'error'` | `Short selling is disabled. No AAPL position held.` | E4: allowShortSelling |

All enforcement rejections use `status: 'error'`. The message text is human-readable and includes the specific limit value and the offending value where applicable.

### `triggerAnalysis()` Return Type

The return type is unchanged: `{ threadIds: string[] }`.

When pre-analysis guards block the cycle, the return is `{ threadIds: [] }` -- identical to the existing behavior when the watchlist is empty. The caller cannot distinguish between "no symbols to analyze" and "analysis blocked by guard" from the return value alone. The distinction is logged via `console.log()`.

## Observability via SessionState

The `SessionState` interface (broadcast to the frontend via WebSocket) includes `lastError: string | null`. When enforcement guards skip an analysis cycle, the state is **not** updated with an error because skipping is intentional behavior, not an error condition. The state update happens normally:

```ts
// triggerAnalysis() returns { threadIds: [] } on guard skip
// The caller (runScheduledCycle or manual trigger) sees an empty result
// cycleCount and lastCycleAt ARE still updated for guard skips
```

For execution-level rejections, the error message is returned to the caller (`approveProposal()`) which returns it to the frontend via the WebSocket RPC response. The frontend already handles error responses from `approveProposal()` and displays them to the user.

## Frontend Impact

### Trade Approval Flow

The frontend calls `approveProposal(proposalId)` and handles the response:

```ts
// Existing pattern in session UI
const result = await sessionAgent.approveProposal(proposalId);
if (result.status === 'error') {
  // Show error toast or inline message
  showError(result.message);
}
```

Enforcement rejections use the same `status: 'error'` pattern. The existing error handling in the frontend will display the enforcement message to the user without any code changes. The message text (e.g., "Maximum 3 positions reached. Close a position first.") is specific enough for the user to understand what happened and what to do.

### Analysis Trigger Flow

When the user manually triggers analysis via the chat interface or a button, `triggerAnalysis()` returns `{ threadIds: [] }`. The frontend already handles empty thread ID arrays (no new threads to display). The skip reason is logged server-side but not surfaced to the user in the current UI. A future enhancement could add a "last skip reason" field to `SessionState`.

## Security Considerations

### No Bypass Path

Enforcement is applied unconditionally in `handleTradeDecision()`. There is no parameter or flag to bypass the checks. The only way to change enforcement behavior is to update the config values through the authenticated trading config API (`PUT /api/trading-config`), which requires a valid user session.

### Config Consistency

`handleTradeDecision()` calls `this.loadEffectiveConfig()` at the start of execution, reading the latest config from PostgreSQL and DO SQLite. This means config changes take effect on the next trade decision, not retroactively on already-approved trades. There is no race condition between config updates and trade execution because:

1. The user updates config via the REST API (writes to PostgreSQL)
2. The user approves a proposal via WebSocket RPC
3. `handleTradeDecision()` loads fresh config (reads from PostgreSQL)
4. Enforcement checks use the fresh config

Steps 2 and 3 happen sequentially within the same Durable Object method call.

### Rate Limiting

The `getClock()` and `getPortfolioHistory()` calls in pre-analysis guards add broker API load. With the default 120-second analysis interval, this means ~30 additional broker calls per hour (2 calls per cycle: clock + history). Alpaca's rate limit is 200 requests per minute for paper trading and 200 for live trading. The additional load is negligible.

---

## WebSocket Message Compatibility

The enforcement changes do not modify the WebSocket message format. The `broadcastThread()` method in `SessionAgent` broadcasts `SessionState` updates. The state shape is unchanged. Thread and proposal data shapes are unchanged. The only visible difference is:

1. Fewer threads/proposals may be created (when pre-analysis guards skip symbols)
2. Some approved proposals may be rejected with enforcement messages (when pre-execution guards block)

Both are represented through existing data structures that the frontend already handles.
