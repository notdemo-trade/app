# Phase 2: Credentials & Configuration


## Overview

Enable BYOK (bring your own keys) model where users store encrypted API keys and configure trading parameters.

**Deploy outcome**: Users can add Alpaca keys, LLM provider keys, configure trading limits, validate connections.


## Context

Phase 1 establishes auth + per-user API tokens. Phase 2 adds credential storage for external services (Alpaca, LLMs) and trading configuration per user.

**Source configs (mahoraga)**:
- `AgentConfig`: trading intervals, limits, LLM settings, options config
- `PolicyConfig`: position limits, loss limits, order restrictions
- `AlpacaClientConfig`: apiKey, apiSecret, paper mode
- `LLM factory`: OpenAI, Anthropic, Google, xAI, DeepSeek support

---


## Goals & Non-Goals

**Goals**:
- Encrypted credential storage (AES-256-GCM)
- Alpaca key management (paper/live toggle)
- Multi-provider LLM key storage
- Per-user trading configuration
- Connection validation on save

**Non-Goals**:
- Automatic key rotation (future)
- Key sharing between users
- Credential backup/export
- Rate limiting validation endpoint (defer to Phase 5+)
- Social/notification credentials — Twitter, StockTwits, Telegram (defer to signal source phase)

## Environment Setup

### Additional Environment Variables (Phase 2)

Add to existing environment files from Phase 1:

```bash
# Encryption (for user credential storage)
CREDENTIALS_ENCRYPTION_KEY="<generate-with: openssl rand -base64 32>"
```

**Setup**:

1. Generate encryption key:
   ```bash
   openssl rand -base64 32
   ```

2. Add to both apps' environment files:
   - `apps/data-service/.staging.vars`
   - `apps/user-application/.env.staging`

3. Sync to Cloudflare:
   ```bash
   cd apps/data-service && ./sync-secrets.sh staging
   cd apps/user-application && ./sync-secrets.sh staging
   ```

**Security notes**:
- This key encrypts all user API credentials (Alpaca, LLM providers)
- Store securely - losing this key makes all credentials unrecoverable
- Use different keys for staging vs production
- 32 bytes (256 bits) required for AES-256-GCM

## Features

### 2.1 Credentials Schema

Encrypted storage for user API keys in Neon Postgres.

**Encryption approach**:
- AES-256-GCM with per-user salt
- Master encryption key in CF secrets (`CREDENTIALS_ENCRYPTION_KEY`)
- Derive per-user key: `HKDF(master_key, user_id + salt)`

**Why not at-rest DB encryption only**:
- Defense in depth - compromised DB dump useless without master key
- Different keys per user limits blast radius

### 2.2 Alpaca Credentials UI

- Add/edit Alpaca key + secret
- Paper/live toggle (default paper)
- Display connection status (last verified)

### 2.3 LLM Provider Credentials UI

Support 5 providers (from mahoraga `ai-sdk.ts`):

| Provider | Env Key | Models |
|----------|---------|--------|
| OpenAI | `OPENAI_API_KEY` | gpt-4o-mini, gpt-4o, o1, o1-mini |
| Anthropic | `ANTHROPIC_API_KEY` | claude-3-7-sonnet-latest, claude-sonnet-4-0, claude-opus-4-1, claude-3-5-haiku-latest |
| Google | `GOOGLE_GENERATIVE_AI_API_KEY` | gemini-2.5-pro, gemini-2.5-flash, gemini-2.0-flash, gemini-3-pro-preview |
| xAI | `XAI_API_KEY` | grok-4, grok-3, grok-4-fast-reasoning |
| DeepSeek | `DEEPSEEK_API_KEY` | deepseek-chat, deepseek-reasoner |

### 2.4 Provider Selection

Per-user model selection:
- **Research model**: used for market research/sentiment
- **Analyst model**: used for trade decisions

Format: `provider/model` (e.g., `openai/gpt-4o`, `anthropic/claude-sonnet-4-0`)

### 2.5 Trading Config Schema

Based on mahoraga `AgentConfig` + `PolicyConfig`:

**Core limits**:
- `max_position_value`: max $ per position (default: 5000)
- `max_positions`: max concurrent positions (default: 10)
- `max_notional_per_trade`: max $ per single trade (default: 5000)
- `max_daily_loss_pct`: daily loss limit as % of equity (default: 2%)

**Position management**:
- `take_profit_pct`: auto-close at gain % (default: 15%)
- `stop_loss_pct`: auto-close at loss % (default: 8%)
- `position_size_pct_of_cash`: % of available cash per trade (default: 10%)

**Advanced (phase 3+)**:
- Options config (min/max DTE, delta range, etc.)
- Crypto config
- Stale position handling

### 2.6 Trading Config UI

Settings page with sections:
- Position limits (max positions, max notional)
- Risk management (stop loss, take profit, daily loss limit)
- Model selection (research model, analyst model)

### 2.7 Credential Validation

Test connection on save:
- **Alpaca**: call `GET /v2/account` - verify returns 200
- **LLM providers**: GET models endpoint (or minimal POST for Anthropic)

Return validation result with error message if failed.

**Note**: Paper/live toggle triggers re-save (encrypted blob includes `paper` flag), which clears validation status. Re-validation runs automatically if `validate: true`.

### 2.9 Credential Expiration Warning

Show warning on dashboard when credentials approach expiration:
- Check `lastValidatedAt` + provider-specific TTL
- Display warning banner 3 days before
- For providers with expiring keys (some API keys have TTL)

### 2.10 Credential Caching in Durable Object

During agent run, decrypt credentials once and cache in DO memory:
- Avoid repeated DB queries + decryption per trade
- Clear cache on credential update (webhook or invalidate on save)
- Cache TTL: duration of agent session
