## Rollback Strategy

### If Step 2 (parameterization) causes issues:

The `config` parameter defaults to `DEFAULT_TA_CONFIG`, which contains the exact same values as the current hardcoded constants. If no user has saved a custom config, behavior is identical to pre-Phase 22.

### If `TechnicalIndicators` shape change breaks consumers:

The `getSmaValue(indicators, period)` helper provides backward-compatible access. A temporary compatibility layer could map the new shape to the old field names if needed for a phased rollout.

### Database rollback:

The `technical_analysis_config` table is independent -- dropping it has zero impact on existing functionality. All code defaults to `DEFAULT_TA_CONFIG` when no row exists.

## Open Questions

1. **Should config be cached in the agent's DO SQLite?** Currently, `getTaConfig()` hits Postgres on every analysis cycle (~every 120s). This is fine for Neon serverless but adds latency. Alternative: cache in `TechnicalAnalysisAgent` SQLite with a TTL, invalidate on config change via a flag.

2. **Should the frontend fetch config separately or receive it with analysis results?** Current design fetches config separately for the settings page and passes it as a prop to indicator panels. An alternative would be to include `config` in the `AnalysisResult` returned by the agent, so the frontend always has the config that was used for that specific analysis.

3. **Per-symbol config in the future?** The current design is one config per user. If per-symbol config is needed later, the table would need a `symbol` column and the unique constraint would change to `(userId, symbol)`. The schema is designed to make this addition straightforward.
