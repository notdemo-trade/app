# Phase 22: Technical Analysis Configuration Profiles — Part 1: Spec

## Overview

Replace all hardcoded technical indicator periods, signal detection thresholds, and agent analysis settings with a per-user configuration system. Users choose from preset profiles (Default, Day Trader, Swing Trader, Position Trader) or fine-tune individual parameters. Configuration is stored in Postgres (`technical_analysis_config` table) and flows through the entire analysis pipeline from `computeTechnicals()` to the frontend display.

### Current Architecture Problem

Technical analysis parameters are scattered across four layers with no way to customize:

1. **`packages/data-ops/src/providers/technicals/calculations.ts`** -- `computeTechnicals()` calls `calculateSMA(closes, 20)`, `calculateSMA(closes, 50)`, `calculateSMA(closes, 200)`, etc. with literal constants. The `TechnicalIndicators` type has field names baked to specific periods (`sma_20`, `sma_50`, `sma_200`, `rsi_14`, `ema_12`, `ema_26`, `atr_14`, `volume_sma_20`).

2. **`packages/data-ops/src/providers/technicals/signals.ts`** -- `detectSignals()` uses `< 30` for RSI oversold, `> 70` for overbought, `> 2.0` for volume spike, `+ 0.3` for BB strength bias, `* 10` for SMA cross strength, `/ 4` for volume strength divisor.

3. **Agent files** -- `technical-analysis-agent.ts` requires `bars.length < 50` and fetches 250 bars. `alpaca-market-data-agent.ts` uses `CACHE_FRESHNESS_MS = 60_000`. `debate-orchestrator-agent.ts` checks `score.totalProposals < 5`. `llm-analysis-agent.ts` uses calibration thresholds `>= 0.5` (good), `>= 0.2` (fair).

4. **Frontend** -- `indicator-panels.tsx` uses `value < 30` / `value > 70` for RSI coloring and `relativeVolume > 2` for volume alert coloring.

### Refactor Outcome

- Single `TechnicalAnalysisConfig` type flows from DB through agents to frontend
- `computeTechnicals()` and `detectSignals()` accept config parameter
- `TechnicalIndicators` type uses dynamic keys (`sma` array instead of `sma_20`, `sma_50`, `sma_200`)
- Agents fetch user config at analysis time
- Frontend reads thresholds from config for display logic
- Four preset profiles cover common trading styles
- Users can fine-tune any parameter individually

---

## Goals

- Define a `technical_analysis_config` Postgres table with one row per user
- Create `TechnicalAnalysisConfigSchema` (Zod) with validated ranges for all parameters
- Parameterize `computeTechnicals()` to accept indicator periods from config
- Parameterize `detectSignals()` to accept signal thresholds from config
- Update `TechnicalIndicators` type to support dynamic SMA/EMA periods
- Provide four preset profiles: Default, Day Trader, Swing Trader, Position Trader
- Add REST API: GET, PUT, POST preset, POST reset
- Add settings UI with grouped form sections and preset selector
- Pass config through agents: `TechnicalAnalysisAgent.analyze()` reads user config
- Update `indicator-panels.tsx` to read thresholds from config

## Non-Goals

- Per-symbol configuration (all symbols use the same user profile) -- future phase
- Multiple saved profiles per user (one active profile, switch via presets) -- future phase
- Backtesting with different configs -- future phase
- Sharing configs between users -- future phase
- Real-time config sync via WebSocket (page refresh is sufficient) -- future phase

## Configuration Schema

### Indicator Periods

| Parameter | Type | Default | Range | Description |
|-----------|------|---------|-------|-------------|
| `smaPeriods` | `number[]` | `[20, 50, 200]` | each 5-500 | Simple Moving Average periods |
| `emaPeriods` | `number[]` | `[12, 26]` | each 5-500 | Exponential Moving Average periods |
| `rsiPeriod` | `integer` | `14` | 5-50 | RSI calculation period |
| `bollingerPeriod` | `integer` | `20` | 10-50 | Bollinger Bands SMA period |
| `bollingerStdDev` | `real` | `2.0` | 1.0-3.0 | Bollinger Bands standard deviation multiplier |
| `atrPeriod` | `integer` | `14` | 5-50 | Average True Range period |
| `volumeSmaPeriod` | `integer` | `20` | 5-50 | Volume SMA period (for relative volume) |
| `macdSignalPeriod` | `integer` | `9` | 5-20 | MACD signal line EMA period |

### Signal Thresholds

| Parameter | Type | Default | Range | Description |
|-----------|------|---------|-------|-------------|
| `rsiOversold` | `integer` | `30` | 10-40 | RSI level below which = oversold signal |
| `rsiOverbought` | `integer` | `70` | 60-90 | RSI level above which = overbought signal |
| `volumeSpikeMultiplier` | `real` | `2.0` | 1.2-5.0 | Relative volume above this = high volume signal |

### Analysis Settings

| Parameter | Type | Default | Range | Description |
|-----------|------|---------|-------|-------------|
| `minBarsRequired` | `integer` | `50` | 20-200 | Minimum bars needed to run analysis |
| `defaultBarsToFetch` | `integer` | `250` | 100-500 | Default number of bars to request from data provider |
| `cacheFreshnessSec` | `integer` | `60` | 10-300 | Market data cache TTL in seconds |

> **Note**: `minPatternSamples` (currently 5 in debate-orchestrator-agent) and calibration thresholds (0.5/0.2 in llm-analysis-agent) are excluded from this config. Those are agent memory/scoring parameters, not technical analysis settings. They belong in a future "Agent Tuning" config if ever exposed.

## Data Flow

### Analysis Pipeline (with config)

```
User saves config (settings UI)
  → PUT /api/technical-config
  → Upsert technical_analysis_config row (Postgres)

Analysis cycle triggers
  → TechnicalAnalysisAgent.analyze(timeframe, bars?)
    → Fetch user's TechnicalAnalysisConfig from Postgres (or defaults)
    → AlpacaMarketDataAgent.fetchBars(symbol, timeframe, { limit: config.defaultBarsToFetch })
    → Validate bars.length >= config.minBarsRequired
    → computeTechnicals(symbol, bars, config)
       → Uses config.smaPeriods, config.emaPeriods, config.rsiPeriod, etc.
    → detectSignals(indicators, config)
       → Uses config.rsiOversold, config.rsiOverbought, config.volumeSpikeMultiplier
    → Return AnalysisResult with config attached

Frontend renders indicators
  → Fetch user's TechnicalAnalysisConfig via server function
  → Pass thresholds to IndicatorPanels
  → RSI coloring uses config.rsiOversold / config.rsiOverbought
  → Volume alert uses config.volumeSpikeMultiplier
```

### Config Resolution Priority

Following the pattern established in Phase 21 (config resolution hierarchy):

```
technical_analysis_config (PG)  ← user's explicit settings (highest priority)
  | fallback
preset profile                  ← when user applies a preset
  | fallback
DEFAULT_TA_CONFIG               ← hardcoded defaults in code (lowest priority)
```

When a user has no `technical_analysis_config` row, `DEFAULT_TA_CONFIG` is used. When a user applies a preset, it overwrites all fields. When a user edits individual fields, only those fields change.

## TechnicalIndicators Type Change

### Current (fixed field names)

```ts
interface TechnicalIndicators {
  symbol: string;
  timestamp: string;
  price: number;
  sma_20: number | null;
  sma_50: number | null;
  sma_200: number | null;
  ema_12: number | null;
  ema_26: number | null;
  rsi_14: number | null;
  macd: { macd: number; signal: number; histogram: number } | null;
  bollinger: { upper: number; middle: number; lower: number; width: number } | null;
  atr_14: number | null;
  volume_sma_20: number | null;
  relative_volume: number | null;
}
```

### New (dynamic, config-driven)

```ts
interface SMAResult {
  period: number;
  value: number | null;
}

interface EMAResult {
  period: number;
  value: number | null;
}

interface TechnicalIndicators {
  symbol: string;
  timestamp: string;
  price: number;
  sma: SMAResult[];
  ema: EMAResult[];
  rsi: number | null;
  macd: { macd: number; signal: number; histogram: number } | null;
  bollinger: { upper: number; middle: number; lower: number; width: number } | null;
  atr: number | null;
  volumeSma: number | null;
  relativeVolume: number | null;
}
```

### Migration Impact

All consumers of `TechnicalIndicators` must update:
- `signals.ts` -- access `ind.sma` array instead of `ind.sma_50` / `ind.sma_200`
- `indicator-panels.tsx` -- iterate `indicators.sma` array
- `technical-analysis-agent.ts` -- stored JSON format changes
- `llm-analysis-agent.ts` -- prompt formatting reads from arrays
- LLM prompts that reference indicators by name

The golden/death cross signal detection needs the two longest SMA values from the `sma` array rather than hardcoded `sma_50` and `sma_200`.

## Relationship to Phase 21

Phase 21 addresses configuration conflicts between `user_trading_config` (PG), session config (DO SQLite), and strategy profiles. Phase 22 is orthogonal -- it introduces a new `technical_analysis_config` table that does not overlap with any existing config surface.

| Config Surface | Phase 21 Scope | Phase 22 Scope |
|---------------|---------------|---------------|
| `user_trading_config` (PG) | Position sizing, LLM model | Not touched |
| `session_config` (DO SQLite) | Orchestration mode, analysis interval | Not touched |
| `strategy_profile` | Risk tolerance, position bias | Not touched |
| `technical_analysis_config` (PG) | N/A (does not exist yet) | New table, full scope |

Phase 22 can be implemented independently of Phase 21. The config resolution function from Phase 21 does not need to know about TA config.
