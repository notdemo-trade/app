### Step 3: Add Dedup Guard to `runAnalysisForSymbol()`

**File**: `apps/data-service/src/agents/session-agent.ts`

Insert the dedup check at the top of `runAnalysisForSymbol()`, before the thread creation:

```ts
const existingActive = this.sql<CountRow>`
    SELECT COUNT(*) as cnt FROM trade_proposals
    WHERE symbol = ${symbol} AND status IN ('pending', 'approved')`;
if ((existingActive[0]?.cnt ?? 0) > 0) {
    return {
        threadId: '',
        summary: `Skipped ${symbol}: active proposal already exists`,
    };
}
```

**Verification**: Start a session, trigger analysis, verify proposals are created. Trigger again without resolving proposals, verify analysis is skipped for symbols with pending proposals. Resolve a proposal, trigger again, verify analysis runs for that symbol.

## Rollback Plan

If issues are discovered after deployment:

1. **Redeploy previous version of `data-service`**: `wrangler rollback` or redeploy the previous commit.
2. **Impact of rollback**: Duplicate proposals may resume. Exit order matching reverts to "most recent" heuristic. Pipeline proposals use the empty-then-patched `threadId` pattern. All are pre-existing behaviors with no data corruption risk.
3. **No data cleanup needed**: The dedup guard is a skip-on-read pattern. It does not modify the `trade_proposals` table schema or data. Rolling back simply re-enables the old behavior.

## Alternatives Considered

### Alternative A: Expiration Sweep Before Dedup Check

Add a sweep at the top of `triggerAnalysis()` that updates `status = 'expired'` for all proposals where `expiresAt < Date.now()`:

```ts
this.sql`UPDATE trade_proposals SET status = 'expired'
    WHERE status = 'pending' AND expires_at < ${Date.now()}`;
```

**Deferred**: This would eliminate the edge case of expired-but-not-updated proposals blocking analysis. However, it changes the proposal lifecycle semantics -- currently, expiration is detected at decision time, and the user sees the "expired" status only when they try to act. A proactive sweep would change proposals to "expired" before the user sees them, which could be confusing. This is a separate design decision.

### Alternative B: Per-Action Dedup (Allow BUY While SELL Pending)

Instead of blocking all analysis for a symbol with any active proposal, only block if the active proposal has the same action:

```sql
SELECT COUNT(*) as cnt FROM trade_proposals
WHERE symbol = ? AND action = ? AND status IN ('pending', 'approved')
```

**Rejected**: This would allow contradictory proposals (BUY and SELL) for the same symbol simultaneously. The user could approve both, resulting in a buy followed immediately by a sell (or vice versa), which is likely unintended. The conservative approach of blocking any active proposal for the symbol is simpler and safer.

### Alternative C: Use `client_order_id` for Exit Order Matching

Set `client_order_id` to the proposal ID when placing the order, then match exit orders by `client_order_id`:

```ts
const exitOrder = orders.find(o =>
    o.clientOrderId === `exit-${outcome.proposalId}` && o.status === 'filled'
);
```

**Deferred**: This requires modifying the order placement flow in `AlpacaBrokerAgent` to set `client_order_id` on both entry and exit orders. It also requires a convention for exit order IDs (e.g., `exit-{proposalId}`), which only works for system-initiated exits (stop-loss, target), not manual closes. The ascending-sort heuristic is a sufficient improvement for now.

## Open Questions

1. **Should the dedup guard also check `status = 'executed'` with `outcomeStatus = 'tracking'`?** A proposal that has been executed and is being tracked for outcomes represents an open position. Generating a new BUY proposal while tracking an existing BUY position could lead to doubling down. However, this is a different concern (position management) and the current design explicitly allows new analysis for symbols with resolved proposals. Deferring this to a future phase that introduces position-aware analysis.

2. **Should skipped symbols be logged to the discussion_messages table?** Currently the skip is silent (no thread, no messages). Adding a system-level log entry would require creating a thread just to log the skip, which contradicts the goal of avoiding empty threads. The `console.log` approach documented in Part 4 is sufficient for debugging.
