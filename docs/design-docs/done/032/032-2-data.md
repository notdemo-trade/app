## DO SQLite Queries

### Dedup Guard Query

The dedup check queries the existing `trade_proposals` table in DO SQLite. No schema changes are needed -- the table already has the `symbol` and `status` columns.

**Existing table** (created in `SessionAgent.onStart()`):

```sql
CREATE TABLE IF NOT EXISTS trade_proposals (
    id                TEXT PRIMARY KEY,
    thread_id         TEXT NOT NULL,
    symbol            TEXT NOT NULL,
    action            TEXT NOT NULL,
    confidence        REAL NOT NULL,
    rationale         TEXT NOT NULL,
    entry_price       REAL,
    target_price      REAL,
    stop_loss         REAL,
    qty               REAL,
    notional          REAL,
    position_size_pct REAL,
    risks             TEXT NOT NULL DEFAULT '[]',
    warnings          TEXT NOT NULL DEFAULT '[]',
    expires_at        INTEGER NOT NULL,
    status            TEXT NOT NULL DEFAULT 'pending',
    created_at        INTEGER NOT NULL,
    decided_at        INTEGER,
    order_id          TEXT,
    filled_qty        REAL,
    filled_avg_price  REAL,
    outcome_status    TEXT NOT NULL DEFAULT 'none'
)
```

**New query** (used in the dedup guard):

```sql
SELECT COUNT(*) as cnt FROM trade_proposals
WHERE symbol = ? AND status IN ('pending', 'approved')
```

This returns a count of active (pending or approved) proposals for a given symbol. If `cnt > 0`, the analysis is skipped.

**Why `approved` is included**: An `approved` proposal has been accepted by the user but the order may not yet be filled. Generating a new proposal for the same symbol while an order is in flight would risk the same over-concentration issue as duplicate pending proposals.

**Terminal statuses excluded**: `executed`, `rejected`, `expired`, `failed` are all terminal. A symbol with only terminal proposals is eligible for new analysis.

## Type Changes

### Modified Interface: `RunPipelineParams`

**File**: `apps/data-service/src/agents/pipeline-orchestrator-agent.ts`

Add `threadId` field:

```ts
export interface RunPipelineParams {
    symbol: string;
    strategyId: string;
    strategy: StrategyTemplate;
    onMessage: (msg: Omit<DiscussionMessage, 'id' | 'threadId' | 'timestamp'>) => void;
    llmPrefs?: { temperature: number; maxTokens: number };
    proposalTimeoutSec?: number;
    scoreWindows?: number[];
    portfolioContext?: PortfolioContext;
    threadId: string; // NEW -- passed through to buildProposal()
}
```

The `threadId` field is required (not optional) because every pipeline invocation originates from a discussion thread. There is no valid call path where `threadId` would be absent.

## Data Flow Changes

### Dedup Guard Data Flow

```
triggerAnalysis()
  -> for each symbol in watchlistSymbols:
       -> runAnalysisForSymbol(symbol, config, portfolioContext)
            -> NEW: SELECT COUNT(*) FROM trade_proposals
                    WHERE symbol = ? AND status IN ('pending', 'approved')
            -> cnt > 0?
                 -> return { threadId: '', summary: 'Skipped: active proposal exists' }
            -> cnt == 0?
                 -> proceed with debate/pipeline analysis (unchanged)
```

### Exit Order Matching Data Flow (unchanged structure, corrected sort)

```
recordOutcome()
  -> fetchOrders() from broker
  -> findExitOrder(orders, outcome)
       -> filter: side == exitSide && status == 'filled' && createdAt > outcome.createdAt
       -> sort: ascending by createdAt  (CHANGED from descending)
       -> return candidates[0]          (first = earliest, not most recent)
```

### Pipeline `threadId` Data Flow

**Before**:
```
runPipelineAnalysis(threadId, symbol, ...)
  -> pipeline.runPipeline({ symbol, ... })           // threadId NOT passed
       -> buildProposal(ctx, params)                  // threadId: ''
       -> return { proposal: { threadId: '' }, ... }
  -> proposal = { ...result.proposal, threadId }      // patched here
  -> storeProposal(proposal)
```

**After**:
```
runPipelineAnalysis(threadId, symbol, ...)
  -> pipeline.runPipeline({ symbol, ..., threadId })  // threadId PASSED
       -> buildProposal(ctx, params)                  // threadId: params.threadId
       -> return { proposal: { threadId: '...' }, ... }
  -> storeProposal(result.proposal)                   // no spread needed
```
