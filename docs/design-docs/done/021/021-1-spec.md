# Phase 21: Fix Configuration Inconsistencies — Part 1: Spec

## Overview

The trading application has three independent configuration layers that define overlapping settings with conflicting default values. When a user starts a session, the effective configuration depends on which code path runs first and whether any layer was explicitly populated -- leading to unpredictable behavior.

This design establishes a single `resolveEffectiveConfig()` function that deterministically merges all configuration sources into one `EffectiveConfig` object, used consistently throughout the session lifecycle.

### Current Architecture Problem

```
user_trading_config (PG)          session_config (DO SQLite)          strategy_profiles (DO SQLite)
  positionSizePctOfCash: 0.1        positionSizePctOfCash: 0.05        conservative: 0.03
  analystModel: openai/gpt-4o       llmModel: workers-ai/llama-3.3     moderate:     0.05
  researchModel: openai/gpt-4o-mini llmProvider: workers-ai             aggressive:   0.10
         |                                  |                                   |
         v                                  v                                   v
      IGNORED by session agent       USED as primary config              USED independently,
                                                                         overrides session config
```

None of these layers reference each other. The session agent reads only from DO SQLite. The user's Postgres trading config is never consulted during analysis cycles. Strategy profiles hardcode their own position sizes without checking what the user configured.

## Context & Background

### Where Defaults Live Today

**1. `user_trading_config` table (Postgres)**

File: `packages/data-ops/src/trading-config/table.ts`

```
positionSizePctOfCash: 0.1   (10%)
researchModel: 'openai/gpt-4o-mini'
analystModel: 'openai/gpt-4o'
```

This is the user-facing configuration stored in Postgres via the settings UI. It represents the user's explicit preferences. Created when the user first saves settings.

**2. `DEFAULT_SESSION_CONFIG` (code constant)**

File: `packages/data-ops/src/agents/session/defaults.ts` (lines 60-72)

```
positionSizePctOfCash: 0.05  (5%)
llmProvider: 'workers-ai'
llmModel: '@cf/meta/llama-3.3-70b-instruct-fp8-fast'
```

This is the fallback when no row exists in the session_config DO SQLite table. The session agent's `loadConfig()` returns this when the `session_config` table has only the seeded row with SQLite-level defaults (which also say 0.05).

**3. `DEFAULT_STRATEGIES` (code constant)**

File: `apps/data-service/src/agents/session-agent-helpers.ts` (lines 226-251)

```
conservative: { positionSizeBias: 0.03 }
moderate:     { positionSizeBias: 0.05 }
aggressive:   { positionSizeBias: 0.10 }
```

These are seeded into the `strategy_templates` DO SQLite table on first start. The `positionSizeBias` field is passed to the orchestrator as part of the `StrategyTemplate` but its relationship to `positionSizePctOfCash` is undefined -- there is no code that reconciles these two fields.

**4. `session_config` SQLite table defaults**

File: `apps/data-service/src/agents/session-agent.ts` (lines 971-985)

```sql
position_size_pct REAL NOT NULL DEFAULT 0.05
llm_provider TEXT NOT NULL DEFAULT 'workers-ai'
llm_model TEXT NOT NULL DEFAULT '@cf/meta/llama-3.3-70b-instruct-fp8-fast'
```

These SQLite DDL defaults match the code constant but conflict with the Postgres table defaults.

### How Config Flows Today

```
SessionAgent.onStart()
  -> seedDefaults()           -- inserts row into session_config with SQLite defaults
  -> loadConfig()             -- reads from session_config SQLite table
                              -- NEVER reads from user_trading_config (PG)

SessionAgent.runAnalysisForSymbol(symbol, config)
  -> getActiveStrategy(config) -- reads strategy from strategy_templates SQLite
  -> runDebateAnalysis(...)    -- passes strategy.positionSizeBias to orchestrator
                               -- config.positionSizePctOfCash used only as fallback
                               --   in createProposal() when consensus.positionSizePct is null

handleTradeDecision(proposalId, approved)
  -> uses proposal.positionSizePct to compute notional
  -> this value came from consensus OR config fallback -- never from user_trading_config
```

### Impact

- A user who sets position size to 10% in the settings UI will have their preference ignored -- the session agent always uses 5% (or the strategy profile's value).
- A user who configures `openai/gpt-4o` as their analyst model will still get `workers-ai/llama-3.3` because the session agent never reads `user_trading_config.analystModel`.
- Strategy profiles operate as independent islands -- changing position size in settings has zero effect on what Conservative/Moderate/Aggressive actually do.

---

## Goals

1. **Single resolution function**: Create `resolveEffectiveConfig()` in `packages/data-ops` that merges all config layers with deterministic priority.
2. **Trading config as source of truth**: `user_trading_config` (PG) values take highest priority when they exist.
3. **Strategy profiles as presets, not overrides**: Strategy `positionSizeBias` is a suggestion; the user's explicit `positionSizePctOfCash` wins.
4. **LLM model alignment**: Session agent uses the user's configured LLM models from `user_trading_config` when available, falling back to session config, then code defaults.
5. **Backward compatible**: Existing sessions with no `user_trading_config` row continue working with current session defaults.
6. **Auditable**: Log which layer each resolved value came from for debugging.

## Non-Goals

- Migrating `user_trading_config` schema (no column additions or removals).
- Changing the settings UI (it already writes to `user_trading_config`).
- Adding new configuration fields.
- Changing strategy profile definitions (only changing how they interact with user config).
- Real-time config sync (config is resolved at cycle start, not mid-cycle).

## Conflict Inventory

### Conflict 1: Position Sizing

| Source | Field | Default | When Used |
|--------|-------|---------|-----------|
| `user_trading_config` (PG) | `positionSizePctOfCash` | `0.1` (10%) | Never -- not read by session agent |
| `session_config` (DO SQLite) | `position_size_pct` | `0.05` (5%) | Fallback in `createProposal()` |
| Strategy profile | `positionSizeBias` | 0.03 / 0.05 / 0.10 | Passed to orchestrator, used in LLM prompts |
| `DEFAULT_SESSION_CONFIG` | `positionSizePctOfCash` | `0.05` (5%) | Code-level fallback |

**Resolution**: `user_trading_config.positionSizePctOfCash` wins. If null/missing, fall back to `session_config.position_size_pct`. Strategy `positionSizeBias` is used only in LLM prompts as analytical context (not for actual order sizing).

### Conflict 2: LLM Model Selection

| Source | Field | Default | When Used |
|--------|-------|---------|-----------|
| `user_trading_config` (PG) | `analystModel` | `openai/gpt-4o` | Never -- not read by session agent |
| `user_trading_config` (PG) | `researchModel` | `openai/gpt-4o-mini` | Never -- not read by session agent |
| `session_config` (DO SQLite) | `llm_provider` + `llm_model` | `workers-ai` + `@cf/meta/llama-3.3-70b-instruct-fp8-fast` | Used by `resolveProviderConfig()` |
| `DEFAULT_SESSION_CONFIG` | `llmProvider` + `llmModel` | Same as above | Code-level fallback |

**Resolution**: Parse `user_trading_config.analystModel` (format: `provider/model`) into provider + model. If present and user has valid credentials for that provider, use it. Otherwise fall back to `session_config.llmProvider` + `session_config.llmModel`. The `researchModel` field maps to a future multi-model setup; for now, analyst model is used for all analysis.

### Conflict 3: Strategy Profiles Ignore Trading Config

| Behavior | Current | Target |
|----------|---------|--------|
| `positionSizeBias` in strategy | Used as the effective position size in LLM prompts | Used as analytical context only; actual sizing comes from resolved config |
| User changes position size in settings | No effect on strategy behavior | Overrides strategy's position size suggestion |
| Applying a strategy profile | Hardcodes position size | Applies profile defaults to session config, user can override |

**Resolution**: When a strategy profile is applied, it writes its defaults to `session_config` as a baseline. The user's `user_trading_config` values always override. The `positionSizeBias` field in `StrategyTemplate` remains for LLM prompt context but is not used for order sizing.

## Design

### New Type: `EffectiveConfig`

Location: `packages/data-ops/src/agents/session/types.ts`

```ts
export interface EffectiveConfig {
  // Resolved values (used for execution)
  positionSizePctOfCash: number;
  llmProvider: LLMProviderName;
  llmModel: string;
  maxPositionValue: number;
  maxPositions: number;
  maxNotionalPerTrade: number;
  maxDailyLossPct: number;
  takeProfitPct: number;
  stopLossPct: number;
  cooldownMinutesAfterLoss: number;
  tradingHoursOnly: boolean;
  extendedHoursAllowed: boolean;
  allowShortSelling: boolean;
  tickerBlacklist: string[];
  tickerAllowlist: string[] | null;

  // Session-specific (not in trading config)
  orchestrationMode: 'debate' | 'pipeline';
  brokerType: string;
  watchlistSymbols: string[];
  analysisIntervalSec: number;
  minConfidenceThreshold: number;
  activeStrategyId: string;
  debateRounds: number;
  proposalTimeoutSec: number;

  // Provenance tracking (for debugging)
  _sources: Record<string, 'trading_config' | 'session_config' | 'strategy_profile' | 'default'>;

```

### New Function: `resolveEffectiveConfig()`

Location: `packages/data-ops/src/agents/session/resolve-config.ts`

```ts
import type { TradingConfig } from '../../trading-config/schema';
import type { StrategyTemplate } from '../llm/types';
import { DEFAULT_SESSION_CONFIG } from './defaults';
import type { EffectiveConfig, SessionConfig } from './types';

interface ConfigSources {
  tradingConfig: TradingConfig | null;   // from PG via getTradingConfig()
  sessionConfig: SessionConfig;           // from DO SQLite via loadConfig()
  activeStrategy: StrategyTemplate | null; // from strategy_templates SQLite
}

export function resolveEffectiveConfig(sources: ConfigSources): EffectiveConfig {
  const { tradingConfig, sessionConfig, activeStrategy } = sources;
  const _sources: Record<string, EffectiveConfig['_sources'][string]> = {};

  // --- Position sizing ---
  // Priority: tradingConfig > sessionConfig > strategy > default
  let positionSizePctOfCash: number;
  if (tradingConfig?.positionSizePctOfCash !== undefined) {
    positionSizePctOfCash = tradingConfig.positionSizePctOfCash;
    _sources['positionSizePctOfCash'] = 'trading_config';
  } else if (sessionConfig.positionSizePctOfCash !== DEFAULT_SESSION_CONFIG.positionSizePctOfCash) {
    // Session config was explicitly changed from default
    positionSizePctOfCash = sessionConfig.positionSizePctOfCash;
    _sources['positionSizePctOfCash'] = 'session_config';
  } else if (activeStrategy?.positionSizeBias) {
    positionSizePctOfCash = activeStrategy.positionSizeBias;
    _sources['positionSizePctOfCash'] = 'strategy_profile';
  } else {
    positionSizePctOfCash = DEFAULT_SESSION_CONFIG.positionSizePctOfCash;
    _sources['positionSizePctOfCash'] = 'default';
  }

  // --- LLM provider/model ---
  // Priority: tradingConfig.analystModel > sessionConfig > default
  let llmProvider = sessionConfig.llmProvider;
  let llmModel = sessionConfig.llmModel;
  _sources['llmProvider'] = 'session_config';
  _sources['llmModel'] = 'session_config';

  if (tradingConfig?.analystModel) {
    const parsed = parseModelString(tradingConfig.analystModel);
    if (parsed) {
      llmProvider = parsed.provider;
      llmModel = parsed.model;
      _sources['llmProvider'] = 'trading_config';
      _sources['llmModel'] = 'trading_config';
    }
  }

  // --- Risk management fields (from trading config only, with defaults) ---
  const maxPositionValue = tradingConfig?.maxPositionValue ?? 5000;
  const maxPositions = tradingConfig?.maxPositions ?? 10;
  const maxNotionalPerTrade = tradingConfig?.maxNotionalPerTrade ?? 5000;
  const maxDailyLossPct = tradingConfig?.maxDailyLossPct ?? 0.02;
  const takeProfitPct = tradingConfig?.takeProfitPct ?? 0.15;
  const stopLossPct = tradingConfig?.stopLossPct ?? 0.08;
  const cooldownMinutesAfterLoss = tradingConfig?.cooldownMinutesAfterLoss ?? 30;
  const tradingHoursOnly = tradingConfig?.tradingHoursOnly ?? true;
  const extendedHoursAllowed = tradingConfig?.extendedHoursAllowed ?? false;
  const allowShortSelling = tradingConfig?.allowShortSelling ?? false;
  const tickerBlacklist = tradingConfig?.tickerBlacklist ?? [];
  const tickerAllowlist = tradingConfig?.tickerAllowlist ?? null;

  return {
    positionSizePctOfCash,
    llmProvider,
    llmModel,
    maxPositionValue,
    maxPositions,
    maxNotionalPerTrade,
    maxDailyLossPct,
    takeProfitPct,
    stopLossPct,
    cooldownMinutesAfterLoss,
    tradingHoursOnly,
    extendedHoursAllowed,
    allowShortSelling,
    tickerBlacklist,
    tickerAllowlist,

    // Session-specific (pass through)
    orchestrationMode: sessionConfig.orchestrationMode,
    brokerType: sessionConfig.brokerType,
    watchlistSymbols: sessionConfig.watchlistSymbols,
    analysisIntervalSec: sessionConfig.analysisIntervalSec,
    minConfidenceThreshold: sessionConfig.minConfidenceThreshold,
    activeStrategyId: sessionConfig.activeStrategyId,
    debateRounds: sessionConfig.debateRounds,
    proposalTimeoutSec: sessionConfig.proposalTimeoutSec,

    _sources,
  };
}

function parseModelString(
  modelStr: string,
): { provider: LLMProviderName; model: string } | null {
  // Format: "provider/model" e.g. "openai/gpt-4o"
  const slashIdx = modelStr.indexOf('/');
  if (slashIdx === -1) return null;

  const provider = modelStr.slice(0, slashIdx);
  const model = modelStr.slice(slashIdx + 1);

  const validProviders = ['openai', 'anthropic', 'google', 'xai', 'deepseek', 'workers-ai'];
  if (!validProviders.includes(provider)) return null;

  return { provider: provider as LLMProviderName, model };
}
```

### Integration Points

#### 1. SessionAgent Loads Trading Config on Start

`apps/data-service/src/agents/session-agent.ts`

The `onStart()` and `runAnalysisForSymbol()` methods need to fetch `user_trading_config` from Postgres and pass it to `resolveEffectiveConfig()`.

**Current flow:**
```
loadConfig() -> SessionConfig (from DO SQLite only)
```

**New flow:**
```
loadConfig()           -> SessionConfig (from DO SQLite)
getTradingConfig(userId) -> TradingConfig | null (from PG)
getActiveStrategy()    -> StrategyTemplate
resolveEffectiveConfig({ tradingConfig, sessionConfig, activeStrategy }) -> EffectiveConfig
```

**Changes to `SessionAgent`:**

```ts
// New private method
private async loadEffectiveConfig(): Promise<EffectiveConfig> {
  const sessionConfig = this.loadConfig();
  const tradingConfig = await getTradingConfig(this.name); // userId = this.name
  const activeStrategy = this.getActiveStrategy(sessionConfig);
  return resolveEffectiveConfig({ tradingConfig, sessionConfig, activeStrategy });
}
```

Methods that currently call `this.loadConfig()` and use the result for execution decisions should switch to `this.loadEffectiveConfig()`:

| Method | Current | Change |
|--------|---------|--------|
| `onChatMessage()` | `loadConfig()` for LLM provider | `loadEffectiveConfig()` for LLM provider |
| `runAnalysisForSymbol()` | `loadConfig()` for all config | `loadEffectiveConfig()` for position sizing + LLM |
| `createProposal()` | `config.positionSizePctOfCash` fallback | `effectiveConfig.positionSizePctOfCash` fallback |
| `handleTradeDecision()` | `proposal.positionSizePct` (unchanged) | Unchanged -- uses proposal's stored value |
| `resolveProviderConfig()` | `config.llmProvider` / `config.llmModel` | `effectiveConfig.llmProvider` / `effectiveConfig.llmModel` |

Methods that only need session-level settings (scheduling, watchlist, orchestration mode) can continue using `loadConfig()` to avoid the async PG call.

#### 2. Strategy Profile Application

When a user selects a strategy profile (e.g., switching from Moderate to Aggressive), the current code just stores the `activeStrategyId`. The strategy's `positionSizeBias` flows into LLM prompts via the `StrategyTemplate` object, but it also silently controls position sizing because `resolveEffectiveConfig()` uses it as a fallback.

The key behavioral change: if the user has set `positionSizePctOfCash` in their trading config, switching strategies does NOT change their position size. The strategy's `positionSizeBias` is advisory context for the LLM, not an execution parameter.

**No code change needed** in `updateConfig()` -- the resolution function handles priority automatically.

#### 3. Model String Alignment

The `user_trading_config` table stores LLM models as `provider/model` strings (e.g., `openai/gpt-4o`). The session config stores them as separate `llmProvider` + `llmModel` fields. The `ModelSelectionSchema` in `packages/data-ops/src/trading-config/schema.ts` validates the `provider/model` format.

The `parseModelString()` helper in `resolveEffectiveConfig` bridges this gap. The `workers-ai` provider uses a different model ID format (`@cf/meta/...`) which does not match the `provider/model` pattern -- this is handled correctly because workers-ai models are never stored in `user_trading_config` (the `ModelSelectionSchema` regex excludes them).

#### 4. Risk Management Fields Flow to Proposals

Currently `createProposal()` only uses `positionSizePctOfCash` from config. With `EffectiveConfig`, additional risk fields become available:

- `maxNotionalPerTrade` -- cap the notional value of any single proposal
- `stopLossPct` / `takeProfitPct` -- provide defaults when the LLM consensus doesn't specify them
- `allowShortSelling` -- suppress sell proposals when user has no position and short selling is disabled

These are not required in Phase 21 but the plumbing makes them available for future use.

export const ModelSelectionSchema = z

## Alternatives Considered

### Alternative A: Sync trading config into session_config on start

Copy all `user_trading_config` values into the DO SQLite `session_config` table during `onStart()`. The session agent then only reads from one place.

**Rejected**: Creates a stale copy problem. If the user updates their trading config via the settings UI, the session agent's copy is stale until restart. Would need a notification mechanism (queue, webhook) to trigger re-sync.

### Alternative B: Remove session_config, always read from PG

Eliminate the DO SQLite config table entirely and always query Postgres.

**Rejected**: Adds latency to every config read (PG round-trip). Session-specific settings (orchestration mode, watchlist, debate rounds) don't belong in the user-level trading config table. The two config layers serve different purposes.

### Alternative C: Merge at the Postgres level

Add session-specific columns to `user_trading_config` and have one table.

**Rejected**: Violates the separation between user-level preferences and session-level state. A user could have multiple sessions (future) with different orchestration modes but the same risk preferences.

## Open Questions

1. **Should `researchModel` be used?** Currently `user_trading_config` has both `analystModel` and `researchModel`. The session agent only uses one model. Should we map `researchModel` to a specific use case (e.g., debate synthesis vs. individual persona analysis), or defer multi-model support?

2. **Config change notification**: When a user updates `user_trading_config` via the settings UI, should we notify the active session agent to re-resolve? Current design resolves config at each cycle start, so changes take effect within one `analysisIntervalSec` (default 120s). Is that acceptable latency?

3. **Strategy application UX**: When a user selects a new strategy profile, should it also update `session_config.positionSizePctOfCash` to the strategy's `positionSizeBias`? This would make the strategy feel like it "does something" for position sizing, while still letting the user override via trading config. Current design does NOT do this -- the strategy is purely advisory.
