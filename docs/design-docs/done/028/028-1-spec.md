# Phase 28: Proposal Status State Machine -- Part 1: Spec

## Overview

The `handleTradeDecision()` method in `SessionAgent` has a bug in its status transition logic. When a user approves a trade proposal, the method immediately persists `status = 'approved'` to DO SQLite before attempting to execute the order via the broker agent. If the broker call throws an exception, the catch block returns an error response to the caller but does not update the proposal status. The proposal remains permanently stuck in `'approved'` -- a zombie state that:

1. Is not cleaned up by `expireProposals()` (targets only `'pending'`)
2. Cannot be retried by the user (the code checks `status !== 'pending'` and rejects re-approval)
3. Creates a confusing entry in the proposals UI (shows "Approved" with no associated order)
4. Accumulates silently across sessions until `resetData()` is manually triggered

### Why This Matters

- **Data integrity**: The `'approved'` status implies the order was accepted but not yet confirmed. In reality, the order was never placed. The status is semantically wrong.
- **User experience**: The user sees "Approved" in their proposal history and assumes the trade is in progress. There is no indication of failure and no way to retry.
- **Operational risk**: Zombie approved proposals inflate the count of "active" approvals. While this does not currently affect position sizing or risk limits (those are computed from broker positions, not proposal counts), it is a latent data inconsistency.
- **Expiration gap**: The `expireProposals()` sweep only handles `status = 'pending'`. If a proposal fails at 11:00 and expires at 11:05, it will never be expired. `resetData()` does handle `'approved'` proposals (line 324: `WHERE status IN ('pending', 'approved')`), but that requires stopping the session and manually triggering a reset.

## Context & Background

### Current Code Path (session-agent.ts:565-627)

```ts
private async handleTradeDecision(
    proposalId: string,
    approved: boolean,
): Promise<{ status: string; message: string }> {
    // ... fetch proposal, check pending, check expiry ...

    const decidedAt = Date.now();
    if (!approved) {
        this.sql`UPDATE trade_proposals SET status = 'rejected', decided_at = ${decidedAt}
            WHERE id = ${proposalId}`;
        return { status: 'rejected', message: 'Trade rejected by user' };
    }

    // BUG: Status set to 'approved' before execution attempt
    this.sql`UPDATE trade_proposals SET status = 'approved', decided_at = ${decidedAt}
        WHERE id = ${proposalId}`;

    try {
        const userId = this.name;
        const broker = await getAgentByName<Env, AlpacaBrokerAgent>(
            this.env.AlpacaBrokerAgent, userId);

        // ... compute qty/notional ...

        const orderResult = await broker.placeOrder({
            symbol: proposal.symbol,
            side: proposal.action,
            type: 'market',
            timeInForce: 'day',
            qty, notional,
        });

        this.sql`UPDATE trade_proposals SET status = 'executed' WHERE id = ${proposalId}`;
        this.createOutcomeTracking(proposal, orderResult);
        return { status: 'executed', message: `Trade executed: ...` };
    } catch (err) {
        // BUG: No status rollback -- proposal stays 'approved'
        const message = err instanceof Error ? err.message : String(err);
        return { status: 'error', message: `Execution failed: ${message}` };
    }
}
```

### Failure Scenarios

| Failure Point | Cause | Current Behavior | Frequency |
|---------------|-------|-----------------|-----------|
| `getAgentByName()` | DO unavailable, stub error | Proposal stays `'approved'` | Rare |
| `broker.getAccount()` | Credential expired, API down | Proposal stays `'approved'` | Occasional |
| `broker.placeOrder()` | Insufficient funds, invalid symbol, rate limit | Proposal stays `'approved'` | Most common |
| Network timeout | Cloudflare internal routing | Proposal stays `'approved'` | Rare |

### Callers

`handleTradeDecision()` is called from two `@callable()` methods:

1. **`approveProposal(proposalId)`** (line 292-300): Called by the UI when user clicks "Approve". After `handleTradeDecision` returns, it broadcasts the thread state.
2. **`rejectProposal(proposalId)`** (line 302-310): Called by the UI when user clicks "Reject". Passes `approved: false`.

Both callers receive the `{ status, message }` response and broadcast it. Neither caller currently handles the `status: 'error'` case specially -- the UI receives the error message but the underlying proposal data is inconsistent.

## Goals

1. **Introduce `'failed'` status**: Add a new terminal-but-retryable status to the `TradeProposal` status union that explicitly represents "approved by user, but execution failed."
2. **Fix the catch block**: On execution failure, transition the proposal to `'failed'` instead of leaving it as `'approved'`.
3. **Enable retry**: Add a `retryProposal()` callable that allows re-attempting execution on a `'failed'` proposal.
4. **Fix expiration sweep**: Include `'failed'` proposals in `expireProposals()` so timed-out failures are automatically cleaned up.
5. **UI support**: Show `'failed'` status with appropriate styling and a retry action button.
6. **Extract shared logic**: Refactor the order execution code into a shared method to avoid duplication between `handleTradeDecision` and `retryProposal`.

## Non-Goals

- **Automatic retry with backoff**: The system does not automatically retry failed proposals. Retry requires explicit user action. Automatic retry would bypass the human-in-the-loop design principle.
- **Retry count limit**: No maximum retry attempts enforced. The expiration timeout provides a natural limit. If a proposal expires, it cannot be retried regardless of failure count.
- **Error categorization**: Failed proposals do not categorize the error type (transient vs. permanent). All failures are treated uniformly. The error message is stored for debugging but does not influence retry behavior.
- **Alerting/notifications**: No push notification or email when a proposal fails. The user sees the failure in the UI during their session.
- **Schema migration**: No DO SQLite schema changes needed. The `status` column is `TEXT` and accepts any string value. The type constraint is in TypeScript only.

## Solution Design

### New Status: `'failed'`

The `'failed'` status represents a proposal that was approved by the user but could not be executed due to a broker error. It is:

| Property | Value |
|----------|-------|
| Terminal? | No -- can transition to `'executed'` via retry, or to `'expired'` via timeout |
| User-visible? | Yes -- shown in proposal list and detail view |
| Retryable? | Yes -- `retryProposal()` callable transitions back to execution attempt |
| Expirable? | Yes -- `expireProposals()` includes `'failed'` in its sweep |
| Includes error message? | Yes -- the failure reason is returned in the response and can be displayed |

### Status Transitions (Complete)

```
pending -----> approved -----> executed -----> tracking -----> resolved
   |              |                ^
   |              v                |
   |           failed ---------> (retry)
   |              |
   |              v
   |           expired
   |
   +---------> rejected
   |
   +---------> expired
```

### Transition Rules

| From | To | Trigger | Condition |
|------|----|---------|-----------|
| `pending` | `approved` | `handleTradeDecision(id, true)` | Not expired |
| `pending` | `rejected` | `handleTradeDecision(id, false)` | -- |
| `pending` | `expired` | `expireProposals()` | `expires_at < now` |
| `approved` | `executed` | `broker.placeOrder()` succeeds | -- |
| `approved` | `failed` | `broker.placeOrder()` throws | -- |
| `failed` | `executed` | `retryProposal()` -> `broker.placeOrder()` succeeds | Not expired |
| `failed` | `failed` | `retryProposal()` -> `broker.placeOrder()` throws again | Not expired |
| `failed` | `expired` | `expireProposals()` | `expires_at < now` |
| `executed` | `tracking` | `createOutcomeTracking()` | Order filled |
| `tracking` | `resolved` | Exit condition met or manual close | -- |

## Alternatives Considered

### Alternative A: Roll back to `'pending'` on failure

Instead of introducing `'failed'`, revert the status to `'pending'` when execution fails.

**Rejected**: This hides the fact that the user approved the trade. The proposal would appear as if it was never acted on. It also re-enters the pending pool, which could cause confusion if the user sees a proposal they already approved reappear as "Pending." The `decided_at` timestamp would also be semantically wrong (decided but status says pending).

### Alternative B: Keep `'approved'` and add retry logic for approved proposals

Allow `retryProposal()` to work on `'approved'` proposals without introducing a new status.

**Rejected**: This conflates two distinct states: "approved and awaiting execution" (the brief moment during the try block) vs. "approved but execution failed." The `'approved'` status should represent a transient intermediate state, not a durable error state. Adding `'failed'` makes the state machine explicit and debuggable.

### Alternative C: Never set `'approved'` -- go directly from `'pending'` to `'executed'` or `'failed'`

Skip the `'approved'` intermediate status entirely. Mark the proposal as `'executed'` only on success, `'failed'` only on failure.

**Rejected**: The `'approved'` status has value as a signal that the user made a decision. It appears in the UI and audit trail. Removing it would lose information about when the user acted. The fix is simpler: keep `'approved'` as the pre-execution state, but ensure `'failed'` is set on the error path.
