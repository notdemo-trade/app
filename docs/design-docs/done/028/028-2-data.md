# Phase 28: Proposal Status State Machine -- Part 2: Data

## Type Changes

### Modified File: `packages/data-ops/src/agents/session/types.ts`

Add `'failed'` to the `TradeProposal.status` union:

```ts
// Before (line 166)
status: 'pending' | 'approved' | 'rejected' | 'expired' | 'executed';

// After
status: 'pending' | 'approved' | 'rejected' | 'expired' | 'executed' | 'failed';
```

This is the only type change needed. The `TradeProposal` interface is the single source of truth for proposal status values across the entire codebase.

## DO SQLite Schema: No Migration Needed

The `trade_proposals` table in the SessionAgent's DO SQLite database stores `status` as a `TEXT` column:

```sql
CREATE TABLE IF NOT EXISTS trade_proposals (
  id TEXT PRIMARY KEY,
  thread_id TEXT NOT NULL,
  symbol TEXT NOT NULL,
  action TEXT NOT NULL,
  confidence REAL NOT NULL,
  rationale TEXT NOT NULL,
  entry_price REAL,
  target_price REAL,
  stop_loss REAL,
  qty REAL,
  notional REAL,
  position_size_pct REAL NOT NULL,
  risks TEXT NOT NULL DEFAULT '[]',
  warnings TEXT NOT NULL DEFAULT '[]',
  expires_at INTEGER NOT NULL,
  status TEXT NOT NULL DEFAULT 'pending',
  created_at INTEGER NOT NULL,
  decided_at INTEGER,
  order_id TEXT,
  filled_qty REAL,
  filled_avg_price REAL,
  outcome_status TEXT NOT NULL DEFAULT 'none'
);
```

The `status` column is `TEXT` with no `CHECK` constraint. Storing `'failed'` requires no DDL changes. The type safety comes from the TypeScript `TradeProposal['status']` union, not from the database schema.

### Existing Data

Any proposals currently stuck in `'approved'` status from prior execution failures will remain as `'approved'`. They will not be automatically migrated to `'failed'`. This is acceptable because:

1. These proposals are already past their expiration time
2. The updated `expireProposals()` does not retroactively process `'approved'` proposals (it only adds `'failed'` to the sweep -- see Part 3)
3. `resetData()` already handles `'approved'` proposals (expires them on reset)
4. The next time the user triggers a manual `resetData()`, these zombies are cleaned up

If a more proactive cleanup is desired, a one-time migration can be added to `onStart()`:

```ts
// Optional: migrate existing zombie 'approved' proposals to 'expired'
this.sql`UPDATE trade_proposals SET status = 'expired', decided_at = ${Date.now()}
    WHERE status = 'approved' AND expires_at < ${Date.now()}`;
```

This is recommended but not required for correctness.

## Row Mapping

### Modified File: `apps/data-service/src/agents/session-agent-helpers.ts`

The `rowToProposal()` function casts `row.status` to `TradeProposal['status']`:

```ts
// Line 181 (existing)
status: row.status as TradeProposal['status'],
```

This cast already works for any string value in the `status` column. Adding `'failed'` to the TypeScript union means the cast is now type-safe for the new value. **No code change is needed in `rowToProposal()`.**

However, for completeness and defensive programming, the mapping function already handles unknown status values gracefully because the `as` cast does not validate at runtime. Any consumer of `TradeProposal` that does exhaustive switch/if checks on `status` will need to handle the new `'failed'` case.

## Return Types

The `handleTradeDecision()` and `retryProposal()` methods return `{ status: string; message: string }`. The `status` field in the response is a loose `string`, not constrained to `TradeProposal['status']`. This is intentional -- it also includes meta-statuses like `'error'` and `'expired'` that are not proposal statuses but response-level outcomes.

After this fix, the response statuses are:

| Response `status` | Meaning | Proposal DB status |
|-------------------|---------|--------------------|
| `'error'` | Proposal not found or invalid state | Unchanged |
| `'expired'` | Proposal timed out before/during action | `'expired'` |
| `'rejected'` | User declined the proposal | `'rejected'` |
| `'executed'` | Order placed successfully | `'executed'` |
| `'failed'` | Order placement threw an exception | `'failed'` |

The new `'failed'` response status aligns with the new proposal DB status. Previously, the response returned `status: 'error'` for execution failures, which conflated infrastructure errors (proposal not found) with business errors (broker rejected the order).
