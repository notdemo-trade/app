# Phase 28: Proposal Status State Machine -- Part 4: API

## Callable Methods (WebSocket RPC)

Trade proposals are managed via `@callable()` methods on the `SessionAgent` Durable Object, not via Hono REST endpoints. The UI communicates with the session agent through a WebSocket connection managed by the `useSession` hook.

### Existing Callables (Unchanged)

| Method | Signature | Purpose |
|--------|-----------|---------|
| `approveProposal` | `(proposalId: string) => Promise<{ status: string; message: string }>` | User approves a pending proposal. Calls `handleTradeDecision(id, true)`. |
| `rejectProposal` | `(proposalId: string) => Promise<{ status: string; message: string }>` | User rejects a pending proposal. Calls `handleTradeDecision(id, false)`. |
| `getProposals` | `(status?: string) => Promise<TradeProposal[]>` | List proposals, optionally filtered by status. |

### New Callable

```ts
@callable()
async retryProposal(proposalId: string): Promise<{ status: string; message: string }>
```

**Purpose**: Retry execution of a proposal that is in `'failed'` status.

**Parameters**:

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `proposalId` | `string` | Yes | The ID of the failed proposal to retry |

**Response**:

| Field | Type | Description |
|-------|------|-------------|
| `status` | `string` | One of: `'executed'`, `'failed'`, `'expired'`, `'error'` |
| `message` | `string` | Human-readable description of the outcome |

**Response Scenarios**:

| Condition | `status` | `message` | Proposal DB status |
|-----------|----------|-----------|-------------------|
| Proposal not found | `'error'` | `'Proposal not found'` | Unchanged |
| Proposal not in `'failed'` | `'error'` | `'Can only retry failed proposals, current status: {status}'` | Unchanged |
| Proposal expired since failure | `'expired'` | `'Proposal has expired since failure'` | `'expired'` |
| Broker call succeeds | `'executed'` | `'Trade executed: {action} {symbol}'` | `'executed'` |
| Broker call fails again | `'failed'` | `'Execution failed: {error}'` | `'failed'` |

## Client-Side Integration

### Session Connection Hook

The `useSession` hook (in `apps/user-application/src/lib/session-connection.ts`) provides callable methods from the SessionAgent. The new `retryProposal` callable is automatically available through the WebSocket RPC channel without any changes to the connection hook infrastructure.

**Current pattern** (from session-connection.ts):

```ts
// Existing callable usage
const result = await session.approveProposal(proposalId);
const result = await session.rejectProposal(proposalId);
```

**New callable usage**:

```ts
const result = await session.retryProposal(proposalId);
```

The session connection hook proxies all `@callable()` methods on the SessionAgent. No changes to `session-connection.ts` are needed -- the new method is automatically proxied.

### Type Safety

The `session` object returned by `useSession` is typed based on the SessionAgent's callable methods. Adding `retryProposal()` with the `@callable()` decorator makes it available with full type inference on the client side.

## State Broadcasting

After both `approveProposal()` and `retryProposal()` complete, the thread state is broadcast to all connected WebSocket clients via `broadcastThread()`. This ensures the UI updates in real-time when:

1. A proposal transitions from `'pending'` to `'approved'` to `'executed'` (success path)
2. A proposal transitions from `'pending'` to `'approved'` to `'failed'` (failure path)
3. A proposal transitions from `'failed'` to `'executed'` (retry success)
4. A proposal transitions from `'failed'` to `'failed'` (retry failure -- updated error message)

The `broadcastThread()` call reads the current thread state including the proposal's latest status and pushes it via `this.setState()`. Connected clients receive the update through their WebSocket connection.

### Existing Code Pattern (approveProposal)

```ts
@callable()
async approveProposal(proposalId: string): Promise<{ status: string; message: string }> {
    const result = await this.handleTradeDecision(proposalId, true);
    this.broadcastThread(
        this.sql<ThreadRow>`SELECT * FROM discussion_threads WHERE proposal_id = ${proposalId}`[0]
            ?.id ?? '',
    );
    return result;
}
```

### New Code Pattern (retryProposal)

```ts
@callable()
async retryProposal(proposalId: string): Promise<{ status: string; message: string }> {
    // ... validation and execution (see Part 3) ...

    this.broadcastThread(
        this.sql<ThreadRow>`SELECT * FROM discussion_threads WHERE proposal_id = ${proposalId}`[0]
            ?.id ?? '',
    );
    return result;
}
```

The broadcast pattern is identical.

## No REST API Changes

This bug fix does not add or modify any Hono REST endpoints. The proposal management flow is entirely through the SessionAgent's DO callables via WebSocket. The REST API (`/api/*`) is not involved in proposal approval, rejection, or retry.

---

## Backward Compatibility

The new `retryProposal` callable is additive. Existing clients that do not call it continue to work unchanged. The only behavioral change is:

1. **`approveProposal` response**: Previously returned `{ status: 'error', message: 'Execution failed: ...' }` on broker failure. Now returns `{ status: 'failed', message: 'Execution failed: ...' }`. Clients that check `status === 'error'` will need to also check for `status === 'failed'`, or use a more general failure detection pattern (e.g., `status !== 'executed'`).

2. **`getProposals` response**: Proposals that previously appeared as `'approved'` (zombie state) will now appear as `'failed'`. Clients that exhaustively render based on status need to handle the new value.

Both changes are handled by the UI updates in Part 5.
