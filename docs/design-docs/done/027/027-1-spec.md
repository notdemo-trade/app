# Phase 27: Pipeline Confidence Threshold & Outcome Confidence Recording -- Part 1: Spec

## Overview

The system supports two orchestration modes: **debate** and **pipeline**. Both modes analyze a symbol, produce a recommendation with a confidence score, and optionally generate a trade proposal. The `minConfidenceThreshold` config value (default 0.7, range 0.0-1.0) is meant to filter out low-conviction signals before they become actionable proposals.

Debate mode enforces this threshold correctly. Pipeline mode does not enforce it at all. Additionally, when pipeline outcomes are recorded for performance tracking, the confidence value is hardcoded to 0 instead of using the actual LLM recommendation confidence.

### Why This Matters

- **User trust**: A user who sets `minConfidenceThreshold = 0.8` expects both modes to respect it. Getting a 0.3-confidence proposal in pipeline mode but not in debate mode is a confusing and potentially costly inconsistency.
- **Performance tracking**: The `pipeline_outcomes` table has a `confidence` column specifically for analyzing whether high-confidence predictions correlate with better returns. With all values at 0, any `WHERE confidence >= 0.7` query or confidence-band aggregation returns no rows or misleading results.
- **Mode switching**: Users can switch between debate and pipeline modes via the session config UI. If the modes produce different filtering behavior for the same signal, mode switching becomes unpredictable.

## Context & Background

### Debate Mode: Correct Behavior

In `session-agent.ts`, the `runDebateAnalysis` method calls the `DebateOrchestratorAgent`, receives a `consensus` object, and applies the confidence gate:

```ts
// session-agent.ts:516-519
const consensus = result.consensus;
if (consensus.action !== 'hold' && consensus.confidence >= config.minConfidenceThreshold) {
    await this.createProposal(threadId, symbol, consensus, config);
}
```

This means: only non-hold actions with confidence at or above the configured threshold produce a proposal. A `buy` recommendation with confidence 0.5 and a threshold of 0.7 is silently skipped.

### Pipeline Mode: Missing Gate

In `pipeline-orchestrator-agent.ts`, the `generate_proposal` step checks only:

```ts
// pipeline-orchestrator-agent.ts:379-398
case 'generate_proposal': {
    if (!ctx.riskValidation?.approved) {
        // ... emit rejection message, return
    }
    if (!ctx.recommendation || ctx.recommendation.action === 'hold') {
        // ... emit hold message, return
    }
    // No confidence check here
    ctx.proposal = this.buildProposal(ctx, params);
    // ... emit proposal message
}
```

The `RunPipelineParams` interface does not include `minConfidenceThreshold`, so the pipeline orchestrator has no access to this config value even if it wanted to check it.

### Outcome Recording: Hardcoded Zero

In `pipeline-orchestrator-agent.ts`, the `recordStepOutcome` method inserts a new row into `pipeline_outcomes`:

```ts
// pipeline-orchestrator-agent.ts:238-245
this.sql`INSERT INTO pipeline_outcomes
    (id, session_id, proposal_id, symbol, action, confidence,
     ta_signals_snapshot, realized_pnl, realized_pnl_pct,
     was_correct, resolved_at, created_at)
    VALUES (${crypto.randomUUID()}, ${pipelineSessionId}, ${proposalId},
        ${outcome.symbol}, ${outcome.action}, ${0},       // <-- hardcoded 0
        ${JSON.stringify(taSignals)}, ${outcome.realizedPnl}, ${outcome.realizedPnlPct},
        ${wasCorrect ? 1 : 0}, ${now}, ${now})`;
```

The `outcome` parameter's type is `{ symbol: string; realizedPnl: number; realizedPnlPct: number; action: string }` -- it has no `confidence` field. The caller (`SessionAgent.distributeOutcome`) builds this object from `ProposalOutcome`, which also does not carry confidence.

However, the confidence value does exist at proposal creation time in `buildProposal`:

```ts
// pipeline-orchestrator-agent.ts:478
confidence: rec.confidence,
```

The value is available in `TradeProposal.confidence` but is never passed through the outcome recording path.

## Goals

1. **Enforce confidence threshold in pipeline mode**: Add `minConfidenceThreshold` to `RunPipelineParams` and check it in the `generate_proposal` step, matching debate mode behavior.
2. **Record actual confidence in pipeline outcomes**: Pass the proposal's confidence value through the outcome recording path so `pipeline_outcomes.confidence` reflects the real LLM confidence.
3. **Maintain backward compatibility**: The `minConfidenceThreshold` parameter is optional with a sensible default (0.7, matching `DEFAULT_SESSION_CONFIG`). Existing callers that do not provide it get the same default behavior.

## Non-Goals

- **Changing debate mode behavior**: The debate mode confidence gate is correct and unchanged.
- **Adding confidence to the `ProposalOutcome` type**: The fix reads confidence from the stored `TradeProposal` rather than threading it through the entire outcome tracking lifecycle.
- **Backfilling existing pipeline_outcomes rows**: Existing rows with `confidence = 0` are not retroactively corrected. Future rows will have correct values.
- **UI changes**: No frontend changes are needed. The confidence gate is backend-only, and the outcome confidence column is used only in backend analytics queries.
- **Changing the pipeline_outcomes schema**: The `confidence REAL NOT NULL` column already exists. This fix changes what value is written to it.

---

## Root Cause Analysis

### Bug 1: Missing Confidence Gate

**Root cause**: The pipeline orchestrator was designed as a simpler, linear alternative to the debate orchestrator. The debate mode's confidence gate lives in `SessionAgent.runDebateAnalysis()`, not in the `DebateOrchestratorAgent` itself. When the pipeline mode was implemented, the corresponding gate was overlooked because `PipelineOrchestratorAgent.executeStep('generate_proposal')` handles proposal generation internally (unlike debate mode where `SessionAgent` creates the proposal after receiving the consensus).

The architectural difference is:
- **Debate**: `SessionAgent` receives consensus -> applies gate -> creates proposal
- **Pipeline**: `PipelineOrchestratorAgent` builds proposal internally -> `SessionAgent` stores it

Since the gate logic is in `SessionAgent` for debate mode, it was not replicated in `PipelineOrchestratorAgent` for pipeline mode.

### Bug 2: Hardcoded Zero Confidence

**Root cause**: The `recordStepOutcome` method was written before the outcome distribution path was fully designed. The `confidence` column was included in the schema with the intent to populate it later, but the placeholder `${0}` was never replaced. The `outcome` parameter type definition confirms this -- it was designed for PnL data only, without a confidence field.

## Behavioral Impact

### Current Behavior (Buggy)

| Scenario | Debate Mode | Pipeline Mode |
|----------|-------------|---------------|
| LLM returns confidence 0.3, threshold 0.7 | No proposal (correctly filtered) | Proposal generated (bug) |
| LLM returns confidence 0.9, threshold 0.7 | Proposal generated | Proposal generated |
| Outcome resolved | N/A (debate uses persona_outcomes) | `pipeline_outcomes.confidence = 0` |

### Target Behavior (Fixed)

| Scenario | Debate Mode | Pipeline Mode |
|----------|-------------|---------------|
| LLM returns confidence 0.3, threshold 0.7 | No proposal | No proposal + system message |
| LLM returns confidence 0.9, threshold 0.7 | Proposal generated | Proposal generated |
| Outcome resolved | N/A | `pipeline_outcomes.confidence = 0.9` (actual value) |

## Alternatives Considered

### Alternative A: Move confidence gate to SessionAgent.runPipelineAnalysis()

Apply the gate in `session-agent.ts` after receiving `result.proposal` from the pipeline, similar to how debate mode does it. Discard the proposal if confidence is below threshold.

**Rejected**: This would mean the pipeline orchestrator still builds a full proposal (including risk validation, position sizing), only for `SessionAgent` to throw it away. Wasted compute. It is better to skip proposal generation entirely inside the pipeline when confidence is insufficient.

### Alternative B: Thread confidence through ProposalOutcome

Add a `confidence` field to the `ProposalOutcome` type and carry it through the entire outcome tracking lifecycle (`createOutcomeTracking` -> `resolveOutcome` -> `distributeOutcome`).

**Rejected**: This would require modifying the `ProposalOutcome` interface, the DO SQLite `proposal_outcomes` table schema (which needs a migration method for the new column), and multiple intermediate functions. The simpler approach is to look up the confidence from the stored `TradeProposal` at distribution time, since the proposal is already queried in `resolveOutcome`.

### Alternative C: Store confidence in PipelineContext and pass it alongside outcome

Have the pipeline context carry confidence explicitly and thread it into the outcome distribution call.

**Rejected**: The confidence is already stored in `TradeProposal.confidence` which is persisted in DO SQLite. Duplicating it in another data path adds maintenance burden without benefit.
