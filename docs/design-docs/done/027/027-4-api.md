### `recordStepOutcome` (Pipeline Orchestrator Callable)

**File**: `apps/data-service/src/agents/pipeline-orchestrator-agent.ts`

**Current parameter type**:

```ts
outcome: {
    symbol: string;
    realizedPnl: number;
    realizedPnlPct: number;
    action: string;
}
```

**Updated parameter type**:

```ts
outcome: {
    symbol: string;
    realizedPnl: number;
    realizedPnlPct: number;
    action: string;
    confidence: number;     // NEW
}
```

This is a `@callable()` method invoked via Durable Object RPC from `SessionAgent.distributeOutcome()`. The caller (`SessionAgent`) is the only consumer. Adding a required field is safe because both the caller and callee are deployed together.

## Backward Compatibility

### `RunPipelineParams.minConfidenceThreshold`

The field is optional (`minConfidenceThreshold?: number`). If omitted, the pipeline uses a hardcoded default of `0.7`:

```ts
const threshold = params.minConfidenceThreshold ?? 0.7;
```

No existing caller is broken. The only caller (`SessionAgent.runPipelineAnalysis`) is updated to pass the value explicitly.

### `recordStepOutcome` outcome.confidence

The `confidence` field is required in the updated type. The only caller (`SessionAgent.distributeOutcome`) is updated in the same deployment to include it. There is no risk of a partial deployment where the caller sends the old shape and the callee expects the new shape, because:

1. Both `SessionAgent` and `PipelineOrchestratorAgent` live in the same Worker (`apps/data-service`)
2. They are deployed atomically via `pnpm run deploy:staging:data-service`
3. The `@callable()` decorator uses JSON serialization -- adding a field to the object is additive

If there were a concern about rolling deployments, the `confidence` field could be made optional in the callee with a fallback. However, single-worker atomic deployment makes this unnecessary.

## No Frontend Changes

No server functions, query keys, or API client calls are modified. The confidence gate operates entirely within the data-service Worker. The outcome confidence recording is a backend-only data quality fix. The frontend already displays proposal confidence from `TradeProposal.confidence`, which was always correct.
