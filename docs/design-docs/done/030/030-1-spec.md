# Phase 030: Schedule Lifecycle Management -- Part 1: Spec

## Overview

The `SessionAgent` Durable Object runs two recurring schedules to power the trading session: an analysis cycle (`runScheduledCycle`) and an outcome tracking cycle (`runOutcomeTrackingCycle`). Both are created in `start()` and should be destroyed in `stop()`. Due to a filtering bug in `stop()` and a missing cleanup step in `start()`, schedules leak and accumulate, causing phantom broker API calls and potential race conditions.

This is a targeted bug fix -- no new features, no schema changes, no API changes. The fix is confined to a single file (`session-agent.ts`) and adds one private helper method.

### Why This Matters

- **Broker API quota**: Alpaca (and other brokers) enforce rate limits. Leaked outcome tracking cycles call `getClock()`, `getPositions()`, and `getOrderHistory()` every 5 minutes per leaked schedule. A user who starts/stops their session 10 times has 10 concurrent outcome trackers running.
- **Data integrity**: Multiple concurrent outcome trackers can race on the same `proposal_outcomes` row, potentially resolving it twice or recording duplicate snapshots.
- **User expectation**: Pressing "Stop" should stop all session activity. The current behavior violates this -- the session appears stopped in the UI but continues making broker API calls in the background.

## Context & Background

### Schedule Primitives

The `AIChatAgent` base class (from `@cloudflare/ai-chat`) provides three schedule primitives:

| Method | Description |
|--------|-------------|
| `this.schedule(date, callback)` | One-shot: fires at `date`, calls `callback` method |
| `this.scheduleEvery(seconds, callback)` | Recurring: fires every `seconds`, calls `callback` method |
| `this.cancelSchedule(id)` | Cancels a schedule by ID |
| `this.getSchedules(filter?)` | Returns all active schedules, optionally filtered |

Schedules are persisted in the DO's internal storage. They survive DO hibernation and restart. They are only removed by explicit cancellation or DO deletion.

### Current Schedule Usage

**`start()` (line 175-181):**
```ts
@callable()
async start(): Promise<SessionState> {
    const config = this.loadConfig();
    this.setState({ ...this.state, enabled: true });
    await this.rescheduleAnalysisCycle(config.analysisIntervalSec);
    await this.scheduleEvery(300, 'runOutcomeTrackingCycle');
    return this.state;
}
```

`rescheduleAnalysisCycle()` correctly cancels existing `runScheduledCycle` schedules before creating a new one. However, `scheduleEvery(300, 'runOutcomeTrackingCycle')` blindly adds a new recurring schedule without checking for or canceling existing ones.

**`stop()` (line 184-194):**
```ts
@callable()
async stop(): Promise<SessionState> {
    this.setState({ ...this.state, enabled: false });
    const existing = this.getSchedules();
    for (const s of existing) {
        if (s.callback === 'runScheduledCycle') {
            await this.cancelSchedule(s.id);
        }
    }
    return this.state;
}
```

The filter `s.callback === 'runScheduledCycle'` explicitly skips `runOutcomeTrackingCycle` schedules. This was likely an oversight when outcome tracking was added after the initial `start()`/`stop()` implementation.

**`onStart()` (line 77-94):**
```ts
async onStart() {
    // ... init ...
    if (this.state.enabled) {
        await this.rescheduleAnalysisCycle(config.analysisIntervalSec);
        await this.scheduleEvery(300, 'runOutcomeTrackingCycle');
    }
}
```

Same issue as `start()`: adds a new outcome tracking schedule without cleaning up any that may have survived from a previous DO instance lifetime.

### Reproduction Scenario

```
1. User calls start()          -> 1 analysis + 1 outcome tracking schedule
2. User calls stop()           -> 0 analysis + 1 outcome tracking schedule (LEAK)
3. User calls start()          -> 1 analysis + 2 outcome tracking schedules (ACCUMULATE)
4. User calls stop()           -> 0 analysis + 2 outcome tracking schedules
5. User calls start()          -> 1 analysis + 3 outcome tracking schedules
...
After N cycles: 0 or 1 analysis + N outcome tracking schedules
```

## Goals

1. **Fix `stop()` to cancel ALL schedules**: Both `runScheduledCycle` and `runOutcomeTrackingCycle` must be canceled when a session is stopped.
2. **Fix `start()` to prevent accumulation**: Cancel all existing schedules before creating new ones, ensuring exactly 2 schedules after any `start()` call.
3. **Fix `onStart()` for DO restart safety**: Same cleanup on DO restart to prevent phantom schedules from a previous instance.
4. **Add safety guard in `runOutcomeTrackingCycle()`**: If the schedule somehow fires after `stop()` (e.g., race between cancellation and execution), exit early when `enabled` is false.
5. **Centralize cancellation logic**: A single `cancelAllSchedules()` helper used by all three methods, preventing future bugs when new schedule types are added.

## Non-Goals

- **Adding a "track outcomes only" mode**: There is an argument for continuing outcome tracking when stopped (to monitor already-executed trades). This is deferred to a future phase. For now, "stop" means "stop everything."
- **Changing schedule intervals or behavior**: The 300-second outcome tracking interval and the analysis cycle scheduling pattern remain unchanged.
- **Adding schedule observability/logging**: While useful, schedule lifecycle logging is out of scope for this bug fix.
- **Changing the `rescheduleAnalysisCycle()` helper**: It already correctly cancels before re-creating. We only extend the pattern to cover all schedule types.
- **UI changes**: The stop/start buttons and session status display remain unchanged.

## Root Cause Analysis

### Root Cause 1: Filter-based cancellation in `stop()`

`stop()` iterates over schedules and cancels only those with `callback === 'runScheduledCycle'`. This was likely written before `runOutcomeTrackingCycle` existed, and the filter was never updated. The fix is to cancel ALL schedules unconditionally -- there is no schedule type that should survive a `stop()`.

### Root Cause 2: Missing idempotency in `scheduleEvery()`

`scheduleEvery()` does not check for existing schedules with the same callback name. It is an additive operation. The caller is responsible for ensuring only one instance exists. Neither `start()` nor `onStart()` fulfills this responsibility for outcome tracking.

### Root Cause 3: Asymmetric schedule management

`rescheduleAnalysisCycle()` encapsulates the "cancel + create" pattern for analysis schedules, but no equivalent helper exists for outcome tracking. The outcome tracking schedule is created inline with a bare `scheduleEvery()` call, making it easy to forget cleanup.

---

## Design Decision: Stop ALL Schedules When Session Is Stopped

**Question**: Should outcome tracking continue when the session is stopped, to keep monitoring already-executed trades?

**Arguments for continuing**:
- Trades executed before stopping still have open positions that need monitoring
- Stopping and re-starting just to track outcomes is inconvenient

**Arguments against continuing**:
- Users expect "stop" to mean "stop all activity"
- Stale API calls waste broker quota
- The session can be restarted to resume tracking
- A dedicated "track outcomes only" mode would be a cleaner solution if this need arises

**Decision**: Stop ALL schedules when session is stopped. If a "track outcomes only" mode is needed, it should be designed as a separate state (`enabled: false, trackingOnly: true`) in a future phase, with explicit UI controls and its own lifecycle management.

## Alternatives Considered

### Alternative A: Fix only `stop()`, leave `start()` as-is

Fix the filter in `stop()` to also cancel `runOutcomeTrackingCycle`, but don't add cleanup to `start()`.

**Rejected**: This prevents leaks from stop-without-restart, but does not prevent accumulation from start/stop/start cycles if `stop()` fails or is interrupted. The `start()` cleanup provides defense-in-depth.

### Alternative B: Use `scheduleEvery()` with deduplication

Modify the schedule call to check for existing schedules with the same callback before adding a new one.

**Rejected**: The `scheduleEvery()` API does not support this natively. Adding a wrapper that checks `getSchedules()` before calling `scheduleEvery()` is functionally equivalent to the proposed `cancelAllSchedules()` approach but less explicit and harder to reason about.

### Alternative C: Reference-count schedules

Track schedule IDs in state and cancel by stored ID.

**Rejected**: Overcomplicates a simple fix. `getSchedules()` already provides the list of active schedules. Storing IDs in state introduces synchronization concerns (state vs. actual schedules could drift).
