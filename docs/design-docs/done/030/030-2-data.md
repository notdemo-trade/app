# Phase 030: Schedule Lifecycle Management -- Part 2: Data

## Schedule State Model

This phase does not introduce any new database tables, columns, or schema changes. The bug exists entirely in the Durable Object's in-memory schedule management. This part documents the schedule state model for clarity.

---

## DO Schedule Storage

Schedules in `AIChatAgent` (the `SessionAgent` base class) are managed by the Cloudflare Durable Object runtime. They are stored in the DO's internal alarm/schedule system, not in SQLite or PostgreSQL.

### Schedule Object Shape

```ts
interface Schedule {
  id: string;         // Unique schedule identifier (runtime-assigned)
  callback: string;   // Method name to invoke (e.g., 'runScheduledCycle', 'runOutcomeTrackingCycle')
  type: 'interval' | 'once';  // Recurring vs. one-shot
  interval?: number;  // Seconds between invocations (for 'interval' type)
  nextAt?: Date;      // Next scheduled execution time
}
```

### Schedule Lifecycle

```
schedule()/scheduleEvery()  -->  [Active]  -->  cancelSchedule()  -->  [Removed]
                                    |
                                    +--  callback fires  -->  [Active] (interval re-arms)
                                    |                    -->  [Removed] (one-shot completes)
                                    |
                                    +--  DO eviction  -->  [Persisted in DO storage]
                                    |                      (restored on next onStart())
                                    |
                                    +--  DO deletion  -->  [Removed permanently]
```

Key properties:
- Schedules survive DO hibernation and restart
- `scheduleEvery()` creates a new independent schedule each time it is called
- There is no built-in deduplication -- multiple schedules with the same callback can coexist
- `cancelSchedule(id)` removes exactly one schedule by ID
- `getSchedules()` returns all active schedules (can filter by `type`)

## Current Schedule Behavior (Buggy)

### After `start()`

| # | Callback | Type | Interval | Created By |
|---|----------|------|----------|------------|
| 1 | `runScheduledCycle` | `once` | N/A (re-scheduled by handler) | `rescheduleAnalysisCycle()` |
| 2 | `runOutcomeTrackingCycle` | `interval` | 300s | `start()` inline |

### After `stop()`

| # | Callback | Type | Interval | Status |
|---|----------|------|----------|--------|
| 1 | `runScheduledCycle` | `once` | N/A | **Canceled** |
| 2 | `runOutcomeTrackingCycle` | `interval` | 300s | **Still active (BUG)** |

### After `start()` -> `stop()` -> `start()`

| # | Callback | Type | Interval | Status |
|---|----------|------|----------|--------|
| 1 | `runScheduledCycle` | `once` | N/A | Active (from 2nd start) |
| 2 | `runOutcomeTrackingCycle` | `interval` | 300s | **Leaked from 1st start (BUG)** |
| 3 | `runOutcomeTrackingCycle` | `interval` | 300s | Active (from 2nd start) |

### After N `start()` -> `stop()` cycles ending with `start()`

| Schedule | Count |
|----------|-------|
| `runScheduledCycle` | 1 (correctly deduplicated by `rescheduleAnalysisCycle()`) |
| `runOutcomeTrackingCycle` | N (one leaked per `start()` call) |

## Target Schedule Behavior (Fixed)

### After `start()`

| # | Callback | Type | Interval |
|---|----------|------|----------|
| 1 | `runScheduledCycle` | `once` | N/A |
| 2 | `runOutcomeTrackingCycle` | `interval` | 300s |

Exactly 2 schedules. `cancelAllSchedules()` runs before creating new ones.

### After `stop()`

| # | Callback | Type | Interval |
|---|----------|------|----------|
| (none) | | | |

Zero schedules. `cancelAllSchedules()` removes everything.

### After `start()` -> `stop()` -> `start()`

| # | Callback | Type | Interval |
|---|----------|------|----------|
| 1 | `runScheduledCycle` | `once` | N/A |
| 2 | `runOutcomeTrackingCycle` | `interval` | 300s |

Exactly 2 schedules regardless of how many start/stop cycles have occurred.

### After DO restart with `enabled: true`

| # | Callback | Type | Interval |
|---|----------|------|----------|
| 1 | `runScheduledCycle` | `once` | N/A |
| 2 | `runOutcomeTrackingCycle` | `interval` | 300s |

`onStart()` calls `cancelAllSchedules()` before re-creating, ensuring a clean slate even if the previous DO instance left orphaned schedules.

### After DO restart with `enabled: false`

| # | Callback | Type | Interval |
|---|----------|------|----------|
| (none) | | | |

No schedules created. Any orphaned schedules from a previous instance are cleaned up by `onStart()` (which should also call `cancelAllSchedules()` before the `if (enabled)` check to handle the edge case where a DO restarted after a `stop()` that failed mid-cancellation).

## State Invariants

After this fix, the following invariants hold:

| Condition | Expected Schedule Count |
|-----------|------------------------|
| `enabled: true`, after `start()` | Exactly 2 (`runScheduledCycle` + `runOutcomeTrackingCycle`) |
| `enabled: false`, after `stop()` | Exactly 0 |
| `enabled: true`, after `onStart()` | Exactly 2 |
| `enabled: false`, after `onStart()` | Exactly 0 |
| Any time `getSchedules()` is called | 0 or 2 (never 1, never >2) |

The last invariant (never 1) holds because `cancelAllSchedules()` runs before both schedules are created, and both are created in sequence. If a failure occurs between the two `schedule*()` calls, the next `start()` or `onStart()` will clean up and re-create both.
