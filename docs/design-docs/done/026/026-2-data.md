# Phase 26: Fix Position Size Units Mismatch — Part 2: Data

## No Schema Changes Required

This bug fix does not require any database schema changes. The mismatch is purely in application logic -- the values stored in both PostgreSQL and DO SQLite are correct in their respective conventions; the bug is in how they are combined at runtime.

### PostgreSQL (via Drizzle)

The `user_trading_config.position_size_pct_of_cash` column stores fractions (e.g., `0.10`). This is correct and consistent with its Zod validation (`z.number().min(0.01).max(1.0)`). No change needed.

### DO SQLite

**`session_config` table:**

The `position_size_pct` column has `DEFAULT 0.05` (fraction). This stores the per-session copy of `positionSizePctOfCash`. The value is correct -- it represents the user's configured fraction. The bug is not in storage but in how the value is used when constructing a `TradeProposal`. No change needed.

**`trade_proposals` table:**

The `position_size_pct` column stores whatever `TradeProposal.positionSizePct` was at proposal creation time. After this fix, new proposals will consistently store whole-number percentages (e.g., `5` for 5%). Historical proposals created via the debate fallback may contain `0.05` instead of `5`. This is cosmetic -- the actual executed trade amount was computed from the (incorrect) value at execution time and recorded in `filled_qty` / `filled_avg_price`. No migration is needed.

**`consensus_results` table (debate-orchestrator-agent):**

The `position_size_pct` column stores the LLM consensus value directly (whole-number or null). This is not affected by the fix -- the consensus parsing is unchanged.

### Types

No type changes needed. The `TradeProposal.positionSizePct` field remains `number`. The semantic meaning (whole-number percentage) is established by convention, not by the type system.

---

## Zod Schemas

No Zod schema changes. The position sizing schemas in `packages/data-ops` validate config inputs (fractions 0.01-1.0), which remain correct. The `TradeProposal` type in `packages/data-ops/src/agents/session/types.ts` has no Zod schema -- it is a plain interface.

---

## Migration

No database migration required. No `drizzle:*:generate` or `drizzle:*:migrate` commands needed.
