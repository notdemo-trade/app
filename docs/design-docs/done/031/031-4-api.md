# Phase 31: Fix Short Position Handling in Exit Logic -- Part 4: API

## No API Changes

This phase does not add, modify, or remove any Hono REST endpoints. All fixes are internal to Durable Object agent methods.

---

## Affected Internal Interfaces

### `SessionAgent` Callables (No Change)

The `SessionAgent` exposes several `@callable()` methods via WebSocket RPC. None of the bug fixes alter callable signatures or behavior:

| Callable | Status | Notes |
|----------|--------|-------|
| `getProposals()` | Unchanged | Returns proposals; the fix changes how proposals are created, not how they are read |
| `getOutcomes()` | Unchanged | Returns outcomes; the fix changes how `exit_reason` is classified, not how outcomes are queried |
| `getOutcomeSnapshots()` | Unchanged | Unrelated to exit logic |
| `approveProposal()` | Unchanged | Approval flow is unaffected |
| `rejectProposal()` | Unchanged | Rejection flow is unaffected |

### `DebateOrchestratorAgent` Callables (No Change)

| Callable | Status | Notes |
|----------|--------|-------|
| `runDebate()` | Internal return value fixed | The `DebateSession.symbol` field in the return value will now contain the actual symbol instead of `''`. Callers already expect `symbol: string` per the `DebateSession` interface. |

## Message Flow Impact

### Exit Proposal Broadcasting

When `createExitProposal()` creates a new proposal, it calls `this.broadcastThread(threadId)` to notify the frontend via WebSocket. The fix does not change the broadcast behavior -- it prevents _duplicate_ broadcasts for short positions.

**Before fix (short position)**: Each tracking cycle creates a new thread and proposal, broadcasting each one. The frontend receives multiple exit proposals for the same position.

**After fix (short position)**: Only the first tracking cycle creates the thread and proposal. Subsequent cycles detect the pending exit proposal via the corrected dedup query and return early.

### Outcome Resolution Broadcasting

When `resolveOutcome()` completes, it updates the `proposal_outcomes` row with the `exit_reason`. This data is included in subsequent `getOutcomes()` callable responses. The fix ensures the correct `exit_reason` value is written.

**Before fix (short position)**: `exit_reason` is often `'manual_close'` even when a stop-loss or target was hit.

**After fix (short position)**: `exit_reason` correctly reflects `'stop_loss'` or `'target_hit'`.

### Debate Session Symbol

The `DebateSession` returned by `runDebate()` is used by `SessionAgent.runDebateAnalysis()` to construct the discussion thread and trade proposal. The `symbol` field flows into the discussion thread metadata and is broadcast to the frontend.

**Before fix**: `session.symbol` is `''`. The `SessionAgent` uses `params.symbol` from its own context (not `session.symbol`) to create proposals, so the empty symbol does not affect proposal creation. However, any logging or debug output that reads `session.symbol` shows an empty string.

**After fix**: `session.symbol` contains the actual ticker (e.g., `'AAPL'`).
