# Phase 25: Fix Broken Outcome Distribution — Part 4: API

## No API Changes Required

This bug fix is entirely within the `SessionAgent` Durable Object's internal logic. No Hono REST endpoints are affected.

### Why No API Changes

The outcome distribution flow is purely internal to the agent system:

1. `SessionAgent.runOutcomeTrackingCycle()` is triggered by the agent's internal scheduler
2. `SessionAgent.resolveOutcome()` is called when position data indicates a trade has closed
3. `SessionAgent.distributeOutcome()` is a private method that calls orchestrator agents via RPC (`getAgentByName`)
4. `recordPersonaOutcome()` and `recordStepOutcome()` are `@callable()` methods on the orchestrator agents, invoked via Durable Object RPC -- not HTTP endpoints

No external client (frontend, API consumer, webhook) is involved in the outcome distribution flow.

### Endpoints That Consume Scoring Data

The following endpoints/callables read data that will begin populating after this fix. They are not modified but will start returning non-empty data:

| Endpoint / Callable | Current Behavior | After Fix |
|---------------------|-----------------|-----------|
| `DebateOrchestratorAgent.getPersonaScores(windowDays)` | Returns empty array | Returns populated `PersonaScore[]` |
| `DebateOrchestratorAgent.getPersonaPatterns(personaId)` | Returns empty array | Returns populated `PersonaPattern[]` |
| `PipelineOrchestratorAgent.getPipelineScores(windowDays)` | Returns empty array | Returns populated `PipelineScore[]` |

These are DO RPC callables, not HTTP endpoints. They are consumed by the `SessionAgent` during analysis cycles (for confidence dampening and persona comparison).

### `TradeProposal` in WebSocket State

The `TradeProposal` type is broadcast to the frontend via WebSocket when proposals are created or updated. Adding the `orchestratorSessionId` field to the type means the WebSocket payload will include this new field. However:

- The field is informational and not rendered by the frontend
- The frontend ignores unknown fields in the proposal object
- No frontend code changes are needed to handle it

### Future Consideration

If a diagnostic or admin endpoint is added to inspect outcome distribution status, the `orchestratorSessionId` field on proposals and outcomes would be useful for debugging. This is not part of this fix.
