# Phase 25: Fix Broken Outcome Distribution — Part 5: UI

## No UI Changes Required

This bug fix is entirely within the backend agent system. No frontend components, routes, or server functions are affected.

### Why No UI Changes

The outcome distribution and scoring pipeline operates entirely within the Durable Object layer:

1. Trade outcomes are resolved by `SessionAgent.runOutcomeTrackingCycle()` (an internal scheduler-driven method)
2. Scores are computed inside orchestrator agents (`recomputeScores()`, `recomputePipelineScores()`)
3. Scores are consumed internally during debate analysis for confidence dampening
4. No UI component currently displays persona scores or pipeline scores

### UI Components That Will Benefit Indirectly

Once scoring data begins populating, future UI features that consume persona/pipeline performance data will start working. These do not exist yet but are referenced in the roadmap:

| Future Feature | Data Source | Status |
|----------------|------------|--------|
| Persona performance dashboard | `getPersonaScores()` | Not yet implemented |
| Pipeline accuracy metrics | `getPipelineScores()` | Not yet implemented |
| Confidence calibration display | `persona_scores.confidence_calibration` | Not yet implemented |
| Best/worst symbol tracking | `persona_scores.best_symbol`, `persona_scores.worst_symbol` | Not yet implemented |

### Session Discussion Thread

The discussion thread UI (`apps/user-application/src/routes/_auth/session/`) displays proposals, but the `orchestratorSessionId` field is not rendered anywhere. The `TradeProposal` object broadcast via WebSocket will include the new field, but the frontend will simply ignore it -- no destructuring or rendering changes needed.
