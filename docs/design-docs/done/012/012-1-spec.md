# Phase 12: Orchestrator Agent — Part 1: Spec

## Overview

OrchestratorAgent: per-user coordinator of sub-agents. Extends `Agent<Env, State>`.

### MVP Scope
Coordinates TA + LLM agents only. Collects technical signals from shared PG, runs LLM analysis, logs **recommendations** for UI display. No trade execution, no signal agents, no approval flow.

**MVP deploy outcome**: Users enable orchestrator → TA indicators computed → LLM produces recommendations → displayed in UI via WebSocket.

### Full Scope (post-MVP)
Owns the full trading loop: gathers signals from Phase 4 agents via `getAgentByName()` RPC, runs TA from Phase 5, triggers LLM analysis from Phase 6, proposes/executes trades through approval flow. Manages agent entitlements.

### Progressive Enhancement
| Phase | Adds |
|-------|------|
| Phase 8 | Trade execution (`proposeTradeFromRecommendation`, `executeOrder`) |
| Phase 9 | Risk policy engine pre-execution |
| Phase 10 | Approval flow (`executeApproval`, `processExpiredApprovals`) |
| Phase 11 | Telegram notification in trade proposal flow |
| Phase 4 | `gatherSignals()` schedule, signal agent RPC |
| Phase 7 | Strategy template customization |

## Context

Previous design: monolithic TradingAgent doing everything. New design: OrchestratorAgent coordinates sub-agents via RPC. Signal gathering, TA, LLM are independent agents.

> `agents` package already installed in Phase 5 (005-6-ops.md). No need to re-add.

## Why Multi-Agent over Monolithic

| Monolithic (old) | Multi-Agent (new) | Win |
|---|---|---|
| One giant DO class | 3 focused agent types (MVP), 7 full | Single responsibility |
| All logic in one instance | Sub-agents independently deployable | Isolation |
| One billing unit | Per-agent x402 billing (future) | Monetization |
| Shared state bloat | Each agent owns its state | Clean boundaries |
| One failure = all fail | Sub-agent failure doesn't crash orchestrator | Resilience |

## Goals & Non-Goals

**MVP Goals**:
- Per-user Orchestrator instance
- Coordinate TA + LLM agents via `getAgentByName()` RPC
- Collect technical signals from shared PG
- LLM analysis triggering with strategy context
- Log recommendations to SQLite for UI display
- Agent entitlements (simple table, which agents user can use)
- Real-time status via WebSocket state sync
- Activity log in SQLite
- Manual trigger for testing
- `scheduleEvery()` for continuous operation

**MVP Non-Goals** (added by later phases):
- Trade execution (Phase 8)
- Approval flow (Phase 10/11)
- Signal agent coordination (Phase 4)
- Strategy customization (Phase 7)
- Payment/billing system (future x402)
- Complex multi-strategy simultaneous execution
- Strategy marketplace
- Agent sharing between users

## Features

### 12.1 OrchestratorAgent
Per-user coordinator. Extends `Agent<Env, AgentState>`. Manages trading loop lifecycle.

### 12.2 Sub-Agent Coordination
MVP: calls TA + LLM via `getAgentByName()` RPC. Phase 4 adds signal agent calls.

### 12.3 Entitlements (MVP)
Simple SQLite table: `{ agentType: string, enabled: boolean }`. Orchestrator checks before calling sub-agents. No payment integration — just enable/disable flags.

### 12.4 Signal Aggregation Loop
**MVP**: no separate signal gathering schedule. `collectAllSignals()` queries PG directly (TA writes there).
**Post-MVP** (Phase 4): `scheduleEvery(dataPollIntervalSec)` calls enabled signal agents.

### 12.5 Analysis + Recommendation Loop
`scheduleEvery(analystIntervalSec)` — collect signals from PG → TA agents → LLM agent (with strategy) → log recommendation.
**Post-MVP** (Phase 8): recommendation → `proposeTradeFromRecommendation()` → approval/execute.

### 12.6 Strategy Templates
Stored in orchestrator SQLite. Default templates: Conservative, Moderate, Aggressive. Passed to LLMAnalysisAgent per-request. Phase 7 adds customization.

### 12.7 Agent Enable/Disable
RPC via `getAgentByName()`. No HTTP routing.

### 12.8 Real-Time Status
`useAgent()` hook — state updates push to UI via WebSocket.

### 12.9 Manual Trigger
`@callable()` method — runs cycle immediately.

### 12.10 Activity Log
SQLite-backed via `this.sql`. Queryable.

## Type Definitions

```ts
// packages/data-ops/src/agents/orchestrator/types.ts

export interface OrchestratorConfig {
  dataPollIntervalSec: number       // Signal gathering (default: 30)
  analystIntervalSec: number        // LLM analysis (default: 120)
  minSentimentScore: number
  minAnalystConfidence: number
  positionSizePctOfCash: number
  maxPositionValue: number
  maxPositions: number
  takeProfitPct: number
  stopLossPct: number
  autoApproveEnabled: boolean
  autoApproveMaxNotional: number
  watchlistSymbols: string[]
  tickerBlacklist: string[]
  activeStrategyId: string          // which strategy template to use
}

export interface OrchestratorState {
  enabled: boolean
  lastDataGatherAt: string | null
  lastAnalysisAt: string | null
  lastTradeAt: string | null
  currentCycleStartedAt: string | null
  cycleCount: number
  errorCount: number
  lastError: string | null
}

export interface AgentEntitlement {
  agentType: string   // "StockTwitsAgent" | "TwitterAgent" | "SecFilingsAgent" | "FredAgent" | "TechnicalAnalysisAgent" | "LLMAnalysisAgent"
  enabled: boolean
}

// MVP: only technicals. Phase 4 adds stocktwits/twitter/sec/fred.
export interface AggregatedSignals {
  technicals: Record<string, TechnicalSignal[]>  // keyed by symbol
  stocktwits?: Signal[]   // Phase 4
  twitter?: Signal[]      // Phase 4
  sec?: Signal[]          // Phase 4
  fred?: Signal[]         // Phase 4
}

export type AgentAction =
  | "started"
  | "stopped"
  | "signals_aggregated"
  | "analysis_started"
  | "analysis_completed"
  | "recommendation_logged"  // MVP: replaces trade_proposed
  | "trade_proposed"         // Phase 8
  | "trade_executed"         // Phase 8
  | "trade_rejected"         // Phase 10
  | "error"

export interface AgentActivity {
  id: string
  timestamp: string
  action: AgentAction
  symbol?: string
  details: string
}

export interface OrchestratorStatus {
  enabled: boolean
  state: OrchestratorState
  config: OrchestratorConfig
  entitlements: AgentEntitlement[]
  recentActivity: AgentActivity[]
  stats: {
    signalsToday: number
    recommendationsToday: number   // MVP: count of recommendation_logged
    proposalsToday: number         // Phase 8
    tradesExecutedToday: number    // Phase 8
  }
}
```
