## Deployment

### Staging

```bash
# 1. Run migration on staging database
pnpm run drizzle:staging:generate && pnpm run drizzle:staging:migrate

# 2. Build and deploy data-ops (dependency)
cd packages/data-ops && pnpm run build

# 3. Deploy data-service
pnpm run deploy:staging:data-service

# 4. Deploy user-application
pnpm run deploy:staging:user-application
```

### Production

```bash
# Same sequence, production targets
pnpm run drizzle:production:generate && pnpm run drizzle:production:migrate
cd packages/data-ops && pnpm run build
pnpm run deploy:production:data-service
pnpm run deploy:production:user-application
```

**Order matters**: The database migration must run before deploying the data-service, because the new code expects the `debate_personas` table and `moderator_prompt` column to exist. The `onConflictDoNothing()` in `seedDefaultPersonas()` makes the migration idempotent -- safe to re-run.

## Rollback Plan

If issues are discovered after deployment:

1. **API rollback**: Redeploy previous version of `data-service`. The new endpoints simply stop existing. The `SessionAgent` fallback to `DEFAULT_PERSONAS` ensures debates continue working.
2. **Frontend rollback**: Redeploy previous version of `user-application`. The `/settings/debate` route disappears.
3. **Database**: The `debate_personas` table and `moderator_prompt` column can remain in place -- they cause no harm to the previous code version. If a full rollback is needed, the migration can be reversed:
   ```sql
   DROP TABLE debate_personas;
   DROP TYPE persona_bias;
   ALTER TABLE user_trading_config DROP COLUMN moderator_prompt;
   ```

The rollback is safe because:
- The old `SessionAgent.loadPersonas()` reads from DO SQLite, not PostgreSQL
- The old code never references the `debate_personas` table
- The `moderator_prompt` column defaults to `NULL`, which the old code ignores
