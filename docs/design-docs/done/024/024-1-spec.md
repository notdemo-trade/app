## Overview

The debate system uses three hardcoded personas (Bull Analyst, Bear Analyst, Risk Manager) to analyze trade opportunities from multiple perspectives before synthesizing a consensus. Users currently have no control over which perspectives are represented, what instructions each persona follows, or how the moderator synthesizes the final recommendation.

This design adds per-user persona storage in PostgreSQL, a CRUD API, and a settings UI so users can tailor the debate to their trading style -- for example, adding a "Momentum Trader" persona, adjusting the Risk Manager to be more aggressive, or removing the Bear Analyst entirely.

### Why This Matters

- Different trading strategies benefit from different analytical perspectives. A day-trader wants momentum and volume personas; a long-term investor wants fundamental and macro personas.
- The moderator prompt controls how conflicting opinions are weighted. A risk-averse user may want the moderator to favor conservative positions; an aggressive user may want it to favor high-conviction calls.
- Persona performance tracking (Phase 20 M8) already records outcomes per `personaId`. Custom personas will naturally participate in this scoring system, giving users feedback on which perspectives improve their results.

## Context & Background

### Current Persona Architecture

**Hardcoded defaults** in `packages/data-ops/src/agents/session/defaults.ts`:

```ts
export const DEFAULT_PERSONAS: PersonaConfig[] = [
  {
    id: 'bull_analyst',
    name: 'Bull Analyst',
    role: 'Identifies buying opportunities and upside catalysts',
    systemPrompt: `You are a bullish market analyst...`,
    bias: 'optimistic',
  },
  {
    id: 'bear_analyst',
    name: 'Bear Analyst',
    role: 'Identifies selling opportunities and downside risks',
    systemPrompt: `You are a bearish market analyst...`,
    bias: 'skeptical',
  },
  {
    id: 'risk_manager',
    name: 'Risk Manager',
    role: 'Evaluates risk/reward and recommends position sizing',
    systemPrompt: `You are a portfolio risk manager...`,
    bias: 'neutral',
  },
];
```

**DO SQLite storage** in `session-agent.ts`:

The `personas` table stores a single JSON blob under key `'current'`. The `loadPersonas()` method reads this blob, falling back to `DEFAULT_PERSONAS`. The `updatePersona()` callable method allows modifying a persona's fields by `id`, but:
- Cannot add new personas
- Cannot remove personas
- Cannot reset to defaults
- Changes are lost when the Durable Object instance is evicted (no PostgreSQL persistence)

**Debate flow** in `debate-orchestrator-agent.ts`:

The orchestrator receives `DebateConfig` (which includes `personas: PersonaConfig[]`) from `SessionAgent.runDebateAnalysis()`. The orchestrator itself does not fetch personas -- it uses whatever is passed to it. This means the integration point is `SessionAgent`, not `DebateOrchestratorAgent`.

**`PersonaConfig` type** in `packages/data-ops/src/agents/debate/types.ts`:

```ts
export interface PersonaConfig {
  id: PersonaId;       // string union: 'bull_analyst' | 'bear_analyst' | 'risk_manager' | string
  name: string;
  role: string;
  systemPrompt: string;
  bias: string;        // freeform string, not constrained
}
```

### Current Moderator Prompt

```ts
export const DEFAULT_MODERATOR_PROMPT = `You are a neutral market analysis moderator...`;
```

Used in `SessionAgent.runDebateAnalysis()`:

```ts
const debateConfig = {
  personas,
  rounds: config.debateRounds,
  moderatorPrompt: DEFAULT_DEBATE_CONFIG.moderatorPrompt,  // always hardcoded
};
```

The moderator prompt is never configurable -- it always uses the default.

## Goals

1. **Persist personas in PostgreSQL**: New `debate_personas` table with per-user rows, surviving DO eviction.
2. **Full CRUD**: Users can create, read, update, delete, and reset personas via API and UI.
3. **Customizable moderator prompt**: Stored in `user_trading_config.moderator_prompt`, editable via settings UI.
4. **Enforce constraints**: Maximum 5 personas per user, minimum 2 active for debates, system prompt length limit.
5. **Lazy seeding**: Default personas created on first API access (not on user registration).
6. **Backward compatible**: Existing sessions work unchanged. `DebateOrchestratorAgent` interface unchanged. Persona performance tracking (`persona_outcomes`, `persona_scores`) continues to work with custom persona IDs.
7. **Default protection**: System-provided default personas can be edited and disabled but not deleted. Reset restores all defaults and removes custom personas.

## Non-Goals

- **Persona marketplace / sharing**: Users cannot share personas with others.
- **Per-symbol persona sets**: All personas apply to all symbols (future phase).
- **Moderator persona customization**: The moderator is a single prompt, not a full persona with bias/role.
- **Changing `PersonaConfig` type structure**: The existing `PersonaConfig` interface remains unchanged; the PostgreSQL schema maps to it.
- **Auto-generating personas via LLM**: Users write their own prompts (future phase could add AI-assisted prompt generation).
- **Persona versioning / history**: No audit trail of prompt changes.

## Persona Model

### Bias Enumeration

The current `PersonaConfig.bias` is a freeform string (`'optimistic'`, `'skeptical'`, `'neutral'`). For the database and validation, we constrain this to a closed enum:

| Value | Description | Current Mapping |
|-------|-------------|-----------------|
| `bullish` | Favors buying, emphasizes upside | `optimistic` (bull_analyst) |
| `bearish` | Favors selling/avoiding, emphasizes downside | `skeptical` (bear_analyst) |
| `neutral` | No directional bias, balanced analysis | `neutral` (risk_manager) |

The existing `PersonaConfig.bias` field type remains `string` for backward compatibility, but the Zod validation schema and PostgreSQL column constrain it to these three values. The hardcoded defaults' bias values (`optimistic`, `skeptical`) are migrated to the new enum values (`bullish`, `bearish`) in the seed data.

### Default vs. Custom Personas

| Aspect | Default Persona | Custom Persona |
|--------|----------------|----------------|
| `isDefault` | `true` | `false` |
| Can delete | No (returns error) | Yes |
| Can disable | Yes (`isActive = false`) | Yes |
| Can edit prompt | Yes | Yes |
| Can edit name | No (display name is fixed) | Yes |
| Can edit bias | Yes | Yes |
| Included in reset | Restored to original values | Deleted |
| Initial `sortOrder` | 1, 2, 3 | Next available (4, 5) |

### Persona ID Format

- Default personas keep their existing IDs: `bull_analyst`, `bear_analyst`, `risk_manager`
- Custom personas get UUID v4 IDs (generated by PostgreSQL `gen_random_uuid()`)
- This ensures backward compatibility with `persona_outcomes` and `persona_scores` tables in the DO SQLite, which reference personas by string ID

### Constraints

| Constraint | Value | Rationale |
|------------|-------|-----------|
| Max personas per user | 5 | More personas = more LLM calls per debate cycle, higher latency and cost |
| Min active personas for debate | 2 | A debate with 1 persona is not a debate |
| Max system prompt length | 2000 chars | Prevent token budget exhaustion; each persona prompt is sent as a system message |
| Max name length | 50 chars | UI display constraint |
| Max role length | 200 chars | Brief description, not a full prompt |

## Moderator Prompt

### Storage

Add a `moderator_prompt` column to the existing `user_trading_config` table rather than a separate table. Rationale:
- The moderator prompt is a single text value per user, not a collection
- `user_trading_config` already stores per-user trading preferences
- Avoids creating a new table for a single column

### Default Behavior

- When `moderator_prompt` is `NULL`, the system uses `DEFAULT_MODERATOR_PROMPT` from `defaults.ts`
- When the user provides a custom prompt, it is stored in the column
- Reset sets the column back to `NULL`

### Integration

`SessionAgent.runDebateAnalysis()` currently hardcodes:
```ts
moderatorPrompt: DEFAULT_DEBATE_CONFIG.moderatorPrompt,
```

After this change:
```ts
moderatorPrompt: userTradingConfig?.moderatorPrompt ?? DEFAULT_DEBATE_CONFIG.moderatorPrompt,
```

## Data Flow

### Persona Loading (Debate Cycle)

```
SessionAgent.runAnalysisForSymbol(symbol, config)
  |
  +-- getDebatePersonas(userId)          // data-ops query -> PostgreSQL
  |     |
  |     +-- rows found? -> filter active -> map to PersonaConfig[]
  |     +-- no rows?    -> seedDefaultPersonas(userId) -> return defaults
  |
  +-- getModeratorPrompt(userId)         // from user_trading_config
  |     |
  |     +-- column non-null? -> use custom prompt
  |     +-- column null?     -> use DEFAULT_MODERATOR_PROMPT
  |
  +-- debateConfig = { personas, rounds, moderatorPrompt }
  |
  +-- debateOrchestrator.runDebate({ ..., config: debateConfig })
       |
       +-- (unchanged -- orchestrator uses whatever config is passed)
```

### Persona CRUD (Settings UI)

```
Settings UI
  |
  +-- GET /api/debate-personas          -> debatePersonaService.list(userId)
  +-- POST /api/debate-personas         -> debatePersonaService.create(userId, data)
  +-- PUT /api/debate-personas/:id      -> debatePersonaService.update(userId, id, data)
  +-- DELETE /api/debate-personas/:id   -> debatePersonaService.remove(userId, id)
  +-- POST /api/debate-personas/reset   -> debatePersonaService.reset(userId)
```

## Alternatives Considered

### Alternative A: Store personas only in DO SQLite

Keep the current pattern of storing personas in the SessionAgent's DO SQLite, but add proper CRUD operations.

**Rejected**: DO SQLite is per-instance and can be evicted. Persona configuration is a user preference that should survive DO lifecycle events. The settings UI also needs to read/write personas when no session is active, which requires a non-DO storage layer.

### Alternative B: Store personas in a JSONB column on `user_trading_config`

Add a `personas` JSONB column to the existing trading config table instead of a dedicated table.

**Rejected**: JSONB makes individual persona CRUD operations awkward (read-modify-write entire array). Constraints like max count and unique names are harder to enforce. No ability to query or index by persona ID for future features (e.g., cross-user analytics on popular persona types).

### Alternative C: Use `agent_configs` table

The existing `agent_configs` table stores per-user agent configuration as a JSONB blob. Personas could be stored there.

**Rejected**: Same JSONB problems as Alternative B. The `agent_configs` table is designed for agent-level runtime state, not user-facing configuration.

### Alternative D: Separate `moderator_prompts` table

Create a dedicated table for moderator prompts.

**Rejected**: Overkill for a single text value per user. Adding a nullable column to `user_trading_config` is simpler and follows the existing pattern of that table holding debate-related user preferences.

## Open Questions

1. **Persona performance continuity**: When a user edits a default persona's system prompt, should the existing `persona_outcomes` and `persona_scores` (which reference `bull_analyst`, `bear_analyst`, `risk_manager`) be reset? The persona ID stays the same but the behavior changed. Current design: no reset -- scores reflect the persona ID regardless of prompt changes. Users should be aware that historical accuracy metrics may not reflect the current prompt.

2. **DO SQLite cache invalidation**: `SessionAgent` currently caches personas in DO SQLite. When the user updates personas via the API (which writes to PostgreSQL), how does the active session pick up the change? Options:
   - (a) Session re-reads from PostgreSQL at each analysis cycle start (simple, up to 120s stale)
   - (b) API notifies SessionAgent via RPC call to invalidate cache (immediate, more complex)
   - Current recommendation: option (a) for simplicity.

3. **Bias impact on scoring**: The `bias` field is used as context in LLM prompts but does not currently affect confidence dampening or score computation. Should a persona's bias influence how its confidence scores are weighted in consensus? Deferred to a future phase.
