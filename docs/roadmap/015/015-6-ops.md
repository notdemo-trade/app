## Alternatives Considered

1. **Absolute rankings (ROI only)**: Favors high-risk strategies
2. **ELO-style ratings**: More complex, not transparent
3. **Real-time WebSocket updates**: Too expensive for MVP
4. **No minimum requirements**: Allows lucky one-hit wonders

---


## Decisions

1. **Composite scoring**: Like sukuna.dev, balances multiple metrics
2. **Hourly updates**: Balance freshness vs compute cost
3. **Minimum 5 trades / 3 days**: Prevents gaming
4. **Anonymous only**: Real identities in Tier 3 (verified)
5. **Public leaderboard**: No auth required for viewing

### Resolution: Sharpe Ratio

**Formula** (trade-level, simplified):
```
returns[] = trades.map(t => (t.exitPrice - t.entryPrice) / t.entryPrice)
avg = mean(returns)
std = stdDev(returns)
sharpe = std === 0 ? 0 : (avg / std) * sqrt(252)
```

**Where**: `packages/data-ops/src/queries/stats.ts` as `calculateSharpe(trades: Trade[]): number`

**Simplifications**:
- Risk-free rate = 0 (negligible for short-term trading)
- Annualized with sqrt(252) (trading days/year)
- Returns 0 when std=0 (no variance) or <2 trades
- Trade-level returns, not daily portfolio returns (simpler, sufficient for leaderboard ranking)

**Null handling**: Phases 14/15 aggregation should treat null Sharpe as 0 (new users with <2 trades get neutral score, not excluded).

### Gaps

- Sharpe always null - composite formula `(ROI*0.4 + Sharpe*0.3 + WinRate*0.15 + (1-DD)*0.15)` breaks
- Starting equity hardcoded to 100000, should come from Alpaca account
- Max drawdown uses `min(pnl_pct)` instead of proper equity curve drawdown
- Rank delete+insert not atomic - visible partial state during update
- Doc says "real-time ranking on trade close" but implementation is hourly cron - contradiction
- Alias search uses LIKE with user input - SQL injection risk
- Composite score ties have no tiebreaker
- Account age 7-day minimum check mentioned but not enforced
