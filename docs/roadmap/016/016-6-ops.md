## Agents SDK Integration Review (2026-02-20)

**Queues for reliable notification delivery.** Current design calls Telegram API inline -- if it fails, notification is lost (logged as "failed" but never retried).

**With Queues**:
```ts
// Producer (in agent/service):
await env.NOTIFICATION_QUEUE.send({
  type: 'notification',
  userId,
  eventType: 'trade_executed',
  payload: buildTradeExecutedPayload({ symbol, side, qty, price, orderType }),
} satisfies NotificationMessage)

// Consumer:
async queue(batch: MessageBatch<NotificationMessage>, env: Env) {
  for (const msg of batch.messages) {
    try {
      await sendTelegramNotification(msg.body)
      msg.ack()
    } catch {
      msg.retry() // auto-retry with backoff, 4-day retention
    }
  }
}
```

**Benefits**: Retry on Telegram API failure (429, 5xx). Batch processing for daily summaries. Decouples notification sending from trade execution path -- agent/service returns immediately, queue delivers async.

**Cooldown**: Queue consumer checks KV cooldown key before sending. If in cooldown, `msg.ack()` (skip, don't retry). Resolves gap "cooldown state in-memory Map won't persist".

**Impact**: Eliminates risk of lost notifications. No schema changes. Add one Queue binding + consumer to wrangler config.

### Resolved Questions

- **Queue strategy**: Single queue for all notification types. Separate queues adds complexity with no current benefit — revisit if priority tiers needed later.
