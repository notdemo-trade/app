## Alternatives Considered

### Order flow: direct submit vs preview+confirm

- **Preview+confirm (chosen)**: safer, shows estimated cost
- **Direct submit**: faster but riskier

### Options: single-leg vs multi-leg

- **Single-leg (chosen)**: simpler MVP, covers basic use cases
- **Multi-leg**: spreads, straddles - defer to post-MVP

### Crypto symbols: pair format vs ticker

- **Pair format (chosen)**: BTC/USD matches Alpaca API
- **Ticker only**: confusing for crypto

---


## Decisions

1. **Order flow**: preview → confirm → submit
2. **Order types**: market, limit, stop, stop-limit for MVP
3. **Options**: single-leg only, no spreads
4. **Crypto TIF**: auto-convert day/fok to gtc
5. **Delta range**: 0.3-0.7 default for practical strikes
6. **Position close**: full or partial qty

## Integration Note: Webhook Registration

When saving Alpaca credentials, **must register webhook** for trade updates (see Phase 9 Daily Loss Tracking).

**Location**: `/apps/data-service/src/hono/handlers/credentials.ts`

```typescript
// After credential save
const registerAlpacaWebhook = async (userId: string, credentials: AlpacaCredential) => {
  const webhookUrl = `${c.env.TELEGRAM_WEBHOOK_BASE_URL}/api/alpaca/webhook/${userId}`

  const alpaca = new AlpacaClient(credentials)
  await alpaca.request('POST', '/v2/webhooks', {
    url: webhookUrl,
    events: ['trade_updates']
  })
}
```

**Purpose**: Enable real-time daily loss tracking (Phase 9 risk management). Without webhook, loss limits cannot be enforced accurately.

**Related**: Phase 9 (`009-phase-9-risk-management.md`) - Daily Loss Tracking Implementation section.

### Resolution: Alpaca Trade Updates

**Decision**: Replace webhook approach with OAuth2 polling via cron. Simpler on CF Workers (no public endpoint, no signature verification, no persistent connections).

**Flow**:
1. Issue OAuth2 token: `POST https://authx.alpaca.markets/v1/oauth2/token` with `grant_type=client_credentials`, `client_id` (API key), `client_secret` (API secret)
2. Token expires in 900s (15min) - cache in KV with TTL
3. Poll `GET /v2/orders?status=all&after={lastCheckTimestamp}` via cron (every 60s during market hours)
4. Diff order statuses against DB, update accordingly
5. Trigger downstream effects (loss tracking, journal entry, notifications)

**Cron expression**: `* 13-21 * * 1-5` (market hours UTC, adjust for ET)

**Why not webhooks**: CF Workers are stateless, public webhook endpoints need signature verification, registration on credential save adds complexity. Polling is simpler and sufficient for <1min latency.
