# Phase 11: Telegram Approvals — Part 1: Spec

## Overview

Telegram bot integration for human-in-the-loop trade approvals and notifications. Users approve/reject trade proposals from Telegram via inline keyboard buttons. Broadcast notifications for trade executions, daily summaries, and risk alerts.

Hooks into the **existing** SessionAgent proposal lifecycle — proposals already live in DO SQLite `trade_proposals` table with pending/approved/rejected/expired/executed/failed status machine. Telegram adds a remote-control interface, not a parallel approval system.

**Deploy outcome**: Users receive trade proposals on Telegram, tap approve/reject, see execution results and daily summaries.

## Context

The multi-agent architecture (Phase 020) established:
- **SessionAgent** — owns proposal lifecycle, `@callable approveProposal(id)` / `rejectProposal(id)` / `retryProposal(id)`
- **AlpacaBrokerAgent** — executes orders via `placeOrder()` RPC
- **Trade proposals** — stored in SessionAgent DO SQLite with automatic expiration via `expireProposals()`
- **Credential system** — encrypted per-user credentials in Postgres (Alpaca, LLM providers)

Phase 11 adds a `telegram` credential type and a webhook-to-SessionAgent bridge.

**Dependencies**:
- Phase 020: SessionAgent with callable approve/reject methods
- Phase 028: Proposal status state machine (pending → approved → executed/failed)
- Phase 029: Risk management enforcement (guards in executeApprovedProposal)

## Goals & Non-Goals

**Goals**:
- Telegram bot setup wizard (create bot, get chatId)
- Pending proposal notifications with inline keyboard (approve/reject)
- Configurable notification preferences per user
- Quiet hours support
- Webhook handler bridging Telegram callbacks to SessionAgent RPC
- Notification types: approvals, executions, rejections, daily summaries, risk alerts

**Non-Goals**:
- Multiple chat destinations per user
- Group chat support
- Bot commands beyond /start (setup only)
- Rich media (images, charts)
- Replacing the web UI approval flow — Telegram is an additional channel

## Features

### 11.1 Telegram Bot Setup Flow

3-step wizard:
1. Create bot via @BotFather, paste botToken
2. Send /start to bot, system captures chatId via webhook
3. Test notification confirms setup

### 11.2 Trade Proposal Notifications

When SessionAgent creates a proposal (via debate or pipeline analysis), dispatch a Telegram message with trade details + inline approve/reject buttons.

### 11.3 Callback Query Handler

Webhook receives Telegram button clicks, maps `approve:{proposalId}` / `reject:{proposalId}` to SessionAgent RPC calls.

### 11.4 Execution & Result Notifications

After order execution (or failure), send result notification with fill price, order ID, or error details.

### 11.5 Notification Types

- `trade_proposal`: Interactive approval request with inline keyboard
- `trade_executed`: Order filled with fill price + order ID
- `trade_rejected`: User rejected or auto-expired
- `trade_failed`: Execution guard blocked or broker error
- `daily_summary`: End-of-day P&L
- `risk_alert`: Daily loss limit breached, kill switch

### 11.6 Notification Preferences

Per-user toggles for each notification type, stored in Postgres `notification_settings` table.

### 11.7 Quiet Hours

Suppress non-critical notifications during configured hours. Risk alerts and trade proposals always sent.
