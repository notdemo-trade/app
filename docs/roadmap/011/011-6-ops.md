# Phase 11: Telegram Approvals — Part 6: Ops & Verification

## Webhook Registration

Register the webhook URL with Telegram when the user saves their bot token. Call during credential save flow:

```ts
// packages/data-ops/src/services/telegram-service.ts (addition)

const TELEGRAM_API = "https://api.telegram.org/bot"

export async function registerWebhook(
  botToken: string,
  webhookUrl: string
): Promise<void> {
  const res = await fetch(`${TELEGRAM_API}${botToken}/setWebhook`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ url: webhookUrl }),
  })

  if (!res.ok) {
    throw new Error(`Failed to register webhook: ${await res.text()}`)
  }
}
```

**Trigger point**: After saving Telegram credential, call:
```ts
await registerWebhook(
  botToken,
  `https://api.notdemo.trade/api/telegram/webhook/${webhookSecret}`
)
```

## Webhook Security

### Problem: Public webhook endpoint

Telegram doesn't sign webhook payloads. Anyone who knows the URL can POST fake updates.

### Solution: Per-user webhook secrets

Instead of `/webhook/:userId` (exposes userId), use a random per-user secret:

```ts
// On credential save, generate a webhook secret
const webhookSecret = crypto.randomUUID()
// Store alongside the Telegram credential
```

Store the `webhookSecret` in the Telegram credential or a separate `webhook_secrets` column. The webhook handler resolves `webhookSecret → userId` via a DB lookup:

```ts
async function resolveUserFromWebhookSecret(
  env: Env,
  webhookSecret: string
): Promise<string | null> {
  // Query credentials table for the telegram credential containing this secret
  // Return the userId if found, null otherwise
}
```

### Additional security measures

1. **IP allowlist**: Telegram sends webhooks from documented IP ranges (149.154.160.0/20, 91.108.4.0/22). Check `X-Forwarded-For` or `CF-Connecting-IP` against these ranges.
2. **Rate limiting**: Apply per-secret rate limiting on the webhook endpoint to prevent abuse.
3. **Callback data validation**: The `proposalId` in callback data is a UUID — validate format before passing to SessionAgent.

## Decisions

1. **No separate approval table** — reuse SessionAgent's `trade_proposals` in DO SQLite
2. **Webhook secret instead of userId in URL** — prevents userId enumeration
3. **Message format**: HTML parse mode (Telegram's recommended format)
4. **Button layout**: Single row [Approve] [Reject]
5. **Quiet hours**: Skip non-critical only (proposals + risk alerts always sent)
6. **Timeout handling**: Existing `expireProposals()` in SessionAgent — no new timer
7. **Credential storage**: Reuse existing encrypted credential system with new `telegram` provider

## Alternatives Considered

### Notification channel: Telegram vs Discord vs Email

- **Telegram (chosen)**: Interactive buttons for approve/reject, mobile-first, instant delivery
- **Discord**: Webhooks only (no interactive buttons without running a bot process)
- **Email**: Too slow for time-sensitive trade approvals

### Separate approval table vs SessionAgent DO SQLite

- **Reuse DO SQLite (chosen)**: No data duplication, proposals already have full lifecycle tracking, approve/reject RPCs already exist
- **Separate PG table**: Would duplicate state, require synchronization, more failure modes

### Webhook URL: userId vs secret

- **Random secret (chosen)**: Prevents userId enumeration via brute force, no auth info in URL
- **Raw userId**: Simpler but leaks user identifiers in webhook URLs
