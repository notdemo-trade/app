## Alternatives Considered

### Execution model: Agents SDK vs raw DO

- **Agents SDK (chosen)**: built-in scheduling, state sync, SQLite, client SDK, streaming
- **Raw DO (previous)**: more control, but massive boilerplate for state, routing, scheduling

### State persistence: SQLite vs DO storage + DB mirror

- **SQLite (Agent built-in) + DB mirror (chosen)**: queryable activity log, config persistence, no manual cap
- **DO storage + DB mirror (previous)**: manual get/put, 100-entry cap, no queries

### Client: useAgent vs React Query polling

- **useAgent hook (chosen)**: real-time state sync via WebSocket, no polling overhead
- **React Query polling (previous)**: 10s stale time, unnecessary requests, delayed updates

### Scheduling: scheduleEvery vs raw alarm

- **scheduleEvery (chosen)**: named schedules, overlap prevention, SQLite persistence, cancel by ID
- **Raw alarm (previous)**: single alarm, manual interval tracking, manual rescheduling

---


## Decisions

1. **SDK**: Cloudflare Agents SDK (`agents` package)
2. **Execution**: `scheduleEvery()` for signal gathering + analysis loops
3. **State**: `this.state` + `setState()` auto-synced to UI via WebSocket
4. **Storage**: SQLite for activity log, config, approval timeouts
5. **DB mirror**: PostgreSQL for cross-user queries + audit
6. **Client**: `useAgent()` hook replaces React Query polling for agent state
7. **Intervals**: configurable per-user (30s data, 120s analysis default), in seconds
8. **RPC**: `getAgentByName()` for worker→agent calls (API handlers, Telegram webhook)
9. **Streaming**: `@callable({ streaming: true })` for LLM analysis output
10. **Auto-approve**: separate notional limit (unchanged)

**runAnalysis() (line 459-490)**:
```typescript
async runAnalysis(): Promise<void> {
  const signals = await this.gatherSignals()
  if (signals.length === 0) return

  const recommendations: TradeProposal[] = []

  for (const signal of signals) {
    // Get technicals from Phase 5
    const technicals = await analyzeTechnicals(signal.symbol)

    // Get LLM recommendation from Phase 6
    const llmRec = await generateTradeRecommendation({
      symbol: signal.symbol,
      signals: [signal],
      technicals,
      model: this.config.analystModel
    })

    if (llmRec.action !== 'hold' && llmRec.confidence >= this.config.minConfidenceThreshold) {
      recommendations.push(llmRec)
    }
  }

  this.setState({ ...this.state, lastAnalysisAt: new Date().toISOString() })

  // Process recommendations
  for (const rec of recommendations) {
    await this.handleTradeProposal(rec)
  }
}
```

**executeOrder() (line 586-603)**:
```typescript
async executeOrder(params: OrderParams): Promise<{ orderId: string; fillPrice: number }> {
  const userId = this.getUserId()

  // Get Alpaca credentials
  const alpacaCreds = await getCredential(this.env.DB, userId, 'alpaca')
  if (!alpacaCreds) throw new Error('Alpaca credentials not configured')

  const alpacaClient = new AlpacaClient(alpacaCreds.data)

  // Submit order via Phase 8
  const order = await alpacaClient.submitOrder({
    symbol: params.symbol,
    qty: params.quantity,
    side: params.action,
    type: 'market',
    time_in_force: 'day'
  })

  // Log to activity
  await this.logActivity('order_executed', params.symbol, { orderId: order.id })

  return { orderId: order.id, fillPrice: parseFloat(order.filled_avg_price) }
}
```

## SDK Review (2026-02-20)

### NEW SDK features Phase 12 should adopt

**1. Queue Tasks** — Trade proposals should use `queue()` instead of direct execution. Benefits: persistence, FIFO ordering, retry with backoff, auto-dequeue on success.

```ts
// Signal → analysis → queue trade → callback executes
await this.queue("processTrade", { symbol, action, quantity, estimatedPrice })

// In handler:
async processTrade(params: TradeQueueItem) {
  await this.executeOrder(params)
}
```

Prevents lost trades if agent hibernates mid-execution. Queue survives restarts unlike in-flight promises. Affects `proposeTradeFromRecommendation()` in 012-3-logic.md.

**2. Retry APIs** — `this.retry()` for all external API calls. Current design has zero retry logic.

```ts
const order = await this.retry(
  () => alpacaClient.submitOrder(params),
  { maxAttempts: 3, shouldRetry: (err) => err.status >= 500 }
)
```

Apply to: Alpaca orders, signal fetching, LLM calls. All three `executeOrder()`, `gatherSignals()`, `runAnalysis()` paths need this.

**3. Workflows for Approval Flow** — Replace manual approval tracking with SDK Workflows.

```ts
// Instead of: approval_timeouts SQLite table + processExpiredApprovals schedule
await this.runWorkflow("tradeApproval", { symbol, action, quantity })

// Workflow lifecycle:
// waitForApproval() → approveWorkflow() / rejectWorkflow()
// Built-in timeout, state tracking, lifecycle callbacks
```

Eliminates from 012-3-logic.md:
- `approval_timeouts` SQLite table
- `processExpiredApprovals` schedule (60s)
- Manual status tracking in `proposeTradeFromRecommendation()`

Adds: `onWorkflowComplete`, `onWorkflowError` lifecycle hooks.

**4. Observability** — `observability.emit()` for agent monitoring. Current design logs to SQLite only.

Emit events for: state changes, schedule execution, errors, trade execution. Complements existing `logActivity()` with structured telemetry.

**5. validateStateChange()** — Add state validation to prevent invalid transitions.

```ts
validateStateChange(oldState: AgentState, newState: AgentState): boolean {
  // Block: enabling without credentials configured
  // Block: enabling while already enabled
  return true
}
```

**6. Readonly Connections** — Admin/viewer connections that see agent state but can't trigger actions.

```ts
shouldConnectionBeReadonly(connection: Connection): boolean {
  return connection.metadata?.role === "viewer"
}
```

**7. onBeforeConnect Auth** — Auth hook via routing options instead of relying solely on Hono middleware for WS connections.

```ts
routeAgentRequest(request, env, {
  onBeforeConnect: async (req) => { /* verify JWT */ }
})
```

**8. AI Model Routing** — Use AI Gateway model routing for LLM analysis (Phase 6 integration). Auto-fallback between providers.
