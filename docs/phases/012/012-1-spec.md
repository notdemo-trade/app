# Phase 12: Autonomous Agent (Cloudflare Agents SDK)


## Overview

Per-user trading agent using [Cloudflare Agents SDK](https://developers.cloudflare.com/agents/). Gathers signals, analyzes with LLM, proposes trades through approval flow, executes on approval. `scheduleEvery()` for continuous operation, built-in state sync to UI via WebSocket.

**Deploy outcome**: Users enable their agent, see real-time state updates, watch it gather signals, analyze, propose trades automatically.


## Context

Phase 10 established approval flow. Phase 11 added Telegram HITL approvals. Phase 12 adds the autonomous agent that creates approvals and handles execution on approval.

## Why Agents SDK over raw Durable Objects

Previous design used raw DO with manual `alarm()` handler, internal HTTP routing, and DO storage. Agents SDK wraps DO with higher-level abstractions:

| Raw DO (old) | Agents SDK (new) | Win |
|---|---|---|
| `state.storage.get/put` | `this.state` + `setState()` | Auto-syncs to connected clients via WS |
| Manual `alarm()` + interval tracking | `scheduleEvery(30, "method")` | Named schedules, overlap prevention, persist in SQLite |
| Internal `fetch()` routing | `getAgentByName()` + direct RPC | Type-safe method calls, no HTTP boilerplate |
| No WebSocket (deferred) | Built-in WS + `onConnect/onMessage` | Real-time UI for free |
| React Query polling | `useAgent()` hook | Bidirectional state sync, auto-reconnect |
| Manual activity log (100 cap) | `this.sql` (per-instance SQLite) | Queryable, no manual cap |
| No streaming | `@callable({ streaming: true })` | Stream LLM analysis to UI |

**What still needs PostgreSQL**: cross-user queries (admin), `pending_approvals` (shared w/ Telegram webhook), `agentActivityLog` (full audit), `agentConfigs` (survives DO eviction).

## Goals & Non-Goals

**Goals**:
- Per-user Agent instance for agent state
- Agent enable/disable controls
- Configurable polling intervals via `scheduleEvery()`
- Signal gathering loop (schedule-based)
- LLM analysis loop (separate schedule)
- Trade proposal via approval flow
- Auto-execute if auto-approve enabled
- Real-time agent status via WebSocket state sync
- Agent activity log in SQLite
- Manual trigger for testing
- Streaming LLM analysis to UI

**Non-Goals**:
- Complex multi-strategy agents
- Strategy marketplace
- Agent sharing between users

## Features

### 12.1 TradingAgent (Agents SDK)

Per-user Agent instance holding state, running scheduled loops. Extends `Agent<Env, AgentState>`.

### 12.2 Agent State (auto-synced)

`this.state` syncs to all connected clients automatically. Running/paused, last run timestamps, cycle count.

### 12.3 Signal Processing Loop

`scheduleEvery(dataPollIntervalSec)` — gathers signals from Phase 4 data.

### 12.4 Trading Loop

`scheduleEvery(analystIntervalSec)` — analyze signals → generate recommendations → check auto-approve → create Telegram approval or execute immediately.

### 12.5 Agent Enable/Disable

Direct RPC via `getAgentByName()`. No HTTP routing needed.

### 12.6 Agent Status (real-time)

`useAgent()` hook — state updates push to UI via WebSocket. No polling.

### 12.7 Agent Config

RPC method + SQLite persistence. Config updates take effect on next schedule tick.

### 12.8 Manual Trigger

`@callable()` method — runs cycle immediately.

### 12.9 Agent Status UI

Real-time via `useAgent()` hook — running indicator, last activity, stats update live.

### 12.10 Agent Controls UI

Enable/disable/trigger via `agent.stub.enable()` etc.

### 12.11 Activity Log

SQLite-backed via `this.sql`. No manual 100-entry cap. Queryable.

### 12.12 Streaming Analysis (new)

`@callable({ streaming: true })` — stream LLM analysis results to UI in real-time.

## Type Definitions

```ts
// packages/data-ops/src/agent/types.ts

export interface AgentConfig {
  // Polling intervals (seconds for scheduleEvery)
  dataPollIntervalSec: number       // Signal gathering (default: 30)
  analystIntervalSec: number        // LLM analysis (default: 120)

  // Thresholds
  minSentimentScore: number         // Min sentiment to consider (0-1)
  minAnalystConfidence: number      // Min LLM confidence to propose trade (0-1)

  // Position sizing
  positionSizePctOfCash: number     // % of cash per trade
  maxPositionValue: number          // Max $ per position
  maxPositions: number              // Max concurrent positions

  // Risk params
  takeProfitPct: number
  stopLossPct: number

  // Auto-approve
  autoApproveEnabled: boolean
  autoApproveMaxNotional: number    // Max $ for auto-approve

  // Symbols
  watchlistSymbols: string[]
  tickerBlacklist: string[]
}

// Agent state — synced to UI via WebSocket automatically
export interface AgentState {
  enabled: boolean
  lastDataGatherAt: string | null
  lastAnalysisAt: string | null
  lastTradeAt: string | null
  currentCycleStartedAt: string | null
  cycleCount: number
  errorCount: number
  lastError: string | null
}

export interface AgentActivity {
  id: string
  timestamp: string
  action: AgentAction
  symbol?: string
  details: string // JSON stringified for SQLite
}

export type AgentAction =
  | "started"
  | "stopped"
  | "signal_gathered"
  | "analysis_started"
  | "analysis_completed"
  | "trade_proposed"
  | "trade_executed"
  | "trade_rejected"
  | "error"

export interface AgentStatus {
  enabled: boolean
  state: AgentState
  config: AgentConfig
  recentActivity: AgentActivity[]
  stats: {
    signalsToday: number
    proposalsToday: number
    tradesExecutedToday: number
  }
}
```
