## Security Considerations

- Kill switch confirmation required to disable
- Policy changes logged for audit
- Daily loss reset only at market open (cron)
- All validations run server-side (not client-side only)

## Alternatives Considered

### Validation: server-only vs client+server

- **Server-only (chosen)**: can't be bypassed, single source of truth
- **Client+server**: better UX but more code duplication

### Kill switch: confirmation type

- **Typed confirmation (chosen)**: "CONFIRM_RESUME_TRADING"
- **Secret hash**: more secure but less usable
- **2FA**: overkill for MVP

### Daily loss reset: automatic vs manual

- **Automatic at market open (chosen)**: consistent behavior
- **Manual reset**: more control but easy to forget

---


## Decisions

1. **Validation location**: server-side only, API returns violations
2. **Kill switch scope**: per-user, not global
3. **Daily reset time**: 9 AM UTC (market open)
4. **Cooldown trigger**: automatic after hitting daily loss limit
5. **Policy persistence**: database, defaults if not configured
6. **Options policy**: separate config nested in main policy

## Daily Loss Tracking Implementation (P0 Blocker)

**Status**: `dailyLossUsd` field exists but has no update mechanism. Must be fixed before staging deployment.

### Current State

Risk state schema includes daily loss tracking:
- Field: `dailyLossUsd: real("daily_loss_usd").notNull().default(0)`
- Query exists: `recordDailyLoss(db, userId, lossUsd)` (lines 638-664)
- Policy engine checks limit (lines 257-268)
- **Missing**: No automatic updates when positions close with losses

### Two Implementation Options

#### Option A: Alpaca Webhook (Recommended)

Register webhook to receive trade fill events. Track losses when positions close.

**Implementation**:

```typescript
// apps/data-service/src/hono/handlers/alpaca-webhook.ts
import { recordDailyLoss } from '@repo/data-ops/db/queries/risk-state'

export const alpacaWebhookHandler = async (c: Context) => {
  const event = await c.req.json()
  const userId = c.req.param('userId')

  // Only track fills that close positions (sell/buy to cover)
  if (event.event === 'fill') {
    const order = event.order
    const isClosing = (order.side === 'sell' && !order.qty_available) ||
                      (order.side === 'buy' && order.position_intent === 'buy_to_close')

    if (isClosing) {
      // Calculate P&L (simplistic - improve with actual position cost basis)
      const pnl = parseFloat(order.filled_avg_price) * parseFloat(order.filled_qty)

      if (pnl < 0) {
        await recordDailyLoss(c.env.DB, userId, Math.abs(pnl))
      }
    }
  }

  return c.json({ received: true })
}

// Register webhook during Alpaca credential save
// apps/data-service/src/hono/handlers/credentials.ts
const registerAlpacaWebhook = async (userId: string, credentials: AlpacaCredential) => {
  const webhookUrl = `${c.env.TELEGRAM_WEBHOOK_BASE_URL}/api/alpaca/webhook/${userId}`

  const alpaca = new AlpacaClient(credentials)
  await alpaca.request('POST', '/v2/webhooks', {
    url: webhookUrl,
    events: ['trade_updates']
  })
}
```

**Pros**:
- Real-time updates
- No polling overhead
- Accurate fill timing

**Cons**:
- Webhook endpoint complexity
- Requires public URL
- Retry handling needed

#### Option B: Position Close Tracking

Poll positions periodically, compare with last snapshot to detect closes.

**Implementation**:

```typescript
// apps/data-service/src/cron/track-daily-loss.ts
export async function trackDailyLossFromPositions(db: DbClient) {
  const users = await getAllUserIds(db)

  for (const userId of users) {
    const trading = await getAlpacaTradingClient(db, userId, env.CREDENTIALS_ENCRYPTION_KEY)
    if (!trading) continue

    const positions = await trading.getPositions()
    const lastSnapshot = await getLastPositionSnapshot(db, userId)

    // Compare snapshots to detect closed positions
    for (const prevPos of lastSnapshot) {
      const stillOpen = positions.find(p => p.symbol === prevPos.symbol)
      if (!stillOpen) {
        const loss = prevPos.unrealized_pl < 0 ? Math.abs(prevPos.unrealized_pl) : 0
        if (loss > 0) {
          await recordDailyLoss(db, userId, loss)
        }
      }
    }

    // Store current snapshot
    await savePositionSnapshot(db, userId, positions)
  }
}

// Cron: every 5 minutes during market hours
// wrangler.toml: crons = ["*/5 9-16 * * 1-5"]
```

**Pros**:
- Simpler implementation
- No webhook infrastructure

**Cons**:
- Delayed updates (5min lag)
- Increased API calls
- Snapshot storage overhead

### Recommendation

**Use Option A (Webhook)** for production:
- Immediate loss tracking prevents circumventing limits
- Lower API usage vs polling
- More accurate for risk management

**Option B as fallback** if webhook registration fails during credential save.

### Resolution: Daily Loss Tracking via Polling

**Decision**: Use OAuth2 polling (see Phase 8 resolution) instead of webhooks. Cron polls order fills, calculates daily P&L delta, updates `riskStates.dailyLossUsd`.

**Flow**:
1. Cron fetches filled orders since last check via OAuth2 Bearer token
2. For each fill: calculate realized P&L = `(exit_price - entry_price) * qty`
3. If loss: increment `dailyLossUsd` in `riskStates` table
4. If `dailyLossUsd >= dailyLossLimitUsd`: block new orders + send notification
5. Daily reset cron zeroes `dailyLossUsd` at market open

### Resolution: Kill Switch → Cancel All Orders

**Flow**: When kill switch activates:
1. Set `riskStates.killSwitchActive = true` in DB
2. Call Alpaca `DELETE /v2/orders` (cancels ALL open orders for the account)
3. Log cancellation in activity log with reason
4. Send notification via Telegram (if configured)
5. Block all new order submissions until manually resumed

**Implementation**: In `activateKillSwitch(userId)` service function:
```ts
async function activateKillSwitch(userId: string, reason: string) {
  await updateRiskState(userId, { killSwitchActive: true, killSwitchReason: reason, killSwitchAt: new Date() })
  const client = await getAlpacaClientForUser(userId)
  await client.request('DELETE', '/v2/orders') // cancels all open orders
  await sendNotification(userId, 'kill_switch', { reason })
}
```

**Resume**: Requires exact string `"CONFIRM_RESUME_TRADING"` + sets `killSwitchActive = false`.

## Agents SDK Integration Review (2026-02-20)

**Minimal SDK impact.** Policy engine is synchronous validation (no external calls, no async steps). No retry, workflow, or queue benefit -- it runs in-memory against DB state.

**One exception**: Kill switch activation calls Alpaca `DELETE /v2/orders` (external API). Wrap in a Workflow step with retry if the Alpaca call fails. Currently if cancel-all-orders fails, kill switch is set but orders remain open -- dangerous.

```ts
// Workflow: KillSwitchWorkflow
await step.do('set-kill-switch', () => enableKillSwitch(db, userId, reason))
await step.do('cancel-orders', { retries: { limit: 3, delay: '2 seconds' } }, () =>
  alpacaClient.request('DELETE', '/v2/orders')
)
await step.do('notify', () => sendNotification(userId, 'kill_switch', { reason }))
```

**Impact**: Single Workflow for kill switch reliability. Rest of Phase 9 unchanged.
