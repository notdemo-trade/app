## Monitoring Setup

### Sentry Integration

```ts
// packages/data-ops/src/utils/sentry.ts

import * as Sentry from "@sentry/cloudflare"

export function initSentry(env: { SENTRY_DSN?: string; ENVIRONMENT?: string }) {
  if (!env.SENTRY_DSN) return

  Sentry.init({
    dsn: env.SENTRY_DSN,
    environment: env.ENVIRONMENT ?? "production",
    tracesSampleRate: 0.1, // 10% of transactions
  })
}

export function captureException(err: Error, context?: Record<string, unknown>) {
  console.error(err)
  Sentry.captureException(err, { extra: context })
}

export function captureMessage(message: string, level: Sentry.SeverityLevel = "info") {
  Sentry.captureMessage(message, level)
}
```

### Worker Entry Point

```ts
// apps/data-service/src/index.ts

import * as Sentry from "@sentry/cloudflare"
import { app } from "./hono/app"

export default Sentry.withSentry(
  (env) => ({
    dsn: env.SENTRY_DSN,
    tracesSampleRate: 0.1,
  }),
  {
    async fetch(request, env, ctx) {
      try {
        return await app.fetch(request, env, ctx)
      } catch (err) {
        Sentry.captureException(err)
        return new Response("Internal Server Error", { status: 500 })
      }
    },
  }
)
```

## Database Backup Strategy

### Neon Features (Automatic)

- **Point-in-time recovery**: 7-day retention (free tier), 30-day (paid)
- **Branching**: Create instant database copies
- **Auto-suspend**: Compute scales to zero when idle

### Manual Backup Script

```bash
#!/bin/bash
# scripts/backup-db.sh

# Load database credentials
if [ -f "packages/data-ops/.env" ]; then
  source packages/data-ops/.env
fi

DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="backup_${DATE}.sql"

# Use separate env vars (matching your actual setup)
export PGHOST="$DATABASE_HOST"
export PGUSER="$DATABASE_USERNAME"
export PGPASSWORD="$DATABASE_PASSWORD"
export PGDATABASE="neondb"

pg_dump > "$BACKUP_FILE"
gzip "$BACKUP_FILE"

# Upload to R2 (optional)
wrangler r2 object put "backups/${BACKUP_FILE}.gz" --file "${BACKUP_FILE}.gz"

echo "Backup completed: ${BACKUP_FILE}.gz"
```

### Recovery Procedure

1. Neon dashboard → Project → Branches
2. Create branch from point-in-time
3. Test recovery on branch
4. Promote branch if needed

## GitHub Actions Deployment

```yaml
# .github/workflows/deploy-production.yml

name: Deploy Production

on:
  push:
    branches: [main]
    paths:
      - 'apps/**'
      - 'packages/**'
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production

    steps:
      - uses: actions/checkout@v4

      - uses: pnpm/action-setup@v2
        with:
          version: 9

      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'pnpm'

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Build
        run: pnpm run build

      - name: Run tests
        run: pnpm run test

      - name: Deploy data-service
        run: pnpm run deploy:production:data-service
        env:
          CLOUDFLARE_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}

      - name: Deploy user-application
        run: pnpm run deploy:production:user-application
        env:
          CLOUDFLARE_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}

      - name: Notify success
        if: success()
        run: |
          curl -X POST "${{ secrets.DISCORD_WEBHOOK }}" \
            -H "Content-Type: application/json" \
            -d '{"content": "✅ Production deployed successfully"}'

      - name: Notify failure
        if: failure()
        run: |
          curl -X POST "${{ secrets.DISCORD_WEBHOOK }}" \
            -H "Content-Type: application/json" \
            -d '{"content": "❌ Production deployment failed"}'
```

## Rollback Procedure

### If deployment fails:

```bash
# Rollback to previous version
wrangler rollback --env production

# Or deploy specific version
wrangler deploy --env production --compatibility-date 2024-01-01
```

### If database issue:

1. Neon dashboard → Create branch from last known good
2. Update DATABASE_HOST in .production.vars to point to branch endpoint
3. Sync secrets: `./sync-secrets.sh production`
4. Redeploy workers

### If critical bug:

1. Disable agent for all users (kill switch)
2. Hotfix and deploy
3. Re-enable agents

### Resolution: Rate Limiting Architecture

**Decision**: Switch from in-memory `Map` to KV-backed rate limiting.

**Implementation plan**:
1. Uncomment KV namespace bindings in `wrangler.jsonc` for all 3 environments (IDs already provisioned)
2. Run `pnpm run cf-typegen` to add `CACHE: KVNamespace` to Env type
3. Rewrite `rate-limiter.ts`:
   - Access KV via `c.env.CACHE` (Hono context)
   - Key pattern: `rl:{ip}` with JSON value `{count, resetTime}`
   - Set `expirationTtl` = `windowMs / 1000` for auto-cleanup
   - Graceful fallback: if KV not bound, skip rate limiting (dev compat)
   - Add `Retry-After` header on 429 responses
4. Add `keyPrefix` option per-route to isolate rate limit buckets

**Why KV over DO**: Rate limiting is read-heavy, eventual consistency acceptable (slight over-allow OK). KV is simpler, cheaper, no DO class needed.

**Affected files**:
- `apps/data-service/wrangler.jsonc` - uncomment KV bindings
- `apps/data-service/src/hono/middleware/rate-limiter.ts` - rewrite
- `apps/data-service/worker-configuration.d.ts` - regenerate via cf-typegen
