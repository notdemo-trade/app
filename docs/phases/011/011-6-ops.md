## Webhook Registration

Register webhook URL with Telegram when user completes setup:

```ts
// packages/data-ops/src/services/telegram-service.ts (additions)

export async function registerWebhook(
  botToken: string,
  webhookUrl: string
): Promise<void> {
  const res = await fetch(`${TELEGRAM_API}${botToken}/setWebhook`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ url: webhookUrl }),
  })

  if (!res.ok) {
    throw new Error(`Failed to register webhook: ${await res.text()}`)
  }
}

// Call during setup:
// await registerWebhook(botToken, `https://api.notdemo.trade/api/telegram/webhook/${userId}`)
```

## Security Considerations

1. **Webhook verification**: Telegram doesn't sign webhooks. Use per-user URLs (`/webhook/:userId`) to isolate.
2. **User ID validation**: Verify `approval.userId === userId` before processing.
3. **Timeout enforcement**: DO alarms auto-reject stale approvals.
4. **Rate limiting**: Apply rate limits on webhook endpoint.
5. **Chat ID capture**: Only capture from `/start` command, not arbitrary messages.
6. **Credential encryption**: Bot tokens encrypted at rest (Phase 2 encryption).

## Alternatives Considered

### Notification channel: Telegram vs Discord vs Email

- **Telegram (chosen)**: Interactive buttons, mobile-first, instant
- **Discord**: Webhooks only (no buttons without bot hosting)
- **Email**: Too slow for trade approvals

### Approval storage: DB vs DO storage

- **DB (chosen)**: Queryable, persists across DO restarts, relational
- **DO storage**: Faster but lost on eviction

### Timeout handling: DO alarm vs cron

- **DO alarm (chosen)**: Per-approval precision, no polling
- **Cron**: Simpler but polling-based, less precise

---


## Decisions

1. **Approval timeout**: 15 minutes default (configurable 1-60)
2. **Webhook URL pattern**: `/api/telegram/webhook/:userId`
3. **Message format**: HTML parse mode
4. **Button layout**: Single row [Approve] [Reject]
5. **Quiet hours**: Skip non-critical (approvals + risk alerts always sent)
6. **Auto-reject on timeout**: Yes, with notification

## Agents SDK Integration Review (2026-02-20)

**Telegram webhook calls `instance.sendEvent()` instead of DB update + Agent RPC.** Direct consequence of Phase 10 Workflow adoption.

**Current design**: Telegram callback -> update `pending_approvals` DB -> `getAgentByName()` RPC -> agent executes order. Three systems coordinating.

**With Workflows**:
```ts
// Telegram webhook handler (simplified)
if (action === 'approve') {
  const instance = await env.TRADE_APPROVAL_WORKFLOW.get(approvalId)
  await instance.sendEvent({ type: 'trade_approval', payload: { action: 'approve' } })
}
```

**Eliminates**: `getAgentByName()` RPC call from webhook, `executeApproval()` method on TradingAgent, `approval_timeouts` SQLite table, `processExpiredApprovals()` scheduled task. The Workflow already handles execution + timeout.

**Timeout handling**: Workflow `timeout: '15 minutes'` replaces DO alarm / `scheduleEvery(60, "processExpiredApprovals")`. More precise, no polling.

**Impact**: Webhook handler shrinks to ~20 lines (parse callback, sendEvent, answerCallbackQuery). Resolves circular dependency gap -- webhook no longer imports TradingAgent types.

### Resolved Questions

- **Workflow instance IDs**: Generate server-side UUIDs. Don't pass user/Telegram-provided IDs as workflow instance IDs.
